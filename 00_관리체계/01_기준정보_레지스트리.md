## 2. 권한 모델 매핑표 (Entra ↔ Supabase ↔ RLS)

> 현재 권한 정의가 **CLAUDE.md(텍스트)·HTML 기획·SQL 함수** 3곳에 흩어져 있어 여기로 일원화한다.
> 상세 설계: `08/03_실구축기획/03_협력사인증_권한_행수준보안_보안.md`. 운영 적용 현황: `이관/00_현황스냅샷 §5`.

| 역할 | Entra 보안그룹(사내) | Supabase 역할/클레임(데모) | RLS 적용 | 데이터 경계 |
|---|---|---|---|---|
| 시스템 관리자 | `AI-Portal-Admins` | `portal_admin` | 전체 | 전 데이터 + 협력사 계정 발급 |
| 자금 | `AI-Portal-Finance` | (미정) | 자금 화면 | 자금·금액 화면 인가 |
| 인사 | `AI-Portal-HR` | (미정) | 인사 화면 | 급여·PII 인가 |
| 영업(확장) | `AI-Portal-Sales` | (미정) | 영업 화면 | 수주 화면 |
| 협력사(사외) | **테넌트 분리** | `vendor` + `vendor_bp`(거래처코드 배열) | **행수준** | 자기 거래처(`bp_cd`) 발주만 |

- **사내 인증** = Microsoft Entra ID(SSO+MFA), 앱 `JEIL-AX-Portal`. 협력사는 동일 테넌트에 넣지 않음(`CLAUDE.md §5.5`).
- **권한 판정은 백엔드/DB에서**(프론트 메뉴 숨김은 UX일 뿐 — `CLAUDE.md §5.4`).
- RLS 헬퍼: `is_vendor_admin()`(사내 관리자 판정), `vendor_bp` JWT 클레임(협력사 거래처 격리). 정의: `이관/sql/01_schema_rls.sql`·`06_vendor_provisioning.sql`.

## 3. ERP 테이블 매핑 카탈로그 (요약 + 포인터)

> ERP는 **읽기 전용 + 중간 DB 경유**(`CLAUDE.md §4`). 운영 MSSQL 직접 쓰기·조회 금지.
> **상세 단일 출처**: `08/03_실구축기획/02_데이터모델_ERP매핑_포털스키마.md`. 아래는 빠른 참조용 요약.

### 3.1 ERP(UNIERP) → 읽기전용 사본(`erp_ro`)

| 업무 | ERP 테이블 | 포털 별칭 | 핵심 컬럼 |
|---|---|---|---|
| 구매요청 | `M_PUR_REQ` | `pr` | PR_NO, PR_STS, REQ_QTY, ORD_QTY, RCPT_QTY |
| 발주(헤더) | `M_PUR_ORD`(_HDR) | `po_header` | PO_NO, BP_CD, ORD_DT, PROJECT_CD |
| 외주구분 | `M_CONFIG_PROCESS` | - | SUBCONTRA_FLG(외주/사급) |
| 입출고 | `M_PUR_GOODS_MVMT` | - | 입고·매입 수량 |
| 검사 | `Q_INSPECTION` | - | 검사신청·결과(합/부) |
| 품목 | `M_IV` | - | 품목 마스터 |
| 거래처 | `B_BIZ_PARTNER` | `vendor_master` | BP_CD, BP_NM(협력사 마스터 원천) |

### 3.2 포털 전용 쓰기 스키마(`sp_*` / 데모는 Supabase) — ERP에 직접 안 씀

`sp_order_state`(진행상태·10단계) · `sp_photo`(증빙사진, 원본은 Storage) · `sp_message`(양방향) · `sp_insp_request`(검수요청) · `sp_inspection`(판정) · `sp_inspection_log`(판정이력). 격리 키 = `bp_cd`. 정의: `이관/sql/01_schema_rls.sql`.

> 신규 화면은 이 매핑을 **재사용·확장**하고, 변경 시 §3 + `08/03/02` 양쪽을 갱신한다.

## 4. 협력사 마스터 정책

| 항목 | 내용 |
|---|---|
| 원천 | ERP `B_BIZ_PARTNER` → 포털 `vendor_master`(`bp_cd` PK, `bp_nm`, `status`) |
| 데모 데이터 | 17개사 샘플(상호 부분 마스킹, 사업자번호 데모값) |
| 온보딩 모델 | **관리자 주도**: `portal_admin`이 ERP 거래처 선택 → 담당자 이메일로 계정·임시비번 발급. 자유가입 제거 |
| 계정 발급 | Edge Function `vendor-provision`(생성)·`approve-vendor`(승인)·`vendor-reset-password`(비번 재설정) |

| 동기화(운영) | ERP `B_BIZ_PARTNER` → `vendor_master` 야간배치(미구축, 유니포인트 협의 대상) |

## 5. 발주·검사 프로세스 10단계 (상태머신)

> `sp_order_state.status`(new·prod·insp·done) + `step`(1~10). 협력사 포털·사내 대시보드 공통 기준.

| step | 단계 | status | 주체 |
|---|---|---|---|
| 1 | 구매요청(PR) 발생 | new | ERP |
| 2 | 발주(PO) 생성 | new | ERP/구매 |
| 3 | 협력사 발주 확인 | prod | 협력사 |
| 4 | 제작 시작 | prod | 협력사 |
| 5 | 제작 완료 | prod | 협력사 |
| 6 | 증빙 사진 등록 | prod | 협력사 |
| 7 | 검수 요청 | insp | 협력사 |
| 8 | 검수 판정(합/부 + 판정자 기록) | insp | 사내 |
| 9 | 입고 | done | ERP |
| 10 | 매입 | done | ERP |

상세: `08/03_실구축기획/02 §3`.
