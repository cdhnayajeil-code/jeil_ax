1프로젝트 개요

사내 구매팀은 ERP(발주 모듈)에서 외주 협력사에 발주를 내고, 협력사는 가공·제작·출하를 진행한다. 그 과정의 증빙(작업·완성 사진), 진행상태, 검수 결과가 전화·메일·메신저로 분산되어 추적이 어려웠다. 본 프로젝트는 발주번호를 축으로 협력사의 등록 활동과 사내 검사·입고·매입을 한 흐름으로 연결하는 것을 목표로 한다.

두 개의 화면으로 구성된다. ① 협력사 모바일 포털 — 협력사가 자사 발주를 조회하고 사진 등록·진행상태 변경·검수요청·메시지를 입력하는 하이브리드앱형 화면. ② 사내 외주 발주·검사 진행현황 대시보드 — 구매·품질·생산·사업관리가 구매요청→발주→협력사→입고→매입 전 과정을 발주별로 보고, 검수 판정(합/부)·메시지를 처리하는 운영 화면. 두 화면은 공유 데이터(localStorage)로 실시간 양방향 연동된다.

핵심 원칙ERP는 읽기전용 연계(발주/구매요청/입고/매입/검사 조회)만 사용한다. 협력사가 생성하는 사진·상태·메시지·검수요청과 사내 검수 판정은 포털 전용 데이터로 보관하여 ERP 본 시스템에 영향을 주지 않는다.

2요구사항 이력 (지시 내용 전체)

지금까지 지시된 내용을 순서대로 정리한다. 각 단계의 요청(원문 요지)과 반영 결과를 함께 기록한다.

① 1차 — 협력사용 발주·사진등록 포털 기획 + 하이브리드앱
"구매발주 업체들의 발주·품목을 ERP 기반으로, 협력사가 로그인해 구매팀과 컨택하며 발주 기준 사진을 등록·확인하는 외주협력사용 페이지를 1차 초안 기획·문서화하고 HTML 하이브리드앱으로 구성"
기획서(Markdown + HTML 2종) + 모바일 하이브리드앱 데모 제작. 로그인·발주 목록/상세·사진 등록(실제 업로드 미리보기)·구매팀 메시지·진행상태 4단계. 로그인 가정: 협력사 계정(ID/PW).
② 단일 HTML 통합 · 관리자 표기 정정
"HTML 하나로 만들고, 관리자는 최동혁, 팀은 넣지 말고 생성"
기획서 + 데모를 단일 HTML로 통합(데모는 iframe 격리 내장). 관리자 최동혁, 팀(부서) 표기 제거.
③ 2차 — 사내 운영페이지에 외주 발주·검사 대시보드 추가
"AI.JEILM.CO.KR 챗봇 데모의 사내 운영페이지에 추가. ERP 발주구분에서 외주일 경우 검사신청/결과를 적용. 진행사항을 대시보드/현황으로 품질·구매·생산·사업관리가 구매요청/발주 기준으로 확인. 고도화 기획 문서화 후 2차 데모"
2차 고도화 기획서 작성. 사내 대시보드 제작(발주구분·외주 검사신청/결과, 구매요청·발주 기준, 팀 뷰). 챗봇 포털(04)의 부서 운영 페이지에 카드 추가, 공유용 통합본 재빌드.
④ 협력사 등록 내용 ↔ 사내 대시보드 라이브 연동
"기존 모바일 협력사 기능은 유지하되 등록한 내용이 연결·반영되도록. 기존 협력사 모바일 발주사진 포털을 연계해 맥락에 맞게 재검토"
공유 데이터(localStorage, 키 jeilax_link_v1)로 모바일 ↔ 대시보드 실시간 연동 구현. 협력사 등록 사진(증빙)·상태 변경·메시지·검수요청(검사신청 자동생성)이 대시보드에 반영. 발주번호/품목/거래처 정합화. 모바일 포털 standalone 분리.
⑤ ERP 실제 적용 검증 + 전체 프로세스 한눈에 + 모바일 프로세스
"ERP 실제 DB 적용 확인·검토. 대시보드/현황을 한눈에. 구매요청–발주–(협력사: 발주확인·접수·생산·사진등록·검수요청·출하)–입고–매입 전체 프로세스를 발주별로 가시화. 모바일에서도 간단히 데모"
ERP 사전(스키마 단일 진실원천)으로 테이블·컬럼 검증. 전체 10단계 프로세스 보드(JEIL→협력사→ERP 3존)를 발주별 카드로 가시화. 프로세스↔ERP 매핑표 명시. 모바일 발주 상세에 전체 프로세스 스트립 추가.
⑥ 대시보드 사진 샘플 표시
"대시보드에서 사진도 샘플로 넣어서 확인할 수 있게"
발주 카드에 사진 썸네일 갤러리 + 클릭 확대(라이트박스) 추가. 협력사 실제 등록 사진이 있으면 그것을, 없으면 샘플을 표시.
⑦ 모바일 간결화 + 대시보드 조회/구분/현황 개선 + 사진 인식형 자료
"모바일은 협력사 관점으로 간결하게 필요한 내용만. 팀뷰 구분은 나누지 말고, 조회조건 상단에 프로젝트코드·협력사 품목·기간. 발주구분을 나눠서 표시. 대시보드 현황도 직관적으로. 사진 샘플도 이해되도록 사진형 자료"
대시보드 재설계: 팀뷰 제거, 상단 조회조건(프로젝트코드·협력사/품목 검색·기간), 발주구분별 그룹화, KPI 5종 + 진행 단계 분포 막대로 단순화. 사진을 내용 인식형 SVG 일러스트(가공·완성·포장·출하·PCB·검사·베어링·밸브 등)로 교체(모바일·대시보드 공통).
⑧ 발주번호·품목 실제 ERP 형식 + 모바일 메시지 버튼 + 단계취소
"발주번호·품목도 실제 ERP DB 기준 샘플링. 모바일에서 메시지를 보내고 볼 수 있는 버튼 추가해 직접 연결. 모바일 단계취소 기능도 적용"
발주번호 실제 ERP 형식(PO+YYYYMMDD+일련번호), 실제 품목(PLT-AL5T·UNI-25A-S·ORG-P22N·BRG-6204Z 등)·거래처 적용(양쪽 동일 PO 키). 모바일 발주 목록/상세에 메시지 버튼, 단계 취소(이전 단계 복귀, 검사신청 자동취소) 추가.
⑨ 모바일 표시 정돈 + 메시지 양방향(현황 메뉴 버튼)
"모바일 단계취소 버튼·전체 표시를 자연스럽게. 메시지 기능을 현황(대시보드) 메뉴에서 버튼 추가해 적용"
완료 상태 버튼 레이아웃 정돈(글자 깨짐 수정). 메시지 양방향 실시간 동기화 + 대시보드 카드에 💬 메시지 버튼·대화 모달(구매팀 답장) 추가.
⑩ 검수 판정(합/부) + 판정 인원 기록
"현황 메뉴에서 검수요청 단계 항목은 검수 버튼을 눌러 합/부를 선택하고, 합/부를 선택한 인원이 기록에 남도록 생성"
대시보드 검수요청 단계 카드에 🔬 검수 판정 버튼 → 검사자(인원) 입력·합격/불합격 선택 → 판정 결과·판정자·일시·소견을 기록(검수 판정 이력). 합격 시 입고 진행, 불합격 시 보류. 협력사 모바일에 검수 결과 배너 표시·기록 보존.
⑪ 모바일 자연스러운 하이브리드 앱화 + 최신 트렌드 적용
"협력사발주포털은 모바일에서 접속해도 자연스럽게 보이도록 하이브리드 앱 형태로 업데이트하고 최신 트렌드에 맞게 적용"
반응형 전환: 실제 모바일(폭 ≤600px)에서는 폰 목업 베젤·노치를 제거하고 풀스크린(100dvh + safe-area-inset) 엣지투엣지 앱으로 렌더, 데스크톱에서는 발표용 폰 목업 유지. PWA(하이브리드 앱): theme-color·apple-mobile-web-app-capable·status-bar 메타 + 동적 manifest(standalone·앱 아이콘) 주입으로 "홈 화면에 추가" 설치 가능, 로그인 후 설치 배너 노출(iOS는 공유→추가 안내). 모던 인터랙션: 하단 탭바 backdrop-blur·활성 탭 스프링 애니메이션, 동적 뷰포트(dvh)·overscroll 제어·확대 차단. 데모 원본·pages/ 배포본·통합본 동기화 재빌드.

4시스템 구성 · 데이터 연계 구조

ERP(읽기전용)에서 발주·구매요청·입고·매입·검사 정보를 가져오고, 협력사·사내가 생성하는 데이터는 포털 전용 저장소에 둔다. 데모에서는 포털 전용 저장소를 브라우저 localStorage로 구현하여, 같은 사이트(또는 같은 브라우저)에서 두 화면이 실시간 공유한다.

[ERP UNIERP DB] --읽기전용--> [연계 게이트웨이] --> 협력사 모바일 포털 ⇄ 사내 대시보드 └──────── 공유데이터(localStorage: jeilax_link_v1) ────────┘

6ERP 실제 적용 검증 (테이블 매핑)

ERP 사전(스키마 스냅샷, 2026-04-15 기준)으로 아래 테이블·컬럼의 실존을 확인했다. (운영 DB 직접 쿼리는 망 분리로 미수행 — 사전이 스키마 단일 진실원천)

구간 / 항목ERP 테이블 · 컬럼
구매요청M_PUR_REQ (PR_NO·PR_STS·REQ_QTY·ORD_QTY·RCPT_QTY)
발주 / 발주구분M_PUR_ORD_HDR/DTL (PO_NO·RELEASE_FLG) · M_CONFIG_PROCESS (PO_TYPE_CD·PO_TYPE_NM·SUBCONTRA_FLG 사급여부)
입고 · 출하 · 검사판정M_PUR_GOODS_MVMT (MVMT_RCPT_NO 입고번호·MVMT_RCPT_DT·DN_NO 출하번호·INSPECT_REQ_NO·INSPECT_STS·INSPECT_GOOD_QTY/BAD_QTY)
검사 의뢰 / 결과Q_INSPECTION_* (INSP_REQ_NO 검사의뢰번호·INSP_RESULT_NO 검사결과번호·합격/불합격)
매입M_IV_HDR/DTL (IV_QTY)
거래처 / 품목B_BP · B_ITEM
실제 형식·값 적용발주번호는 실제 ERP 형식 PO+YYYYMMDD+일련번호(예: PO202606220012), 구매요청 PR+YYYYMMDD+seq, 검사 IR/IQ+YYYYMMDD+seq. 품목은 실제 ERP 품목(알루미늄 플레이트 5T PLT-AL5T, 유니온 배관 25A SUS UNI-25A-S, 오링 P-22 NBR ORG-P22N, 볼베어링 6204ZZ BRG-6204Z 등)과 실제 거래처(오성테크·이천베아링·신호종합배관·대신스텐레스·지아이텍)를 적용했다.

8데모 실행 · 연동 확인 방법

아래 두 파일을 같은 브라우저에서(또는 같은 사이트에 배포하여) 두 탭으로 열면 실시간 연동을 확인할 수 있다.

📱협력사 모바일 포털 열기02_데모/협력사_모바일_포털.html 🖥️사내 외주 발주·검사 대시보드 열기02_데모/사내_외주발주_검사_대시보드.html
연동 시나리오 예시 ① 모바일에서 발주(예: PO202606200008) 사진 등록·상태를 "검수요청"으로 변경 → 대시보드 해당 카드가 검사 단계로 갱신되고 사진 증빙·검사신청 표시.
② 대시보드에서 그 카드의 🔬 검수 판정 → 검사자 입력·합격 선택 → 프로세스 입고로 진행, 판정 인원 기록.
③ 모바일에 검수 결과 배너 표시. 메시지도 양쪽에서 주고받으면 실시간 반영.

※ 데모는 목업이며 수치·사진은 예시다. 협력사 계정 격리, 실제 사진, 알림 등은 실제 구축 시 적용한다. 내 정보 화면의 "연계 데이터 초기화"로 데모 상태를 리셋할 수 있다.
