협력사 발주·검사 포털 — ERP 연계 실구축 종합기획 (CTO 실행개요)

작성일: 2026-06-26 · 작성: 최동혁 · 단계: 실구축 기획 (데모 → 운영 전환) 대상: 08 협력사 발주·검사 포털의 ERP 실연계 + 운영 가동 관련: 08_협력사발주포털/00_종합정리_요구사항이력.html, 실제구축준비 자료/03·05·07, 루트 CLAUDE.md

1. 한 줄 정의

사내 구매팀이 UNIERP에서 외주 협력사에 낸 발주를 축으로, 협력사(사외)가 모바일로 사진·진행상태·검수요청·메시지를 등록하고, 사내(구매·품질·생산·사업관리)가 10단계 프로세스 보드에서 검수 판정(합/부)·입고·매입을 처리하는 양방향 실시간 포털을, ERP 읽기전용 연계 + 포털 전용 쓰기 DB 구조로 실제 운영에 올린다.

2. 지금이 분기점인 이유 (데모의 한계 = 실구축 과제)

현재 산출물은 동작하는 데모(목업) 다. 두 화면(협력사_모바일_포털.html, 사내_외주발주_검사_대시보드.html)이 브라우저 localStorage(키 jeilax_link_v1)로 한 브라우저 안에서만 실시간 연동된다. 운영으로 가려면 데모가 회피했던 것들을 모두 실체화해야 한다.

데모(현재) 실구축(목표) 본 기획에서 다루는 문서
ERP 수치를 하드코딩/샘플 ERP 읽기전용 중간 DB에서 실데이터 01·02
협력사 ID/PW 가정(hanil-mt) 사외 협력사 인증 경계(2안 비교) + 행수준 보안 03
한 브라우저 localStorage 공유 백엔드 API + 포털 DB로 다기기·다사용자 양방향 04
사진은 메타데이터만, 샘플 SVG 실제 파일 업로드·검증·스토리지·수명주기 03·04
알림 없음 협력사/사내 이벤트 알림(메시지·검수결과) 04
누가 봤는지 기록 없음 감사 로그(개인정보·금액·판정 우선) 03

핵심 통찰: 일반 부서 챗봇·대시보드(03·05·07 문서)는 읽기 전용이면 충분하지만, 협력사 포털은 사외 사용자가 데이터를 생성(쓰기) 한다. 그래서 일반 포털 기획을 그대로 쓸 수 없고, 인증 경계·쓰기 DB·행수준 보안·파일 업로드라는 별도 설계 트랙이 필요하다. 본 기획의 존재 이유가 이것이다.

3. 목표 아키텍처 (한눈에)

[사외 IDC]                          [Azure / ai.jeilm.co.kr]                       [사용자]
┌──────────────┐   야간배치 ETL   ┌───────────────────────────┐
│ UNIERP MSSQL │ ───읽기전용────▶ │ 중간DB erp_ro (발주·검사 사본) │
│  (운영DB)    │   (View 경유)    │                            │
└──────────────┘                  │ 포털DB sub_portal (쓰기)    │   REST/SSE   ┌─────────────┐
                                  │  주문동기화·사진·메시지·검수  │ ◀──────────▶ │ 협력사 모바일 │ (사외)
                                  │                            │   인증경계A   │  PWA        │
                                  │ 백엔드 API (RBAC + RLS)     │              └─────────────┘
                                  │ 파일 스토리지(Blob)         │   REST/SSE   ┌─────────────┐
                                  └───────────────────────────┘ ◀──────────▶ │ 사내 대시보드 │ (사내)
                                                                 Entra SSO    │  (구매·품질) │
                                                                              └─────────────┘

세 가지 데이터 영역으로 분리한다.

  1. ERP 사본(erp_ro) — UNIERP에서 야간 배치로 가져온 발주·구매요청·입고·매입·검사 읽기 전용 데이터. (03·07 문서의 중간 DB 구조를 협력사 포털 영역으로 확장.)
  2. 포털 쓰기 DB(sub_portal) — 협력사·사내가 생성하는 사진 메타·메시지·진행상태·검수요청·검수판정 이력. ERP에 절대 쓰지 않고 발주번호(PO) 키로 매핑.
  3. 파일 스토리지 — 협력사가 올리는 실제 사진 원본. DB에는 메타·경로만.

4. CTO가 지금(착수 전) 정의·결정·구성해야 하는 것 — 요약

상세는 05 문서. 여기서는 "착수를 막는 선행 결정"만 추린다.

4-1. 경영진/관리자 결정 (사람이 정해야 — 코드로 못 푸는 것)

  • D1. 협력사 인증 방식 — Entra External ID(B2B 게스트) vs 협력사 전용 자체 인증. (03 문서 2안 비교, 본 기획에서는 결정 보류 — 비교표로 상신)
  • D2. 협력사 식별 ↔ ERP 거래처(B_BP) 매핑 기준 — 협력사 계정을 어느 ERP 거래처코드에 묶을지(행수준 보안의 근거). 1계정=1거래처? 담당자 다수?
  • D3. 검수 판정의 법적 효력 범위 — 포털 검수 판정(합/부)이 ERP 검사결과(Q_INSPECTION)에 반영되는지/참고용인지. 운영 ERP 쓰기 금지 원칙(§CLAUDE.md 1.2)과 충돌 여부.
  • D4. 사진/증빙의 보존연한·소유권 — 협력사 업로드물의 보관 기간·삭제 정책·분쟁 시 증거 효력.
  • D5. 파일럿 협력사 선정 — 외주 비중 높고 협조적인 1~2개사.

4-2. 유니포인트(ERP 벤더) 협의 (외부 의존 — 조기 발송)

  • 읽기전용 계정·뷰(발주/검사/입출고/매입 한정), 거래처-협력사 매핑 컬럼, 방화벽/VPN, 검사 테이블 스키마 확정. (05 문서 협의표)

4-3. 기술 구성 (착수 후 즉시 — 코드성)

  • 백엔드 API 스택 확정(§CLAUDE.md 3.2 하이브리드 권고), 포털 쓰기 DB 스키마, 파일 스토리지·검증, 인증 경계 분리, 실시간 채널(SSE/폴링), 알림 채널.

6. 전체 로드맵 (요약 — 상세는 06 문서)

전사 AI 포털 16주 마스터(05 문서)의 STEP 5(ERP 연계, W9~12) 에 협력사 포털을 얹되, 협력사 포털 고유 작업(인증 경계·쓰기 DB·파일·실시간)은 별도 트랙으로 W3부터 병행한다.

페이즈 기간(상대) 핵심 산출 게이트
P0 사전정의·협의 W1~2 D1~D5 결정 상신, 유니포인트 협의서 발송, 스키마 확정 경영진 승인·벤더 회신
P1 기반 구축 W3~6 포털 쓰기 DB, 백엔드 API 골격, 인증 경계, 파일 스토리지 로그인·업로드 E2E
P2 ERP 연계 W7~10 발주·검사 야간배치, 대시보드 실데이터, 행수준 보안 실데이터 5일 무오류
P3 실시간·알림 W9~11 localStorage→API 전환, SSE 양방향, 알림 다기기 양방향 검증
P4 파일럿 W12~14 협력사 1~2사 + 사내 2팀 실사용 피드백 반영
P5 오픈·운영 W15~16 전체 협력사 확대, 운영 캘린더, 감사·DR Go-Live 게이트 통과

고정 기한: Entra Client Secret 만료 2026-12-09 — 인증에 Entra를 쓰면(D1) 이 갱신과 일정 연결.

7. 리스크 톱5와 선제 대응

# 리스크 영향 선제 대응
R1 유니포인트 회신 지연 → 거래처-협력사 매핑·뷰 미확정 ERP 연계 전체 지연 P0에서 즉시 협의서 발송, ERP 트랙 독립화, 회신 전까지 스키마 사전 기반 목 데이터로 선개발
R2 사외 협력사 인증·행수준 보안 허점 → 타사 발주 노출 보안 사고·신뢰 상실 권한 판정 백엔드/DB RLS에서, 프론트 토큰만으로 데이터 미제공(§CLAUDE.md 5.4), 모의 침투 테스트
R3 파일 업로드 악성/대용량 스토리지·보안 확장자·MIME·용량·이미지 재인코딩 검증, 별도 스토리지, 바이러스 스캔
R4 검수 판정의 ERP 반영 기대 ↔ 운영 DB 쓰기 금지 충돌 업무 혼선 D3 사전 합의 — 포털은 "판정 기록", ERP 반영은 사내 담당이 ERP에서 별도 수행
R5 OneDrive 동기화 지연로 배포 충돌(§CLAUDE.md 8.3) 라이브 깨짐 /tmp clone→빌드→푸시 절차 준수, 통합본은 빌드 스크립트로만

8. 비기능 요구(NFR) 기준선

  • 성능: 운영 ERP 부하 0(야간 배치만). 대시보드 조회 P95 < 1.5s. 모바일 사진 업로드 10MB 이내, 서버 재인코딩.
  • 가용성: 포털 99.5%/월. 중간 DB 장애 시 ERP와 격리(재배치 복구).
  • 보안: 전송 HTTPS 강제, 비밀값 Key Vault, 행수준 보안, 감사 로그 1년 보존.
  • 확장성: 협력사 수십~수백 계정, 발주 월 수백~수천 건 가정. 사진 누적 증가 → 스토리지 수명주기 정책.
  • 데이터 신선도: 발주·검사 ERP 데이터 "기준 시각" 화면 표기. 협력사 등록(쓰기)은 실시간.
