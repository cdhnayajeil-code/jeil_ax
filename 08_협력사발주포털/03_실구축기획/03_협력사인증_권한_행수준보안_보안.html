03. 협력사 인증 · 권한 · 행수준 보안 · 보안

작성일: 2026-06-26 · 작성: 최동혁 · 대상 페이즈: P1, P5 관련: 00·01·02, 루트 CLAUDE.md §5·§6 핵심 결정: 협력사 인증 방식(D1)은 본 기획에서 결정 보류 — 아래 2안 비교를 경영진에 상신한다.

1. 보안 설계의 출발점 — "사외 사용자가 쓰기를 한다"

전사 포털 보안은 "사내 직원이 읽는다"를 전제한다. 협력사 포털은 사외(통제 밖) 사용자가 데이터를 생성·업로드하므로, 다음 3중 방어가 필수다.

  1. 인증 경계 분리 — 협력사를 사내 Entra 테넌트에 섞지 않는다(§CLAUDE.md 5.5).
  2. 행수준 보안(RLS) — 협력사는 자기 거래처 발주만. 백엔드/DB에서 강제.
  3. 입력·파일 검증 — 업로드물은 모두 적대적 입력으로 간주.

2. 협력사 인증 — 2안 비교 (★결정 보류, 상신용)

기준 A안. Entra External ID / B2B 게스트 B안. 협력사 전용 자체 인증
개념 MS Entra의 외부 사용자(게스트) 기능으로 협력사를 초대, MS 인증 흐름 사용 포털이 자체 ID/PW + OTP(MFA) 발급·관리
사내 SSO 통합 ◎ 동일 Entra 관리 콘솔에서 통합 관리 △ 별도 사용자 저장소 운영
MFA ◎ 조건부 액세스로 일괄 적용 ○ 자체 OTP 구현 필요
협력사 진입장벽 △ MS 계정/초대 수락 절차, 비IT 협력사 혼란 가능 ◎ 익숙한 ID/PW, 진입 단순
비밀번호·계정 관리 부담 ◎ MS가 담당(재설정·잠금) △ 자체 구현(재설정·잠금·정책)
보안 성숙도 ◎ 검증된 IdP, 토큰·세션 표준 △ 자체 구현 품질에 의존
사내 테넌트 영향 게스트가 디렉터리에 들어옴(분리 정책 필요) 사내 테넌트 완전 무관
비용 Entra External ID 과금(MAU 기반) 확인 필요 구현·운영 인건비
오프보딩 ◎ 게스트 제거로 즉시 차단 ○ 계정 비활성
권고 적합 거래처 수 적당·MS 친화·관리 일원화 우선 협력사 다수·비IT·진입 단순 우선

공통 (어느 안이든)

  • 토큰/세션은 HttpOnly 쿠키 + 백엔드 세션(BFF) 권장(데모 localStorage는 XSS 노출, §CLAUDE.md 6).
  • 로그인 실패 제한·세션 만료·디바이스 바인딩.
  • 인증 주체(auth_subject)를 sub_meta.supplier_account에 저장해 RLS 컨텍스트로 사용.

상신 포인트 (경영진 결정 D1)

"협력사 수·IT 친숙도·관리 일원화 vs 진입 편의 중 무엇을 우선하는가?" 에 따라 A/B 결정. 결정 전까지 인증 어댑터 인터페이스(resolveSupplier(authContext) → account_id)로 추상화해, 어느 안이 채택돼도 비즈니스 코드는 불변이 되게 한다.

3. 권한 모델 (RBAC) — 사내/사외 분리

역할 경계 접근
AI-Portal-Subcontractor(협력사) A /supplier/* — 자기 거래처 발주만, 사진·상태·검수요청·메시지 쓰기
AI-Portal-Purchase(구매) B /internal/* — 전체 발주 보드, 메시지 답장
AI-Portal-Quality(품질) B 검수 판정(합/부) 권한
AI-Portal-Production(생산)·-PM(사업관리) B 현황 조회
AI-Portal-Admins B 전체 + 계정·매핑·감사

원칙: 메뉴 숨김은 UX일 뿐, 실제 차단은 API/DB에서(§CLAUDE.md 5.4). 프론트 토큰만으로 민감 데이터를 내려주지 않는다.

4. 행수준 보안(RLS) — 협력사 격리의 핵심

4-1. 흐름

협력사 로그인(경계 A) → auth_subject → supplier_account.account_id
   → supplier_bp_map 에서 허용 bp_cd 집합 S 조회
   → 모든 /supplier 조회에 강제 필터: erp_ro.po_header.BP_CD ∈ S
   → 쓰기도 동일 PO가 S에 속할 때만 허용

4-2. 2중 강제 (애플리케이션 + DB)

  • 애플리케이션 계층: 모든 /supplier 쿼리에 bp_cd 필터 자동 주입(우회 불가하게 공통 미들웨어).
  • DB 계층(권장): PostgreSQL Row Level Security 정책으로, 세션 변수 app.supplier_bp에 속하지 않는 행은 원천 차단.
ALTER TABLE erp_ro.po_header ENABLE ROW LEVEL SECURITY;
CREATE POLICY supplier_isolation ON erp_ro.po_header
  USING (bp_cd = ANY (current_setting('app.supplier_bp', true)::text[]));
  • 효과: 애플리케이션 버그로 필터를 빠뜨려도 DB가 한 번 더 막는다. 협력사 A가 협력사 B의 발주를 구조적으로 볼 수 없다.

4-3. 검증 (반드시 테스트)

  • 협력사 계정으로 타 거래처 PO 직접 호출 → 403/빈 결과 확인.
  • 매핑 없는 계정 → 전체 빈 결과.
  • 사진/메시지 쓰기도 자기 PO 외 차단.

5. 파일 업로드 보안 (사진 증빙)

협력사가 올리는 사진은 적대적 입력으로 간주한다.

단계 통제
수신 확장자 화이트리스트(jpg/png/heic), MIME 검사, 용량 상한(예 10MB), 매수 제한
검증 실제 이미지 디코드 성공 여부(위장 파일 차단), EXIF 제거(위치정보·민감)
재인코딩 서버에서 리사이즈·재인코딩(임베디드 페이로드 무력화)
저장 별도 Blob 스토리지(웹 루트 밖), DB엔 경로·메타만
노출 직접 URL 금지 → 단기 서명 URL(만료) 발급, 권한 확인 후
악성 바이러스 스캔(가능 시), 실행 권한 제거
수명주기 보존연한(D4) 후 자동 아카이브/삭제, 발주 종결 후 정책

6. 감사·로깅·전송 보안

  • 감사 로그(audit_log): 누가·언제·어떤 PO/화면/사진/판정을 봤·했는지 전건. 개인정보·금액·검수판정 우선. 1년 보존(§CLAUDE.md 6).
  • 검수 판정 비가역 기록: 판정자·일시·소견을 inspection_log에 누적(수정 불가, 정정은 신규 행).
  • 전송 보안: HTTPS 강제(isHttps() 참일 때만 동작), HSTS.
  • 비밀값: 인증 시크릿·DB 접속·스토리지 키는 Key Vault만. 코드·Git·문서 금지(§CLAUDE.md 1.1, 8.2).
  • XSS/인젝션: 메시지·코멘트 출력 인코딩, 쿼리 파라미터 바인딩만(§CLAUDE.md 4.5).

7. 위협 모델 (요약)

위협 방어
협력사 A가 B의 발주 열람 RLS 2중(앱+DB), 직접 호출 테스트
토큰 탈취(XSS) HttpOnly 쿠키/BFF, 출력 인코딩, CSP
악성 파일 업로드 검증·재인코딩·별도 스토리지·스캔
권한 상승(협력사→사내 API) 경계 분리, 엔드포인트별 역할 검사
대량 스크래핑 행 수 상한·레이트리밋·감사
오프보딩 누락(퇴사·계약종료) 분기 계정·매핑 정합성 점검, 즉시 비활성 절차
