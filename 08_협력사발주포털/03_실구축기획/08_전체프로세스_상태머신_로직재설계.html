08. 전체 프로세스 상태머신 — 로직 재설계 (검수 합격/불합격 흐름)

작성 2026-07-07 · 관리 최동혁 · 상태: Active (현행 기준) 목적: 검수 불합격 후 흐름이 막히는 구조적 결함을 해소하기 위해, 표준(SAP QM·ISO 9001 8.7) 벤치마킹 기준으로 발주 전체 프로세스의 상태머신을 재정의하고 시스템(포털 DB·api.js·사내 대시보드·협력사 포털)에 적용한 기준 문서.

1. 왜 재설계했나 — 현행 로직의 구조적 결함

전수 조사 결과, 판정(합/부)과 발주 상태가 완전히 분리된 두 축이어서 아래 결함이 확인됐다.

# 결함 내용
R1 판정이 상태를 안 바꿈 judge()가 sp_inspection.result만 기록하고 sp_order_state.status는 insp 그대로 → 합격해도 완료가 아니고, 불합격해도 아무 전이가 없음
R2 불합격 교착(사용자 보고 버그) 대시보드 판정 버튼 조건이 검수요청 단계 && 불합격 아님 → 불합격 판정 즉시 버튼이 사라지고, 재검수요청을 해도 옛 불합격 판정이 리셋되지 않아 영구히 재판정 불가
R3 합격 없이 완료 가능 협력사가 검수요청 상태에서 "다음 단계" 클릭만으로 done 전이 — 품질 판정과 무관하게 완료됨
R4 모순 표시 불합격 배너와 "품질팀 검사 대기중" 배너가 동시 표출, 완료+불합격 시 KPI 이중 계상
R5 완료 후 되돌리기 허용 협력사가 done→insp 단계취소 가능(종결 상태가 아님)
R6 step 컬럼 고아화 sp_order_state.step 숫자가 화면 10단계 인덱스와 불일치·미사용(write-only)

2. 벤치마킹 요약 (표준 근거)

  • SAP QM (가장 정형화된 레퍼런스): 입고 시 검사로트 자동 생성 → 결과기록 → Usage Decision(사용결정=판정) 1회 → 판정 코드에 따라 재고 전기(합격=가용재고, 불합격=차단재고·반품·폐기). 재검사는 판정을 덮어쓰지 않고 새 검사로트를 발번.
  • ISO 9001 8.7 (부적합 산출물): 불합격품은 식별·격리 후 재작업/수리/특채/반품/폐기로 분기, 재작업 후 재검증(재검사) 필수, 모든 판정은 판정자·일시·사유와 함께 불변 이력 보존.
  • 핵심 불변식: ① 검사의뢰 1건 = 판정 1회 — 재검사는 차수를 올린 새 의뢰. ② 발주에는 현재 상태 1개, 판정 상세는 append-only 이력 테이블로 분리. ③ 특채는 별도 상태가 아닌 합격의 하위 판정코드(상태 폭발 방지).

3. 목표 상태머신 (v2 — 5상태, 단순 유지)

기존 4상태(new/prod/insp/done)의 맥락을 유지하고 rework(재작업) 1개만 추가한다.

                  협력사                       협력사(검수요청)                 사내 판정(합격)
   new(신규접수) ────────► prod(생산중) ─────────────────────► insp(검사대기) ──────────► done(완료)
        ▲    취소           ▲   ▲                                  │    ▲                   [종결]
        └───────────────────┘   │ 취소(재작업 보류)                 │    │
                                │                          사내 판정(불합격)│ 협력사(재검수요청, 차수+1)
                                └────────────── rework(재작업) ◄───┘    │
                                                     └──────────────────┘
   ※ DIV(일반·무검사): prod ──협력사(완료 처리)──► done   (검사 생략 유형 — SIE 계열만 검수 필수)

3.1 전이표 (전체 — 이것만이 허용되는 전이)

From To 트리거 주체 시스템 동작
new prod 생산 시작 협력사 status 갱신
prod insp 검수요청 협력사 검사신청 IR 발번(차수 접미사) + 현재 판정 리셋 + status=insp
prod done 완료 처리 (DIV 무검사 전용) 협력사 status=done
insp done 판정=합격 사내(품질) 판정 기록(현재+이력) + status=done 자동 전이
insp rework 판정=불합격 사내(품질) 판정 기록(현재+이력) + status=rework 자동 전이
rework insp 재작업 완료 → 재검수요청 협력사 새 IR 발번(차수+1) + 현재 판정 리셋 + status=insp
insp prod 검수요청 취소(판정 전만 가능) 협력사 검사신청 cancelled=true + status=prod
prod new 단계 취소 협력사 status 갱신
rework prod 재작업 보류(생산중으로 표기) 협력사 status 갱신
done — 전이 없음(종결) — 협력사 되돌리기 불가. 예외 처리는 사내 관리자

3.2 불변식 (Invariant)

  1. done은 오직 ① 합격 판정 ② DIV 무검사 완료로만 진입 — 협력사가 검사 유형(SIE 계열)에서 임의 완료 불가.
  2. 판정은 status=insp에서만 가능하고, 판정 즉시 상태가 자동 전이되므로 "insp + 판정 있음" 상태는 존재하지 않는다 → 판정 버튼 조건 = status==='insp' && 현재 판정 없음(교착 원천 차단).
  3. 재검수요청은 반드시 현재 판정을 리셋한다(sp_inspection 행 삭제). 판정 이력은 sp_inspection_log(append-only)에 영구 보존 — 검사의뢰 1건=판정 1회 불변식.
  4. 검사 차수 = 해당 발주의 검사신청 누적 건수. IR번호는 IR+일자+거래처 + 차수 접미사(-2, -3…).
  5. sp_order_state.step은 대시보드 10단계 인덱스(0~9)와 동일 기준으로 기록(참고용 — 화면은 status로 판단).

3.3 상태 정의 (기초 세팅)

status 이름 의미 10단계 표시(step) 협력사 가능 액션 사내 가능 액션
new 신규 접수 발주 확인·접수 대기 3 접수 생산 시작 메시지
prod 생산중 제작·사진 등록 4 생산(사진 있으면 5) 사진 등록·검수요청(SIE)·완료(DIV)·취소 사진 검토·메시지
insp 검사 대기 검수요청됨·판정 대기 6 검수요청 취소(판정 전) 검수 판정(합/부)
rework 재작업(불합격) 불합격 → 협력사 재작업 4 생산(재작업 표시) 사진 재등록·재검수요청·보류 사진 검토·메시지
done 완료 합격(또는 DIV 무검사) → 입고·매입 9 매입 (없음 — 종결) (없음)

4. 데이터 규칙 (테이블별)

테이블 역할 규칙
sp_order_state 발주 현재 상태 1행 status ∈ 5상태(check 제약), 전이는 §3.1만 허용
sp_insp_request 검사의뢰(차수 관리) append-only + cancelled 플래그. 재요청=새 행(차수+1)
sp_inspection 현재 유효 판정 1행 판정 시 upsert, 재검수요청 시 삭제(리셋)
sp_inspection_log 판정 불변 이력 append-only — 수정·삭제 금지, 판정자·일시·소견 보존

5. 화면 규칙

사내 대시보드: 판정 버튼 = insp && 판정 없음(불합격 후 재검수요청 오면 다시 활성 — R2 해소). rework는 "재작업 (불합격)" 배지 + 생산 단계 위치. 합격은 status=done으로 "완료" 집계(KPI 이중 계상 해소). 판정 이력은 모달에서 전체 표시.

협력사 포털: insp에서는 "검사 대기" 배너만(다음 단계 버튼 없음 — R3 해소). rework에서는 불합격 배너(판정자·소견) + "▶ 재작업 완료 → 재검수요청" 버튼. done은 종결(버튼 없음 — R5 해소). DIV는 검수요청 대신 "완료 처리(무검사)".

7. 검증 시나리오 (전체 흐름이 끊기지 않는지)

  1. 정상 합격 흐름: new → 생산 시작 → 사진 등록 → 검수요청(IR 발번) → 사내 합격 판정 → 자동 done → 협력사 화면 종결 표시.
  2. 불합격→재검수 루프(핵심): 검수요청 → 불합격 판정(사유) → 자동 rework + 협력사 불합격 배너 → 사진 재등록 → 재검수요청(IR-2 발번, 판정 리셋) → 대시보드에 판정 버튼 재활성 → 합격 → done. (기존: 불합격 시 영구 교착)
  3. 가드: insp에서 협력사 "다음 단계" 버튼 없음 / done에서 취소 버튼 없음 / DIV는 검수요청 없이 완료 처리.
  4. 판정 이력: 불합격→합격 2건이 판정자·일시와 함께 모달에 누적 표시.

8. 확장 여지 (이번 범위 제외 — 단순성 유지)

  • 불합격 처리 유형 분기(반품/폐기/특채) — 현재는 재작업 단일 경로. 필요 시 sp_inspection_log에 처리유형 컬럼 추가로 확장(특채는 합격 하위 코드로).
  • 검사중(INSP) 상태 분리(현재는 검사대기=검사중 통합), 부분합격(수량 분할).
  • ERP Q_INSPECTION 실연계 시 본 상태머신의 insp/판정 축을 ERP 검사의뢰·판정으로 대체(§4 매핑 유지).
