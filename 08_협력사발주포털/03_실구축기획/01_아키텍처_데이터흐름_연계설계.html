01. 아키텍처·데이터흐름·연계 설계

작성일: 2026-06-26 · 작성: 최동혁 · 대상 페이즈: P1~P3 관련: 00 종합기획, 02 데이터모델, 실제구축준비 자료/07_ERP_챗봇_중간DB_연계설계

1. 설계 원칙 (협력사 포털 특화)

전사 AI 포털의 일반 원칙(읽기전용·중간DB·화이트리스트)을 계승하되, 사외 사용자가 쓰기를 한다는 점에서 다음 4개 원칙을 추가한다.

원칙 내용 근거
P-1 읽기/쓰기 물리 분리 ERP 사본은 읽기전용(erp_ro), 협력사·사내 생성물은 별도 쓰기 DB(sub_portal). 두 영역은 PO(발주번호) 키로만 조인 CLAUDE.md §4.3
P-2 인증 경계 이중화 사내(Entra SSO)와 사외(협력사) 인증을 물리적으로 분리. 협력사를 사내 테넌트에 섞지 않음 CLAUDE.md §5.5
P-3 행수준 보안(RLS) 협력사는 자기 거래처 발주만. 권한 판정은 백엔드/DB에서, 프론트는 UX만 CLAUDE.md §5.4
P-4 ERP 무영향 협력사 포털의 어떤 동작도 운영 ERP에 쓰지 않음. 검수 판정조차 포털 DB 기록 CLAUDE.md §1.2

2. 3계층 + 인증 경계 구조

┌─ 계층 1: 원천 (사외 IDC, 유니포인트 관리) ──────────────────────────┐
│  UNIERP MSSQL — 발주/구매요청/입고/매입/검사 운영 데이터            │
│  접근: 읽기전용 계정(jeilax_ro) + View 경유 + 야간배치 1회          │
└────────────────────────────────┬───────────────────────────────────┘
                                 │ VPN/TLS · 증분 추출
                                 ▼
┌─ 계층 2: 연계·애플리케이션 (Azure / ai.jeilm.co.kr) ────────────────┐
│  ┌──────────────┐   ┌────────────────────────────────────────────┐ │
│  │ ETL 배치 Job │──▶│ PostgreSQL                                 │ │
│  │ 매일 03:00   │   │  erp_ro     — 발주·검사 ERP 사본(읽기)      │ │
│  └──────────────┘   │  sub_portal — 사진·메시지·상태·검수(쓰기)   │ │
│                     │  sub_meta   — 동기화 로그·감사·알림 큐      │ │
│  ┌──────────────────┴────────────────────────┐                    │ │
│  │ 백엔드 API (RBAC + RLS + 화이트리스트)      │  ┌──────────────┐ │ │
│  │  /supplier/* (사외)   /internal/* (사내)   │──│ Blob 스토리지 │ │ │
│  │  실시간 채널(SSE)  ·  알림 디스패처         │  │ 사진 원본     │ │ │
│  └───────┬───────────────────────┬───────────┘  └──────────────┘ │ │
└──────────│───────────────────────│──────────────────────────────────┘
   인증경계 A│ (협력사)       인증경계 B│ (사내 Entra SSO)
            ▼                       ▼
┌─ 계층 3: 클라이언트 ────────────────────────────────────────────────┐
│  협력사 모바일 PWA          사내 대시보드(구매·품질·생산·사업관리)   │
└─────────────────────────────────────────────────────────────────────┘

핵심은 하나의 백엔드 API가 두 인증 경계를 가진다는 것이다. /supplier/* 엔드포인트는 협력사 인증(경계 A)을 통과한 요청만, /internal/* 는 Entra SSO(경계 B)를 통과한 요청만 받는다. 같은 발주(PO)를 양쪽이 보지만, 무엇을 볼/쓸 수 있는지는 경계와 RLS가 다르게 통제한다.

3. 일반 포털과의 차이 (왜 별도 설계인가)

항목 일반 부서 포털(03·07) 협력사 발주·검사 포털(본 기획)
사용자 사내 직원만(Entra) 사내 + 사외 협력사
데이터 방향 읽기 전용 읽기(ERP) + 쓰기(협력사·사내 생성물)
DB erp_ro/erp_secure erp_ro + sub_portal(쓰기)
인증 단일(Entra) 이중 경계(사내 Entra + 사외 협력사)
보안 핵심 그룹 인가(RBAC) RBAC + 행수준 보안(RLS) + 파일 검증
동기화 야간 배치(읽기) 야간 배치(ERP) + 실시간(쓰기 양방향)
파일 없음/문서 인덱싱 사진 업로드·스토리지·수명주기

→ 결론: 협력사 포털은 읽기 파이프라인은 일반 포털과 공유하되, 쓰기·인증·파일·실시간은 신규 트랙으로 설계한다. 본 문서·02·03·04가 그 신규 트랙이다.

4. 데이터 흐름 — 5개 시나리오

4-1. ERP 발주 → 협력사·대시보드 노출 (읽기, 야간)

UNIERP M_PUR_ORD ──배치──▶ erp_ro.po_header/line ──API──▶ 협력사 목록 / 대시보드 보드
                                  │
                          (거래처-협력사 매핑으로 RLS 적용)

협력사는 자기 거래처(B_BP)에 묶인 발주만 목록에서 본다. 사내는 발주구분(SIE/SIF/DIV) 그룹으로 전체를 본다.

4-2. 협력사 사진 등록 (쓰기, 실시간)

협력사 모바일 ──multipart──▶ /supplier/orders/{po}/photos
   → 파일 검증(확장자·MIME·용량·재인코딩) → Blob 저장
   → sub_portal.photo(메타·경로·태그·등록자) INSERT
   → SSE 이벤트 push → 사내 대시보드 썸네일 즉시 갱신

4-3. 진행상태 변경·검수요청 (쓰기, 상태머신)

협력사 "검수요청" ──▶ /supplier/orders/{po}/status
   → 상태머신 전이 검증(생산중→검수요청 허용?) 
   → sub_portal.order_state UPDATE + 검사신청 자동생성(inspReqNo)
   → SSE → 대시보드 카드 "검사 단계"로 이동
   ※ 단계 취소 시 역전이 + 검사신청 자동 철회

4-4. 사내 검수 판정 (쓰기, 이력)

대시보드 "🔬 검수 판정" ──▶ /internal/orders/{po}/inspection
   → 검사자(인원)·합/부·소견 입력
   → sub_portal.inspection + inspection_log(판정자·일시) INSERT
   → 합격 시 프로세스 입고 진행 / 불합격 보류
   → SSE → 협력사 모바일 검수결과 배너
   ※ ERP Q_INSPECTION 에는 쓰지 않음(D3 결정 따름)

4-5. 양방향 메시지 (쓰기, 실시간)

협력사/사내 메시지 ──▶ /{role}/orders/{po}/messages
   → sub_portal.message INSERT(발신주체·본문)
   → SSE → 상대 화면 즉시 + 알림 큐 적재

5. 연계 게이트웨이 — 책임 분리

백엔드 API는 다음 5개 책임 모듈로 나눈다. (스택 미정 — §CLAUDE.md 3.2 하이브리드: ERP 읽기는 .NET 친화, 포털 쓰기·실시간은 Node/Next 친화. 인터페이스 추상화로 교체비용 최소화.)

모듈 책임 비고
AuthN/AuthZ 인증 경계 A/B 검증, 세션, RBAC, RLS 컨텍스트 주입 03 문서
ERP Query erp_ro 화이트리스트 조회만. 자유 SQL 금지 02·07
Portal Write sub_portal 쓰기, 상태머신 전이 검증, 트랜잭션 02·04
File 업로드 검증·재인코딩·Blob 저장·서명 URL 발급·수명주기 03·04
Realtime/Notify SSE 채널, 알림 큐 디스패치(웹푸시·메일·Teams) 04
