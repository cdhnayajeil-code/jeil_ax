# 04. 실시간 동기화 · API 설계 · 파일 업로드 · 알림

> 작성일: 2026-06-26 · 작성: 최동혁 · 대상 페이즈: P1, P3
> 관련: 01 아키텍처, 02 데이터모델, 03 보안
> 목표: 데모의 `localStorage(jeilax_link_v1)` 한 브라우저 공유를, **백엔드 API + 다기기·다사용자 실시간 양방향**으로 대체한다.

---

## 1. 무엇을 대체하나 — 데모 → 운영

| 데모 메커니즘 | 한계 | 운영 대체 |
|---|---|---|
| `localStorage` 공유 객체 | 같은 브라우저 안에서만 | 백엔드 DB(`sub_portal`) 단일 진실원천 |
| `storage` 이벤트로 타 탭 갱신 | 같은 origin·기기만 | **SSE(서버→클라이언트 push)** 로 모든 접속 기기 갱신 |
| 사진 메타만 공유(용량) | 실제 사진 없음 | **Blob 업로드 + 서명 URL** |
| 알림 없음 | 상대가 봐야 인지 | **알림 큐**(웹푸시·메일·Teams) |

---

## 2. 실시간 채널 — 방식 비교 및 권고

| 방식 | 장점 | 단점 | 적합도 |
|---|---|---|---|
| **SSE(Server-Sent Events)** | 단방향 push 단순, HTTP 기반, 모바일/프록시 친화, 재연결 자동 | 서버→클라 단방향(클라→서버는 일반 REST) | ◎ **권고** — 본 포털은 "쓰기는 REST, 갱신 알림은 push" 패턴에 정확히 부합 |
| WebSocket | 양방향 풀듀플렉스 | 인프라·상태관리 복잡, 프록시 이슈 | △ 과한 복잡도 |
| 폴링(주기 조회) | 가장 단순 | 지연·부하·배터리 | ○ SSE 미지원 환경 폴백 |

**권고 구조:** 클라이언트는 쓰기를 **REST**로 보내고, 갱신은 **SSE 구독**으로 받는다. 발주(PO) 또는 사용자 단위 채널로 이벤트를 라우팅한다. SSE 불가 환경(일부 사내 프록시)은 **15~30초 폴링 폴백**.

```
[협력사 사진 등록] --POST--> 백엔드 --INSERT--> sub_portal.photo
                                  └--SSE event(po:PO...,type:photo)--> 구독 중인 대시보드 즉시 갱신
                                  └--notify_queue 적재--> 디스패처 --> 사내 알림
```

## 3. API 명세 (초안)

경계별로 prefix 분리. 모든 응답에 ERP 데이터는 "기준 시각" 포함. 모든 호출은 인증·RBAC·RLS·감사를 통과.

### 3-1. 협력사 (경계 A, `/supplier`)
| 메서드·경로 | 설명 | 비고 |
|---|---|---|
| `GET /supplier/orders` | 자기 거래처 발주 목록 | RLS 필터 자동 |
| `GET /supplier/orders/{po}` | 발주 상세 + 10단계 위치 | |
| `POST /supplier/orders/{po}/photos` | 사진 업로드(multipart) | 파일 검증(03) |
| `DELETE /supplier/orders/{po}/photos/{id}` | 사진 삭제 | 본인 등록분 |
| `PATCH /supplier/orders/{po}/status` | 진행상태 전이(다음/취소) | 상태머신 검증(02) |
| `POST /supplier/orders/{po}/insp-request` | 검수요청(검사신청 자동생성) | 외주만 |
| `GET/POST /supplier/orders/{po}/messages` | 메시지 조회/발신 | |
| `GET /supplier/stream` | SSE 구독(자기 PO 이벤트) | |

### 3-2. 사내 (경계 B, `/internal`)
| 메서드·경로 | 설명 | 비고 |
|---|---|---|
| `GET /internal/orders` | 조회조건(프로젝트·품목·기간·발주구분) 보드 | 발주구분 그룹 |
| `GET /internal/orders/{po}` | 발주 상세 + 사진 + 검사 | |
| `POST /internal/orders/{po}/inspection` | 검수 판정(합/부·판정자·소견) | 품질 역할(03) |
| `GET/POST /internal/orders/{po}/messages` | 메시지 답장 | |
| `PATCH /internal/photos/{id}/confirm` | 사진 확인 처리 | |
| `GET /internal/stream` | SSE 구독(전체/담당 이벤트) | |
| `GET /internal/dashboard/kpi` | KPI 5종 + 단계 분포 | 집계 |

### 3-3. 공통 규약
- 인증: 경계별 세션 쿠키(BFF). 권한: 엔드포인트별 역할 + RLS.
- 멱등성: 사진 업로드·상태 전이에 클라이언트 토큰으로 중복 방지.
- 에러: 표준 코드(401 인증·403 권한/RLS·409 상태충돌·413 용량초과·422 검증).
- 레이트리밋: 협력사 업로드·조회 한도. 행 수 상한.

## 4. 파일 업로드 파이프라인

```
모바일 선택/촬영
  → (클라) 미리보기·기본 리사이즈
  → POST multipart  /supplier/orders/{po}/photos
  → (서버) 확장자·MIME·용량 검증 → 이미지 디코드 확인 → EXIF 제거·재인코딩
  → Blob 저장(웹루트 밖)  → sub_portal.photo INSERT(경로·태그·등록자)
  → SSE(photo) → 대시보드 썸네일 갱신
  → 응답: photo id + 단기 서명 URL
조회 시: 서명 URL 재발급(권한 확인 후, 만료 짧게)
```
세부 보안 통제는 03 문서 §5. 스토리지 선택(Azure Blob vs 사내)은 호스팅 결정과 연동(01 인프라).

## 5. 알림 (Notification)

업무 단절(전화·메신저 분산) 해소가 본 프로젝트 동기이므로, **핵심 이벤트는 알림으로 밀어준다.**

| 이벤트 | 수신자 | 채널(우선순위) |
|---|---|---|
| 협력사 사진 등록·검수요청 | 사내 구매·품질 | 대시보드 배지 → Teams/메일 |
| 사내 검수 판정(합/부) | 협력사 | 모바일 웹푸시(PWA) → 메일 |
| 신규 메시지(양방향) | 상대 | 인앱 배지 → 푸시 |
| 발주 신규(야간배치 후) | 협력사 | 인앱 → (선택)메일 |
| 출하 지연·납기 임박 | 양측 | 일배치 알림 |

구현: `notify_queue` 적재 → 디스패처가 채널별 발송·재시도·결과 기록. PWA 웹푸시는 권한 동의 기반(데모의 PWA/standalone 계승). 과다 알림 방지를 위해 묶음·요약 정책.

## 6. 동기화 정합성·충돌 처리
- **단일 진실원천:** 모든 쓰기는 백엔드 DB. 클라이언트 상태는 캐시일 뿐.
- **낙관적 동시성:** 상태 전이·검수 판정에 버전/타임스탬프 검사 → 동시 변경 시 409 후 재조회.
- **순서 보장:** 메시지·이벤트에 서버 시퀀스. SSE 끊김 시 마지막 이벤트 ID로 재동기화.
- **신선도 표기:** ERP 사본 데이터엔 "기준 시각", 협력사 쓰기엔 실시간 반영.
