## 1. 절대 규칙 (NEVER — 위반 금지)

1. **비밀값을 코드/커밋에 절대 넣지 않는다.** Entra Client Secret, MS SQL 접속 계정·비밀번호, 연결 문자열, API 키, 토큰, 내부 서버 IP를 소스에 하드코딩하거나 커밋하지 않는다. 반드시 환경변수(`.env`)·시크릿 매니저로 분리한다.
2. **ERP(UNIERP) 운영 MS SQL에 직접 쓰기/직접 조회를 하지 않는다.** 읽기 전용 + 중간 DB 경유만 허용(§4). 가상드라이브 우회, 운영 DB 직접 SELECT, DDL/DML은 금지.
3. **`MS_connect/`, `doc/`, `.claude/` 폴더 내용을 커밋하거나 공개 배포본에 포함하지 않는다.** (Entra 식별자·Secret·내부 작업현황 포함. `.gitignore`로 제외됨 — 해제 금지.)
4. **새로 만드는 문서·코드에 Entra Tenant/Client ID, Secret 값을 적지 않는다.** 기술 원본 값은 `MS_connect/`에만 둔다.
5. **배포(특히 OneDrive 동기화 폴더)에서 작업 폴더를 직접 `git push` 빌드하지 않는다.** OneDrive(E:) ↔ 빌드환경 동기화가 수십 분 지연될 수 있음 → §8 배포 절차를 따른다.
6. **금전·발주 확정·전자결재 등 실거래 액션을 사용자 대신 실행하지 않는다.** 데이터 조회·집계·초안 작성까지만. 최종 승인 클릭은 사람이 한다.
7. **개인정보(급여·주민번호·인사평가 등)는 권한 그룹(§5) 밖으로 노출되는 화면/로그/캐시에 남기지 않는다.**

## 3. 아키텍처 방향 (스택 미정 — 결정 기준과 권고)

> 스택은 아직 확정 전이다. 아래 권고를 **기본값**으로 삼되, 확정되면 본 절을 갱신한다.

### 3.1 권고 구조 (3계층 분리)

```
[브라우저/모바일]  ── Entra(Teams) 로그인 ──►  [백엔드 API]  ── 읽기전용 ──►  [중간 DB(복제본)]  ◄─ 야간배치 ─  [ERP 운영 MSSQL(사외 IDC)]
   프론트엔드                                  인증·권한·집계                  포털 전용 데이터(쓰기 가능)
```

핵심은 **프론트가 ERP/MSSQL에 직접 붙지 않는다**는 것. 반드시 **백엔드 API 계층**을 둔다. 현재 데모는 프론트만 있고 이 계층이 없음 → 실구축의 1순위 신설 대상.

### 3.2 스택 후보 비교 (결정 시 이 표 기준)

| 항목 | 후보 A: .NET/C# + MSSQL | 후보 B: Node/Next.js + Postgres 중간DB |
|---|---|---|
| ERP MSSQL 친화 | ◎ (동일 생태계, EF/ADO 직결) | ○ (드라이버 경유) |
| Entra/Teams 연동 | ◎ (MSAL.NET 성숙) | ○ (MSAL.js/next-auth) |
| 사내 IIS/Windows 호스팅 | ◎ | △ |
| 프론트 개발 속도·UI | ○ | ◎ (현 HTML 데모 계승 쉬움) |
| 클라우드(Azure) 배포 | ◎ | ◎ |
| 권장 적용 | ERP 연동 백엔드 API | 포털 프론트 + 챗봇 오케스트레이션 |

**현실적 권고**: 프론트/챗봇은 후보 B 계열로 빠르게, **ERP 연동 백엔드는 후보 A(.NET)** 또는 별도 읽기 API로 분리하는 **하이브리드**. 단일 스택 강제하지 말 것. 결정 전까지 새 코드는 "API 호출 인터페이스"를 추상화해 스택 교체 비용을 낮춘다.

### 3.3 스택 확정 전 코딩 원칙

- 비즈니스 로직을 HTML 인라인 `<script>`에 묻지 말고, 데이터 접근을 **API 클라이언트 모듈 1곳**으로 모은다(현재는 mock, 이후 실 API로 교체).
- 화면은 "데이터 출처 = API"라고 가정하고 작성한다. localStorage 의존(`jeilax_auth`, `jeilax_link_v1`)은 데모 한정임을 주석으로 명시한다.

## 4. ERP(MS SQL) 연계 규칙

1. **운영 MSSQL은 읽기 전용.** 포털은 **중간 DB(복제/스테이징)** 만 바라본다. 운영 → 중간 DB는 **야간 배치**(또는 CDC) 동기화. 실시간 단건 조회가 꼭 필요하면 읽기 전용 뷰/저장프로시저로 화이트리스트 한정.
2. **네트워크**: 파일럿은 IP 허용, 운영은 VPN. 접속 정보·연결 문자열은 환경변수로만.
3. **포털 전용 쓰기 데이터**(사진, 검수 메모, 진행상태, 메시지 등)는 **별도 포털 DB/테이블**에 저장하고 ERP 키로 매핑한다. ERP 테이블에 직접 쓰지 않는다.
4. **알려진 ERP 매핑(협력사 발주·검사)**: `M_PUR_REQ`(발주요청) · `M_PUR_ORD`(발주) · `M_CONFIG_PROCESS`(SUBCONTRA_FLG=외주구분) · `M_PUR_GOODS_MVMT`(입출고) · `Q_INSPECTION`(검사) · `M_IV`(품목). 신규 화면은 이 매핑을 재사용·확장한다.
5. **쿼리는 파라미터 바인딩만.** 문자열 연결 SQL 금지(인젝션 방지). 대량/집계 쿼리는 중간 DB에서, 운영 DB 부하를 만들지 않는다.
6. 유니포인트(ERP 벤더) 협의가 필요한 사항(스키마 접근·뷰 생성)은 임의 진행하지 말고 **사용자에게 확인 요청**.

## 5. 인증·권한 (Entra / Teams 연동 + RBAC)

1. **로그인은 Microsoft Entra ID 단일 경로**(=Teams/MS365 계정). 사내 사용자는 별도 비밀번호 없이 SSO + MFA. 자체 회원가입·자체 비밀번호 저장 금지.
2. **redirect URI**는 SPA 플랫폼에 정확히 등록된 값만 동작(현재 `https://ai.jeilm.co.kr/`, `…github.io/jeil_ax/`). 새 호스트/경로 추가 시 Entra 앱 등록 변경이 선행되어야 함 — 코드만 바꾸면 AADSTS50011로 실패. **Entra 앱 설정 변경은 사용자(관리자)가 직접** 하고, Claude는 변경 목록만 제시한다.
3. **권한 모델(RBAC)은 Entra 보안그룹 기준**으로 설계: `AI-Portal-Admins`, `AI-Portal-Finance`, `AI-Portal-HR`(필요 시 `AI-Portal-Subcontractor`, `AI-Portal-Sales` 등 확장). 화면·API·데이터를 그룹 클레임으로 인가한다.
4. **권한 판정은 백엔드에서.** 프론트의 메뉴 숨김은 UX일 뿐, 실제 데이터 차단은 API/DB 레벨에서 한다. 프론트 토큰만으로 민감 데이터를 내려주지 않는다.
5. **협력사(사외) 사용자**는 사내 직원과 동일 Entra 테넌트에 넣지 않는다. 별도 인증 경계(B2B 게스트 또는 협력사 전용 인증)로 분리하고, 협력사는 **자기 발주 건만** 접근(행 수준 보안). §9 참조.
6. **토큰 저장**: 데모는 localStorage지만, 운영은 XSS 노출 위험 → HttpOnly 쿠키 + 백엔드 세션, 또는 BFF 패턴을 §6에서 검토.

## 6. 보안 기준

- **전송 구간 HTTPS 강제.** 포털 코드는 `isHttps()` 참일 때만 로그인 허용(현 동작 유지).
- **시크릿 회전**: Entra Client Secret 만료 **2026-12-09** → 11월 중 갱신(미갱신 시 로그인 전면 중단). 만료 추적은 일정에 등록한다.
- **운영 전환 시 토큰 보관 재설계**: localStorage 토큰 → BFF/HttpOnly 쿠키 검토(XSS 내성).
- **감사 로그**: 누가·언제·어떤 ERP 데이터/화면을 봤는지 백엔드에서 로깅(개인정보·금액 화면 우선).
- **입력 검증·출력 인코딩**으로 XSS/인젝션 방지. 외부 입력(협력사 업로드 사진·메모)은 파일형식·용량·확장자 검증 + 별도 스토리지.
- **민감 데이터 최소화**: 화면·API 응답에 필요한 컬럼만. 급여·인사·자금 데이터는 그룹 인가 필수.
- 보안 점검이 필요한 변경은 `/security-review` 스킬로 셀프 점검 후 사용자에게 결과 보고.

## 7. 코딩·문서 컨벤션

- **언어**: 코드 주석·문서·커밋 메시지·UI 텍스트는 **한국어**. 부서명 표기 통일(예: "인사총무팀" 아님 → **"인사팀"**). 챗봇 명칭은 **jeil-chat**(구 UNI-Chat 표기 금지).
- **HTML 데모**: 단일 파일 자기완결 원칙(외부 CSS/JS 분리 최소화, 공유 편의). 단 신규 실구축 코드는 모듈 분리.
- **데이터 접근은 한 모듈로.** mock↔실API 전환을 한 곳에서. 화면 곳곳에 fetch 흩지 않는다.
- **파일/폴더 명명**은 기존 규칙 계승: `NN_주제_설명.html`, 날짜는 `YYYY-MM-DD` 또는 `YYYYMMDD`.
- 새 기능은 해당 영역 README / `00_종합` 문서에 요구사항 이력을 갱신한다(특히 08 협력사 포털).

## 8. Git·배포 규칙

1. **커밋 메시지는 한국어, 무엇을·왜를 명확히**(기존 스타일 계승). 예: `운영페이지 외주 협력사 발주·검사 포털 섹션 신설 — 사내 대시보드+협력사 모바일 묶음`.
2. **`MS_connect/ doc/ .claude/` 는 절대 add 하지 않는다.** 커밋 전 `git status`로 민감 폴더·`.env`·키 파일이 스테이징되지 않았는지 확인.
3. **배포 절차(중요)**: OneDrive(E:) 작업폴더에서 바로 push 빌드하지 말 것 — 동기화 지연으로 누락/충돌 발생. 권장: **`/tmp`에서 `git clone` → 패치·빌드 → 커밋/푸시(또는 GitHub 웹 업로드)**. `gh-pages` 반영 후 `ai.jeilm.co.kr`에서 실제 확인.
4. **main과 gh-pages 동기화**에 유의(과거 main 미갱신 사례 있음). 라이브에 올린 변경은 `main`에도 반영한다.
5. `JEIL_AX_포털데모_통합본.html`은 손으로 고치지 말고 **항상 `python _build_bundle.py`로 재생성**한다.
