05. 사용자-부서-사원 매핑·대사 (이메일 SSOT)

관리자: 최동혁(dh.choi@jeilm.co.kr) · 기준일: 2026-07-08 · 상태: Active(실적재 완료·가동) 관련: 00 현재상태 · 01 연계기획 §4.5 · 02 진행상태 · 04 증분·확장·거버넌스 비밀값(접속 IP/ID/비밀번호) 기록 금지 — 서버 지칭은 별칭 "ERP운영DB" 만 사용(CLAUDE.md §1).

1. 목적

ERP Z_USR_MAST_REC(사용자 마스터)의 usr_id(=MS 이메일 계정, SSOT) 와 usr_nm(="부서명_이름" 텍스트) 을 대사(reconciliation) 하여, 이메일 기준으로 사용자 ↔ 부서 ↔ 사원 관계를 중간DB(Supabase erp_ro)에서 관리·조회할 수 있게 한다.

이 매핑은 다음을 가능하게 하는 기반 데이터다. - 포털 로그인(Entra, 이메일 기준) 사용자를 소속 부서로 자동 분류(향후 부서별 화면·권한 인가에 활용 가능). - 챗봇·대시보드에서 "이 사람이 어느 부서 소속인가"를 이메일만으로 조회. - 부서별 재직 인원 명부(누가 어느 부서에 있는지) 확인.

주의: 이 매핑은 현재 부서 소속 확인용 참고 데이터이며, 그 자체가 RBAC(권한 부여) 결정을 내리지 않는다. 실제 화면·데이터 접근 통제는 여전히 Entra 보안그룹 + RLS(CLAUDE.md §5)를 따른다.

2. 원천·규칙 (관리자 확인, 2026-07-08)

항목 내용
원천 테이블 Z_USR_MAST_REC(사용자 마스터), B_ACCT_DEPT(부서 마스터) — 모두 ERP운영DB
PK / SSOT USR_ID = MS 이메일 계정(SSOT로 신뢰). USER_E_MAIL 컬럼은 비신뢰(99건 중 1건만 값 있음 — 사용하지 않음)
대상 필터 USE_YN='Y' AND USR_ID LIKE '%@%' → 현재 99건(전체 811건 중 @포함 434건, 사용중 471건)
usr_nm 형식 부서명_이름, 밑줄(_) 정확히 1개 — 대상 99건 100% 이 형식을 따름(파싱 안전)
상태 표기 퇴사/휴직자는 이름 옆에 (퇴사)/(휴직) 접미
알려진 한계 Z_USR_MAST_REC에는 EMP_NO(사원번호)·DEPT_CD(부서코드) 컬럼이 없음 → 부서 정보는 usr_nm 텍스트 파싱이 유일한 소스(사원 마스터·인사테이블과의 정식 조인 키가 없음). 인사테이블(HR) 실검증은 §8 후속

3. 부서 대사 결과 (실적재 완료, 2026-07-08)

실제 ERP 데이터 적재(usr_master_s 99건 · dept_master_s 351건) 후 대사를 실행한 결과, 대상 99건 중 정상 매핑 89건, 대사 불일치 10건으로 확정됐다.

유형 건수 세부
정상 매핑(v_erp_user_dept) 89건 재직 · 형식정상 · 비테스트 · 부서일치
부서불일치 6건 외부회계 2명 · 재무 1명 · 제일엠앤에스 미국(해외법인) 3명
상태이상 3건 활성계정(USE_YN='Y')인데 이름에 (퇴사)/(휴직) 표기 — 김해품질팀·영업팀·자재물류파트 각 1명
테스트계정 1건 admin(test@jeilm.co.kr)
불일치 합계 10건 v_erp_user_dept_recon에서 조회

정책 확정(관리자 결정, 2026-07-08): 불일치 10건은 정상 매핑(연동용)에서 자동 제외한다 — 마이그레이션 erp_user_dept_exclude_unmatched 적용, 정본 SQL 09_erp_user_dept_mapping.sql §3-2에 반영 완료. 즉 v_erp_user_dept는 기존(재직·형식정상·비테스트)에 더해 부서일치까지 요구하도록 조건이 강화됐다. 불일치 10건은 v_erp_user_dept_recon(대사 리포트)에서만 조회 가능하며, 수기 확인 대상으로 남는다.

이로써 이전에 "결정 대기"였던 부서명 처리방침(제일엠앤에스 미국·외부회계·재무·admin)은 자동제외로 해소됐다 — 별도 등재·별칭 작업 없이 정상 매핑에서 걸러진다. 다만 상태이상 3건(김해품질팀·영업팀·자재물류파트)은 부서 매핑 문제가 아니라 ERP 원천 데이터 정합 이상(퇴사/휴직 표기인데 계정이 활성 상태)이므로, 자동제외와는 별개로 현업 확인 후 ERP측(Z_USR_MAST_REC) 정리를 권고한다(§8).

4. 중간DB 구현 (Supabase erp_ro/public)

정본 SQL: 실제구축준비 자료/이관/sql/09_erp_user_dept_mapping.sql · Supabase 마이그레이션명 erp_user_dept_mapping(2026-07-08 적용).

4.1 원천 미러 테이블 (erp_ro)

테이블 내용 RLS
erp_ro.usr_master_s usr_id(PK)·usr_nm·use_yn·src_updated — 실적재 완료 99건 internal_select_usr_master_s(사내만 SELECT)
erp_ro.dept_master_s B_ACCT_DEPT 미러(org_change_id+dept_cd PK)·dept_nm·par_dept_cd·dept_full_nm·end_dept_fg — 실적재 완료 351건 internal_select_dept_master_s(사내만 SELECT)

4.2 파싱·대사 뷰 (erp_ro, 내부용)

뷰 역할
v_dept_dim 부서명 사전(대사 기준) — dept_master_s에서 부서명 distinct
v_user_dept_map 전체 파싱 결과(이메일·원문·부서명·이름·재직상태·부서일치여부·테스트계정여부) — 대사의 기준 뷰
v_user_dept 정상 매핑만(연동용): 형식정상 · 재직 · 비테스트 · 부서일치(2026-07-08 정책 확정 — 불일치 자동제외 조건 추가)
v_user_dept_recon 불일치만(대사 리포트): 형식오류·테스트계정·상태이상·부서불일치 4개 유형(사유별)
v_dept_roster 부서별 인원수·명부(재직자 이름 목록)

4.3 public 노출 뷰 (사내 전용, security_invoker=true)

뷰 컬럼 용도
public.v_erp_user_dept email·dept_nm·emp_nm·matched_dept_cd·dept_matched·src_updated·synced_at 이메일→부서·이름 조회(정상 매핑만)
public.v_erp_user_dept_recon email·usr_nm_raw·dept_nm·emp_nm·status·recon_type·dept_matched·src_updated 대사 불일치 확인(관리자용)
public.v_erp_dept_roster dept_nm·emp_cnt·dept_matched·members 부서별 명부

erp_ro/etl_meta는 여전히 REST 비노출(CLAUDE.md §4 원칙 준수). 위 3개 뷰만 security_invoker로 노출되며, 기존 erp_ro RLS(internal_select_*)가 그대로 적용돼 사내 인증 사용자만 조회 가능하다(협력사·비로그인 차단).

4.4 적재 RPC

기존 public.erp_etl_upsert(p_table, p_rows)에 usr_master_s·dept_master_s 분기를 추가했다(service_role 전용, REST 비노출 유지).

4.5 ETL job

10_ERP_DB연계/etl/etl_run.py에 job 2종 추가:

job 원천 SQL 방식
usr_master Z_USR_MAST_REC WHERE USE_YN='Y' AND USR_ID LIKE '%@%' 증분(AND UPDT_DT >= ? watermark, 04 증분기획 패턴 재사용)
dept_master B_ACCT_DEPT(전량) 전량(소형 마스터, upsert)

5. 대사 판정 유형 (4종, v_user_dept_recon.recon_type)

유형 조건 의미
형식오류 밑줄 없음 또는 이름 파트 공백 usr_nm이 "부서명_이름" 형식을 따르지 않음(0건 — 대상 99건 100% 정상형식)
테스트계정 부서명이 admin 또는 이메일이 test@/test_*@ 테스트/더미 계정(1건: test@jeilm.co.kr)
상태이상 USE_YN='Y'인데 이름에 (퇴사)/(휴직) 표기 활성 계정인데 이름 표기상 퇴사/휴직(3건: 김해품질팀·영업팀·자재물류파트 각 1명) — 현업 확인 후 ERP측 정리 권고(§8)
부서불일치 파싱 부서명이 B_ACCT_DEPT.DEPT_NM에 없음 6건: 외부회계 2명·재무 1명·제일엠앤에스 미국(해외법인) 3명 — 자동제외 정책으로 처리방침 확정(§3)

6. 조회 방법 (참고 — SELECT만, 사내 세션 필요)

-- 이메일 → 부서·이름 조회 (정상 매핑)
select * from public.v_erp_user_dept where email = '<이메일>';

-- 대사 불일치 전체 확인(유형별)
select recon_type, count(*) from public.v_erp_user_dept_recon group by recon_type;

-- 부서별 명부
select * from public.v_erp_dept_roster order by emp_cnt desc;

위 조회는 참고용 SQL 예시이며, 이 문서관제 에이전트(erp-db-link-manager)는 직접 실행하지 않는다(ERP운영DB·중간DB 조회는 erp-db-connector/erp-db-analyst 소관).

7. 실적재 결과 (완료, 2026-07-08)

관리자 직접 실행(!)으로 실제 ERP 데이터 적재를 완료했다.

job 방식 결과
dept_master 전량(upsert) erp_ro.dept_master_s 351건(B_ACCT_DEPT)
usr_master 증분(watermark) erp_ro.usr_master_s 99건(USE_YN='Y' AND USR_ID LIKE '%@%')

etl_meta.batch_run에 두 job 모두 success 이력이 기록됐다. 대사 결과는 §3 참조.

  • 실행 명령(관리자 ! 직접 실행, 선행 --dry-run 검증): python etl_run.py --job dept_master / python etl_run.py --job usr_master
  • 이 에이전트(erp-db-link-manager)는 문서 관제 담당으로 ERP운영DB 직접 조회·ETL 실행을 하지 않는다 — 실적재 실행은 관리자 또는 erp-db-connector가 수행했다.

8. 알려진 한계·후속 과제

  1. upsert-only 잔존 위험: erp_etl_upsert는 삭제하지 않고 병합만 한다 — 재직 상태였다가 USE_YN='N'으로 전환된(즉 대상 필터에서 빠지는) 계정이 usr_master_s에 그대로 남을 수 있다. 04 증분·확장·거버넌스 기획의 "주기적 --full 정합" 패턴을 이 job에도 적용하거나, 별도 prune(정리) 로직 도입이 필요하다.
  2. 인사 테이블(HR) 실검증 미실시: 현재 부서 대사는 B_ACCT_DEPT(부서명 존재 여부)만 사용한다. 사원번호(EMP_NO) 기반 정식 인사 테이블과의 교차검증은 아직 하지 않았다 — 필요 시 후속 과제.
  3. 부서명 텍스트 파싱 의존: EMP_NO/DEPT_CD 컬럼 부재로 부서 정보가 usr_nm 문자열에만 의존한다. ERP 측에 조직변경(부서명 변경·부서 통폐합)이 있으면 파싱 부서명이 바로 갱신되지 않을 수 있다 — 정기 재적재(dept_master)로 완화하되 근본 해법은 유니포인트 협의 대상.
  4. 개인정보 취급(CLAUDE.md §1.7): 직원 이름은 마스킹 예외 대상이라 이메일·이름·부서 조합 노출이 허용된다(사내 role=internal 한정). 급여·주민번호 등 민감값과의 결합은 없음 — 현재 설계 유지로 충분.
  5. (신규) 상태이상 3건 현업 확인 필요: 김해품질팀·영업팀·자재물류파트 소속 각 1명이 활성계정(USE_YN='Y')인데 이름에 (퇴사)/(휴직) 표기가 남아 있다. ERP 원천의 상태 갱신 누락 가능성 — 현업(인사/해당 부서) 확인 후 ERP측(Z_USR_MAST_REC) 정리를 권고한다. 확인 전까지는 자동제외(§3) 상태로 안전 처리되어 있어 연동 데이터에는 영향 없음.
  6. (신규) 연동 활용 — 부서별 ERP 모듈 제안값(2026-07-08): 본 문서의 erp_ro.v_user_dept(이메일→부서 매핑)가 신규 계층 erp_ro.v_dept_erp_suggest(부서별 ERP 모듈 제안값, 관리자 콘솔 '권한 상세' ◆ 참고표시)의 부서 매핑 원천으로 재사용됐다. 실측 결과 ERP 역할·메뉴 권한만으로는 부서 변별력이 약해(사용자당 평균 17역할) 참고표시로만 쓰기로 확정됐다 — 상세는 01_연계기획.md §4.6·02_진행상태.md 참조.
