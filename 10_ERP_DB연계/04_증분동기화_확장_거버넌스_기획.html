04. 증분 동기화 · 대용량 확장 · 권한 거버넌스 기획

관리자: 최동혁(dh.choi@jeilm.co.kr) · 기준일: 2026-07-08 · 상태: Active 관련: 00 현재상태 · 02 진행상태 · 03 중간DB 실행기획 제품기획 정본: 11_제품기획/06 ERP연계

ERP 데이터를 중간DB(Supabase)로 효율적·안전하게 반복 반영하고, 이후 ERP 전체(방대) 데이터로 확장·권한 관리하기 위한 방향성을 정의한다. 지금은 소규모지만 설계는 대용량·다권한을 전제로 한다.

1. 문제: 전량 재적재의 비효율

초기 ETL은 실행할 때마다 전량 추출→전량 upsert했다. item_master 58,256행·pur_order 2,702행을 매번 다시 읽고 쓴다. 데이터가 커질수록:

  • ERP 운영DB 읽기 부하↑(야간이라도 방대해지면 위험), 네트워크·시간↑
  • Supabase 쓰기·API 호출↑(비용·부하)
  • 조회·검증마다 전량 재실행 유혹 → 불필요한 업데이트 반복

2. 해결 원칙: 증분(Incremental) + 주기적 전량 정합

"평소엔 변경분만, 가끔 전량으로 정합" — 표준 CDC-lite 패턴.

2.1 테이블 유형별 전략 (핵심)

유형 예시 전략 근거
대량 원시/스냅샷 item_master_s(58k), pur_order_s(2.7k) 증분 — UPDT_DT >= watermark 변경분만 대부분 불변, 변경분 극소(오늘 실측 58k→136행)
소량 집계 sales_orders_m, purchase_m 당해년 full 재집계 수백 행뿐, 집계는 증분이 부정확(월합계가 전표 추가로 계속 변함)
롤링 일집계 inventory_d 최근 N일 롤링(현행) 최근 재고 중심, 오래된 일자 불필요

원칙: 증분이 이득인 곳(대량·저변경)만 증분. 소량 집계까지 증분화하면 정확도만 잃고 이득 없음.

2.2 Watermark 메커니즘 (구현됨 2026-07-08)

매 실행:
  1) watermark = erp_ro.<table> 의 max(src_updated)  ── RPC public.erp_etl_watermark(job)
  2) 안전버퍼: 추출 기준 = watermark − LOOKBACK_DAYS(기본 2일)  ── 경계 유실·지연도착 방지
  3) 추출 SQL 에 `AND UPDT_DT >= 기준` 부착 → 변경분만 SELECT
  4) upsert(PK 기준 병합) → 신규·변경만 반영, 나머지는 그대로 유지
최초 실행(watermark 없음) 또는 --full → 전량
  • watermark 원천은 중간DB 테이블 자체의 최신 src_updated(별도 상태 저장 불필요, 항상 실제 적재 상태 반영).
  • LOOKBACK_DAYS 버퍼로 같은 시각·지연 도착 전표를 다음 회차가 다시 덮어써 유실 방지(upsert라 중복 무해).
  • 삭제·취소 반영 한계: 증분은 ERP에서 삭제된 행을 감지 못함(upsert는 지우지 않음). → 주 1회 --full 정합으로 스냅샷 재적재(대안: 향후 truncate+reload 또는 tombstone).

2.3 실행 방법 (현재)

평소(증분):   python etl_run.py --job all
주기 정합:    python etl_run.py --job all --full         # 주1회 권장(삭제·정합)
특정 job:     python etl_run.py --job item_master        # 증분
검증:         python etl_run.py --job all --dry-run      # 추출 건수만
  • 적용 대상: pur_order·item_master(증분). sales·purchase(연 full)·inventory(롤링)는 데이터가 작아 매번 전량이 합리적.
  • 참조 RPC(재현): sql create or replace function public.erp_etl_watermark(p_job text) returns timestamptz language sql stable security definer set search_path='' as $$ select case p_job when 'item_master' then (select max(src_updated) from erp_ro.item_master_s) when 'pur_order' then (select max(src_updated) from erp_ro.pur_order_s) else null end; $$; -- service_role 전용(anon/authenticated revoke)

3. 스케줄링·자동화 (다음 단계)

항목 방향 비고
정기 실행 ETL PC/서버 작업 스케줄러(야간 증분 + 주1회 --full) pg_cron은 ERP MSSQL 직접 접근 불가 → 배치는 ERP 연결 호스트에서
실패 알림 etl_meta.batch_run.status='failed' 감지 → 메일/Teams 알림 미구성(로드맵)
관측 연동 현황 페이지(app/erp-status.html)·v_erp_data_asof 구현됨
전용 계정 읽기전용 jeilax_ro로 접속 계정 전환 P1, 유니포인트 협의

4. 대용량 확장 로드맵 (ERP 전체를 향해)

ERP는 2,658 테이블·수천만 행. 전체를 한 번에 붙이지 않고 계층·단계로 넓힌다.

4.1 온보딩 단계 (테이블 화이트리스트 확장)

단계 범위 방식
P4(현재) 발주·품목·매출·매입·재고 5종 증분/집계, 파일럿
확장1 부서 KPI에 필요한 마스터·전표(거래처·계정·프로젝트·수주 등) table_dict(2,658 사전) 기준 선별, 컬럼 화이트리스트
확장2 고빈도·대용량 원시전표(수불·회계 라인 등) 증분 필수 + 파티셔닝(연/월) + 필요 컬럼만
확장3 준실시간 필요 영역 CDC(변경 로그 기반) 또는 짧은 주기 증분

4.2 대용량 설계 지침

  • 필요 컬럼만(개인정보·미사용 컬럼 제외), 집계는 중간DB에서(운영 부하 0).
  • 파티셔닝: 큰 원시 테이블은 연/월 파티션(pg_partman 검토), 오래된 파티션 아카이브.
  • 증분 우선: 신규 테이블은 UPDT_DT(또는 변경 컬럼) watermark를 표준 장착.
  • 전표 취소·삭제: 스냅샷은 주기 full, 대용량은 tombstone/상태컬럼 + reconcile job.
  • 인덱스·튜닝: 조회 패턴 기반 인덱스(현 성능 advisor의 RLS initplan·다중정책 정리 병행 → 11_제품기획/08).

5. 권한·거버넌스 (디테일 관리)

데이터가 넓어질수록 누가 어디까지 보느냐가 핵심. 계층·행·열 3중 통제.

5.1 데이터 등급·스키마 분리

등급 스키마 예 접근
일반 erp_ro 발주·품목·매출·매입·재고 사내(internal) 전체
민감 erp_secure(P5 예약·미생성) 급여·인사평가·자금·원가 부서·직급 그룹 한정
배치메타 etl_meta batch_run service_role만

5.2 3중 통제

  1. 열(Column) 화이트리스트: 노출 뷰(v_erp_*)에 필요한 컬럼만. 개인정보·급여 등은 뷰에서 제외(직원 이름은 마스킹 예외 — [CLAUDE.md §1.7]).
  2. 행(Row) 보안(RLS): 사내/협력사 격리(현행). 확장 시 부서·거래처·프로젝트 단위 RLS(예: 영업은 자기 담당 거래처만).
  3. 역할(Role) 인가: Entra 보안그룹(AI-Portal-Finance·-HR·-Sales…) → JWT claim → 뷰/RPC/챗봇 Tool 인가. 민감 Tool·뷰는 그룹 확인.

5.3 감사·최소권한

  • 감사 로그: 챗봇(chat_log) 외에, 민감 데이터 조회 접근 로깅 확대(누가·언제·무엇).
  • 최소권한: 접속 계정 jeilax_ro(읽기전용) 전환, service_role은 서버에만, 뷰는 authenticated·사내 RLS.
  • 미노출 유지: erp_ro/etl_meta/erp_secure는 REST 비노출, 진입점은 화이트리스트 뷰/RPC뿐.

5.4 권한 매트릭스(목표 예시)

그룹 발주·품목 매출·매입 재고 자금·원가(erp_secure) 인사·급여
전 사내 ○ ○(요약) ○ ✕ ✕
영업/구매 ○ ○(담당) ○ △ ✕
재무/경영 ○ ○ ○ ○ △
인사 ○ ✕ ✕ ✕ ○
협력사 자기 발주만 ✕ ✕ ✕ ✕

실제 그룹·컬럼 매핑은 경영 결정 + 유니포인트 협의 후 확정(§5는 방향성). Entra 보안그룹 신설은 관리자 작업([CLAUDE.md §11-C]).
