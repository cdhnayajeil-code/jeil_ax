> **v0.2 · 2026-07-07**(최초 v0.1 · 2026-07-03) · 작성: 최동혁(+Claude) · 상태: **초안(D1~D3 결정 자체는 대기) — §7-C 내부 구축 일부 착수**
> `01_연계기획` P3 단계(중간 DB + ETL)를 실행 가능한 수준으로 구체화한 문서다. ERP운영DB 접속 정보(IP/ID/비밀번호)는 기록하지 않는다 — 서버 지칭은 별칭 **"ERP운영DB"** 만 사용.

## 1. 목적·범위

**목적**: ERP운영DB(UNIERP, 사외 IDC MSSQL 2022)의 데이터를 포털·챗봇이 안전하게 조회할 수 있도록, **읽기전용 중간 DB(PostgreSQL)와 야간 ETL 파이프라인을 실제로 구축**한다.

| 구분 | 내용 |
|---|---|
| **범위(In)** | 중간 DB 인스턴스 확보 → 스키마·테이블 생성 → 1차 ETL 4종(품목·영업·구매·재고) 가동 → 초기적재(백필) → 검증·대사 → 운영수칙 |
| **범위 외(Out)** | 챗봇 Tool Use 연결(P4), 민감 영역 `erp_secure`(자금·인사, P5 — 경영 승인 후), 사내 타부서 BI 채널(축 A — 별도 진행) |
| **전제** | 마스터플랜 v2.0 승인, 접근정책안 v0.1 준수, `CLAUDE.md §4`(운영 DB 읽기전용·중간 DB 경유·파라미터 바인딩) |

## 2. 핵심 의사결정 3건 (D1~D3) — 이것부터 확정

### D1. 중간 DB를 어디에 두는가

| 옵션 | 내용 | 장점 | 단점/리스크 |
|---|---|---|---|
| **D1-a (파일럿 권고)** | **기존 Supabase 프로젝트에 `erp_ro`·`etl_meta` 스키마 추가** | 비용 0·즉시 착수, 기존 RLS·Entra OIDC 브리지·MCP 운영 경험 재사용, 포털(`app/`)과 같은 접속 체계 | 무료 티어 용량(DB 500MB)·비상업 조건, 외부 SaaS에 사내 데이터 → **1차 일반등급(마스킹·집계)만 허용** |
| D1-b | Supabase **신규 프로젝트** 분리 | 포털과 장애·권한 격리 | 무료 2프로젝트 관리 부담, 이점 크지 않음 |
| **D1-c (운영 목표)** | **Azure Database for PostgreSQL Flexible Server** | 회사 테넌트 안, VNet·Key Vault·백업 등 운영 요건 충족, `02_Azure이관계획`과 정렬 | 비용 발생, Azure 착수 결정 선행 |

**권고**: **파일럿 = D1-a → 운영 = D1-c.** 스키마·DDL·ETL 코드는 표준 PostgreSQL만 사용해(Supabase 전용 기능 배제) 이관 비용을 0에 가깝게 유지한다.

### D2. ETL 러너(배치 실행기)를 어디서 돌리는가

| 옵션 | 내용 | 장점 | 단점/리스크 |
|---|---|---|---|
| **D2-a (파일럿 권고)** | **사내 상시 PC/서버 + Windows 작업 스케줄러 + Python 스크립트** | 기존 자산 그대로 재사용 — `.erp` DPAPI 자격증명, `query.ps1` 검증된 접속 경로, ERP측 방화벽에 이미 허용된 사내망에서 실행 | 실행 PC가 꺼지면 배치 누락 → **상시 가동 장비 지정 필수** |
| D2-b | n8n 워크플로 | 이미 보유, 알림·재시도 UI | ERP 자격증명을 n8n에 복제 보관해야 함(노출면 증가) |
| **D2-c (운영 목표)** | Azure Functions(Timer) 또는 Container Apps Job + Key Vault | 무인 운영·모니터링·시크릿 관리 정석 | Azure·VPN 선행 필요 |

**권고**: **파일럿 = D2-a → 운영 = D2-c.** 스크립트는 처음부터 "환경변수/자격증명 주입형"으로 작성해 러너 이동 시 코드 수정이 없게 한다.

### D3. 추출 원천 — 무엇에 SELECT 하는가

| 옵션 | 내용 | 판단 |
|---|---|---|
| **D3-a (원칙)** | **리포팅 계층(Tier A) `rpt.v_*` 뷰 경유** | 마스터플랜 원칙(운영계 무접촉). Tier A 구축 후 즉시 전환 |
| **D3-b (파일럿 한시 허용)** | ERP운영DB의 **읽기전용 화이트리스트 뷰 직접 SELECT**(야간 03:00, 저부하 증분 쿼리 한정) | Tier A 구축 전 공백기 한시 운용. 유니포인트 합의 + 접근정책안 §의 읽기전용·야간 조건 충족 시 허용 |

**권고**: **D3-b로 시작하되 "뷰 이름·쿼리 형태를 D3-a와 동일하게" 설계**한다(원천 전환 = 접속 문자열 교체만). Tier A 가동 즉시 D3-a로 전환한다.

## 4. 스키마·테이블 설계 (DDL 초안)

### 4-1. 명명 규칙

- 스키마: `erp_ro`(일반) · `erp_secure`(민감, P5 예약) · `etl_meta`(배치 메타)
- 테이블 접미사: `_m` 월집계 · `_d` 일집계 · `_s` 현재 스냅샷
- 모든 `erp_*` 테이블 공통 컬럼: `synced_at TIMESTAMPTZ`(적재 시각) · `src_updated TIMESTAMPTZ`(원천 수정 시각) · `batch_id UUID`

### 4-2. 1차 테이블 4종

```sql
-- ① 품목 마스터 (스냅샷, 전체 갱신)  ← B_ITEM
CREATE TABLE erp_ro.item_master_s (
  item_code    TEXT PRIMARY KEY,
  item_name    TEXT NOT NULL,
  spec         TEXT,
  unit         TEXT,
  item_class   TEXT,            -- 분류 코드→명칭 변환 적재
  use_yn       BOOLEAN,
  synced_at    TIMESTAMPTZ NOT NULL,
  src_updated  TIMESTAMPTZ,
  batch_id     UUID NOT NULL
);

-- ② 영업 수주/매출 월집계  ← S_BILL_HDR/DTL, S_BILL_COLLECTING
CREATE TABLE erp_ro.sales_orders_m (
  ym           TEXT NOT NULL,          -- 'YYYY-MM'
  bp_code      TEXT NOT NULL,          -- 거래처
  bp_name      TEXT NOT NULL,          -- 코드→명칭 변환 적재
  order_amt    NUMERIC(18,0) DEFAULT 0,
  sales_amt    NUMERIC(18,0) DEFAULT 0,
  collect_amt  NUMERIC(18,0) DEFAULT 0,
  order_cnt    INTEGER DEFAULT 0,
  synced_at    TIMESTAMPTZ NOT NULL,
  src_updated  TIMESTAMPTZ,
  batch_id     UUID NOT NULL,
  PRIMARY KEY (ym, bp_code)
);

-- ③ 구매 거래처별 매입 월집계  ← M_IV_HDR/DTL, B_BP
CREATE TABLE erp_ro.purchase_m (
  ym           TEXT NOT NULL,
  bp_code      TEXT NOT NULL,
  bp_name      TEXT NOT NULL,
  purchase_amt NUMERIC(18,0) DEFAULT 0,
  iv_cnt       INTEGER DEFAULT 0,
  synced_at    TIMESTAMPTZ NOT NULL,
  src_updated  TIMESTAMPTZ,
  batch_id     UUID NOT NULL,
  PRIMARY KEY (ym, bp_code)
);

-- ④ 자재 재고 입출고 일집계  ← ITEM_DOCUMENT, M_PUR_GOODS_MVMT
CREATE TABLE erp_ro.inventory_d (
  ymd          DATE NOT NULL,
  item_code    TEXT NOT NULL,
  wh_code      TEXT NOT NULL,
  in_qty       NUMERIC(18,3) DEFAULT 0,
  out_qty      NUMERIC(18,3) DEFAULT 0,
  stock_qty    NUMERIC(18,3),           -- 일말 재고
  synced_at    TIMESTAMPTZ NOT NULL,
  src_updated  TIMESTAMPTZ,
  batch_id     UUID NOT NULL,
  PRIMARY KEY (ymd, item_code, wh_code)
);
```

> 컬럼 확정은 유니포인트 뷰 스펙 협의 결과에 따른다. 원칙: **원본 전체 복제가 아니라 화면·챗봇이 바로 쓰는 집계 형태**, 코드값은 ETL 변환 단계에서 명칭 병기.

### 4-2-1. 추가 테이블 — ERP 테이블 사전 (`erp_ro.table_dict`, 2026-07-07 신설)

1차 4종과 별개로, 사내 OneDrive `ERP_DB/erp_chat/data`에 이미 구축된 ERP 스키마 메타(2,658개 테이블·452개 컬럼 문서화분)를 적재하는 사전 테이블. **ERP운영DB에 접속하지 않고 로컬 메타 파일만으로 적재**하며(`etl/load_table_dict.py`), 향후 유니포인트 협의(뷰 스펙 결정)와 챗봇 2단계 Tool 설계 시 참조 지식원으로 쓴다. 컬럼: 테이블명·설명·모듈·컬럼수·행수 등(문서화 메타 그대로, 개인정보 없음).

### 4-3. 배치 메타 (`etl_meta`)

```sql
CREATE TABLE etl_meta.batch_run (
  batch_id     UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  job_name     TEXT NOT NULL,           -- 'item_master' 등
  started_at   TIMESTAMPTZ NOT NULL,
  finished_at  TIMESTAMPTZ,
  status       TEXT NOT NULL,           -- running/success/failed/retried
  rows_read    INTEGER,
  rows_upserted INTEGER,
  watermark    TIMESTAMPTZ,             -- 증분 기준(원천 최대 수정시각)
  error_msg    TEXT
);
-- 조회 편의 뷰: 최신 성공 배치 = 화면의 「데이터 기준 시각」
CREATE VIEW etl_meta.v_last_success AS
  SELECT DISTINCT ON (job_name) job_name, finished_at, watermark
  FROM etl_meta.batch_run WHERE status = 'success'
  ORDER BY job_name, finished_at DESC;
```

## 5. ETL 파이프라인 처리방법 (상세)

### 5-1. 잡 구조 (잡 1개 = 테이블 1개)

```
[03:00 스케줄러]
  └─ etl_run.py --job all
      ├─ ① 추출  : ERP 화이트리스트 뷰 SELECT (파라미터 바인딩, 증분 조건)
      ├─ ② 변환  : 코드→명칭, 월/일 집계, 타입 정규화
      ├─ ③ 적재  : PostgreSQL UPSERT (INSERT … ON CONFLICT DO UPDATE)
      ├─ ④ 검증  : 건수·금액 합계 대사 (원천 vs 적재, 오차 0 원칙)
      ├─ ⑤ 기록  : etl_meta.batch_run (성공/실패·건수·watermark)
      └─ ⑥ 알림  : 실패 시 Teams/메일 → 1시간 후 자동 재시도 1회
```

### 5-2. 적재 전략 (테이블별)

| 잡 | 방식 | 이유 |
|---|---|---|
| `item_master_s` | **전체 재적재**(TRUNCATE→INSERT 또는 전건 UPSERT) | 마스터 소량, 삭제 반영 필요 |
| `sales_orders_m` / `purchase_m` | **롤링 윈도 재집계** — 당월+직전 2개월(3개월분)을 매일 다시 집계·UPSERT | 소급 수정(전표 정정)이 최근월에 집중, 증분 감지보다 단순·정확 |
| `inventory_d` | 증분(수정일시 watermark) + 최근 7일 윈도 재적재 | 일 단위 대량, 윈도로 정정 흡수 |

- **삭제 처리**: 원천 행 삭제는 윈도 재집계 범위 안에서 자연 반영. 윈도 밖 소급 정정 발견 시 해당 월 수동 재적재(런북 §7).
- **초기적재(백필)**: 2024-01 ~ 현재를 월 단위 분할 실행(야간·주말). 백필 완료 후 일 배치 전환.

### 5-3. 추출 쿼리 원칙

- **화이트리스트 뷰만** SELECT. 테이블 직접 조회·`SELECT *`·문자열 연결 SQL 금지, 반드시 파라미터 바인딩.
- 증분 조건·기간 조건으로 스캔 범위 최소화(야간이라도 저부하 원칙).
- 1회 페치 상한·쿼리 타임아웃(예: 10분) 설정 — 폭주 쿼리 자동 차단.

### 5-4. 알림·재시도·운영 표시

| 항목 | 정책 |
|---|---|
| 실패 알림 | Teams/메일(협력사 포털에서 쓰는 M365 Graph 발송 체계 재사용 검토) — 잡명·오류·재시도 예정 시각 |
| 재시도 | 실패 1시간 후 자동 1회, 재실패 시 사람 개입(런북) |
| 기준 시각 표기 | 모든 화면·챗봇 답변에 `etl_meta.v_last_success` 기반 「데이터 기준: YYYY-MM-DD 03:00」 |
| 로그 보관 | `batch_run` 90일 보관 후 삭제 배치 |

## 6. 보안·권한 설계

| 영역 | 정책 |
|---|---|
| ERP 접속 자격증명 | 파일럿: 러너 PC `%USERPROFILE%\.erp\`(DPAPI 암호화 — 기존 체계 그대로). 운영: Azure Key Vault. **코드·문서·로그에 값 기록 금지** |
| 중간 DB 쓰기 권한 | ETL 러너 전용 키(Supabase `service_role` 또는 전용 DB 계정)만 쓰기 가능. 프론트에는 절대 노출 금지 |
| 중간 DB 읽기 권한 | `erp_ro` 테이블 **RLS 적용** — 사내 인증(Entra OIDC 브리지) 사용자만 SELECT, `anon` 차단. 협력사(vendor) 롤 접근 불가 |
| 데이터 등급 | 파일럿 적재는 **일반 등급(집계·코드·수량·금액)만**. 개인정보·급여·원가 상세는 원천 뷰 스펙 단계에서 제외(유니포인트 협의 항목) |
| 전송 구간 | ERP→러너, 러너→중간 DB 모두 TLS 강제 |
| 감사 | ETL 이력은 `etl_meta.batch_run`. 사용자 조회 감사는 P4(API 계층)에서 구현 |
