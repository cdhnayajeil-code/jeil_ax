01. ERP DB 연계 — 이후 기획 (통합 정리)

기준일: 2026-07-03(2026-07-08 §4.4 P4 노출 계층, §4.5 사용자-부서 매핑, §4.6 부서별 ERP 모듈 제안값, §4.7 급여 민감권한·erp_secure 착수 → 같은 날 실적재 완료로 갱신 / 2026-07-20 §4.8 챗봇 권한체계 연결·포털 도구 권한구멍 차단 추가 / 2026-07-21 §4.9 연동 현황 뷰 10종 전체 노출·관리자 콘솔 반영 추가, §4.10 매입 상세(iv_dtl) 신규 연동·구매요청 상세 백필 실적재 완료 추가) · 관리: 최동혁 ERP 데이터 활용은 두 축으로 진행된다: A) 사내 타부서 데이터 개방(마스터플랜 v2.0)과 B) AI 포털·챗봇 연계(중간DB 연계설계). 본 문서는 두 기획을 하나의 그림으로 통합 정리한 것이다. 상세는 각 원본 문서를 따른다.

1. 공통 대원칙 (두 축 모두 적용)

원칙 내용
READ-ONLY 절대성 연동 경로 어디에서도 운영계 쓰기 불가. 사본조차 읽기전용.
부하 격리 소비자(부서·포털·챗봇) 트래픽은 운영계에 닿지 않는다 — 야간 사본에서만 조회.
최소 권한 필요한 뷰·컬럼·행만. 기본 거부, 명시 허용. 개인별 급여·주민번호·계좌는 원천 View 단계에서 제외.
스키마 추상화 소비자는 물리 테이블을 모른다 — 뷰(rpt.v_*)·집계 테이블이 계약(Contract).
추적성 누가·언제·무엇을 조회했는지 전량 감사 로깅.
점진성 최소 인프라로 즉시 시작(Crawl) → 수요 증가 시 승격(Walk/Run).

2. 통합 목표 아키텍처

┌────────────────────────────────────────────────────────────┐
│ ERP운영DB (UNIERP · MSSQL 2022 · 사외 IDC · 단일노드)       │
│              ✕ 모든 소비자 직접 접근 금지                   │
└──────────────┬─────────────────────────────────────────────┘
               │ 야간 1회 갱신 (백업·복원 / 스냅샷 / ETL)
               ▼
┌────────────────────────────────────────────────────────────┐
│ ① 리포팅 계층 — 격리된 읽기전용 사본 (Tier A→B)             │
│ ② 추상화 계층 — rpt.v_* 뷰 (마스킹·RLS·스키마 계약)         │
└──────┬──────────────────────┬──────────────────────────────┘
       │ [축 A] 사내 타부서    │ [축 B] AI 포털·챗봇
       ▼                      ▼
 ③-A 직접조회(BI/Excel)   야간 ETL → PostgreSQL 중간 DB
 ③-B API(n8n/REST)          (erp_ro / erp_secure / etl_meta)
 ③-C 파일(CSV/Parquet)          │
       │                        ├─▶ 부서 대시보드(pages/) — REST API
       │                        └─▶ 챗봇(jeil-chat) — Tool Use 화이트리스트
       ▼                        ▼
┌────────────────────────────────────────────────────────────┐
│ ④ 거버넌스 — 인증(Entra)·권한(RBAC)·감사·SLA·승인·정기검토  │
└────────────────────────────────────────────────────────────┘

정합 포인트: 포털 중간 DB의 ETL도 운영계가 아닌 리포팅 계층(①)/읽기전용 뷰(②)에서 추출하도록 일원화한다. 원천 접근 창구를 하나로 유지해 운영 무영향 원칙을 지킨다.

3. 축 A — 사내 타부서 개방 (마스터플랜 v2.0 요약)

  • 리포팅 계층 티어: Tier A(동일 인스턴스 내 리포팅 전용 DB, 즉시·저비용)로 시작 → 소비자·조회량 증가 시 Tier B(별도 리포팅 서버, 물리적 완전 격리)로 승격 → 전사 BI 수요 확인 시 Tier C(BI/DW) 별도 과제. Always On은 일 배치 요건에 과잉투자로 배제.
  • 전달 채널: 소비자 유형별 선택 — BI/Excel은 읽기전용 계정 직접조회(③-A), 자체 앱은 API(③-B), 대량·이관은 야간 파일 Export(③-C). 모두 ② 뷰만 바라본다.
  • 경영 의사결정 1건: Tier A 시작 vs 곧바로 Tier B (마스터플랜 §11). 나머지는 실무 진행.

4. 축 B — AI 포털·챗봇 연계 (07 연계설계 요약)

4.1 중간 DB (PostgreSQL jeil_portal)

portal       — 사용자·크레딧·대화·감사로그(tool_audit)
erp_ro       — 영업·구매·재고·원가 (일반, 전 직원)
erp_secure   — 자금·인사 집계 (민감, Entra 그룹 인가)
knowledge    — 문서 벡터 (pgvector)
etl_meta     — 배치 실행 로그·동기화 상태

우선순위 결정(2026-07-07): 발주 데이터는 정식 1차 4종(품목·영업·구매=매입·재고) 가동 전이라도, 연도를 2026년도로 좁혀 파일럿 우선연동하기로 했다(협력사 발주포털 실데이터 전환 대비, CLAUDE.md §4.4·§9 매핑 재사용). 실행에는 유니포인트 협의(발주요청/발주 테이블 컬럼·연도 필터 기준 확인) 및 ERP_DB_CONN 입력이 선행되어야 하며, 실제 ETL 개발·실행은 10_ERP_DB연계/etl/(별도 개발 세션) 또는 erp-db-analyst(ERP_DB 작업환경) 소관이다. 상세 진행은 02_진행상태.md P3 참조.

4.3 ETL·챗봇 정책

  • 배치: 매일 03:00(유니포인트 합의), 증분 기준 + 소량 집계는 전체 재적재. 실패 시 Teams/메일 알림 + 1시간 후 재시도 1회. 모든 화면·답변에 「데이터 기준: YYYY-MM-DD 03:00」 표기.
  • 챗봇: AI가 SQL을 생성·실행하지 않는다 — 화이트리스트 조회 템플릿(get_sales_monthly 등 6종 초안)만 실행. 권한은 서버 단 Entra 그룹 검사, 1회 최대 200행, portal.tool_audit 전량 로깅.
  • 대시보드와 챗봇은 동일 중간 DB·동일 쿼리 로직 공유 — ETL·테이블 1벌 유지.

4.4 P4 노출 계층 (2026-07-08 착수)

포털·챗봇이 erp_ro/etl_meta를 직접 REST로 조회하지 않도록, public 스키마에 사내 전용 security_invoker 뷰 6종을 신설했다: v_erp_sales_monthly·v_erp_purchase_monthly·v_erp_inventory_daily·v_erp_item·v_erp_pur_order·v_erp_data_asof(job별 최신 성공시각). erp_ro·etl_meta는 여전히 REST 비노출이며, 기존 erp_ro RLS(internal_select_*)가 security_invoker 뷰에도 그대로 적용돼 사내 인증 사용자만 통과한다. 챗봇(jeil-chat) Edge Function은 service_role 전용 GRANT로 서버 측에서만 조회한다. 정본 SQL: 실제구축준비 자료/이관/sql/08_erp_pub_reporting_views.sql.

RLS 격리 실측 테스트(2026-07-08): JWT 시뮬레이션 결과 — 사내(internal): 정상 조회(매출10·매입63·재고656·품목58,246·발주2,699·기준시각5행), 협력사(vendor): 전부 0행, anon: 미부여로 차단.

이 위에서 실제 연결도 시작됐다: app/lib/api.js의 erpApi로 pages/구매_거래처별매입집계_2026.html 1개 화면을 파일럿 실연동했고, 챗봇에 ERP Tool 4종(get_erp_sales_monthly·get_erp_purchase_monthly·get_erp_inventory_status·get_erp_item)을 배포했다(version 7 ACTIVE). 유니포인트 협의(P1)·읽기전용 계정 전환·야간배치 자동화·Entra→Supabase 세션 브리지는 여전히 미해결이다. 상세 진행은 02_진행상태.md P4 참조.

4.5 사용자-부서-사원 매핑·대사 (2026-07-08 신설)

ERP Z_USR_MAST_REC(사용자 마스터)의 usr_id(=MS 이메일, SSOT)와 usr_nm(="부서명_이름")을 대사해, 이메일 기준으로 사용자↔부서↔사원 관계를 중간DB(erp_ro)에서 관리하는 계층을 추가했다. 이는 축 B(포털·챗봇 연계)의 부속 계층으로, 향후 부서 기준 화면 분류·챗봇 응답의 소속 확인 등에 활용할 수 있는 참고 데이터다(실제 권한 인가는 여전히 Entra 보안그룹 + RLS가 담당, CLAUDE.md §5).

  • 대상: USE_YN='Y' AND USR_ID LIKE '%@%' 99건. usr_nm 파싱(부서명_이름) → B_ACCT_DEPT와 대사해 29개 부서명 중 25개(95건) 일치, 불일치 4개(해외법인·외부회계·재무 세분화·admin 테스트계정)는 처리방침 결정 대기.
  • 노출 뷰(사내 전용, security_invoker): public.v_erp_user_dept(정상 매핑)·v_erp_user_dept_recon(대사 불일치 리포트)·v_erp_dept_roster(부서별 명부).
  • 스키마·뷰·ETL job(usr_master·dept_master)은 구축·적용 완료됐으나 실적재는 미실행(0행, 관리자 ! 직접 실행 대기).
  • 상세는 전용 관리 문서 05_사용자부서_매핑대사.md 참조. 정본 SQL: 실제구축준비 자료/이관/sql/09_erp_user_dept_mapping.sql.

4.6 부서별 ERP 모듈 제안값(참고) (2026-07-08 신설)

관리자 콘솔(04)의 '권한 상세' 탭에서 부서별 ERP 모듈 권한(sales/purchase/pur_order/inventory/item)을 편집할 때 참고할 수 있도록, ERP 역할·메뉴 권한 기반 '제안값'을 추가했다. Z_USR_MAST_REC_USR_ROLE_ASSO→Z_USR_ROLE_MNU_AUTHZTN_ASSO(MNU_USE_YN='Y')→Z_CO_MAST_MNU(ModuleInitial)를 조인해 사용자별 접근 모듈(ModuleInitial∈SD/MM/IM/MDM)을 erp_ro.usr_erp_module_s로 추출(ETL job usr_erp_module)하고, §4.5의 이메일→부서 매핑(v_user_dept)과 결합해 부서 과반(≥50%)이 접근하는 모듈을 erp_ro.v_dept_erp_suggest/public.v_erp_dept_erp_suggest로 산출한다.

관리자 결정(2026-07-08): 이 제안값은 콘솔에 ◆ 참고표시로만 노출하고 자동 덮어쓰기하지 않는다 — 실질 통제선은 여전히 편집형 dept_erp_scope(업무기준)다. 이유는 실측 소견: 이 조직은 ERP 읽기권한이 광범위(사용자당 평균 17개 역할)해, 부서 과반 임계로 한정해도 거의 모든 부서에 전 모듈이 제안되어 ERP-역할 신호의 부서 변별력이 약하기 때문이다(이 실측 결과가 '참고표시만' 결정의 타당성을 뒷받침). 정본 SQL: 실제구축준비 자료/이관/sql/12_erp_dept_module_suggest.sql. 실적재 완료(2026-07-08, 397건). 상세 진행은 02_진행상태.md P3·P4, 이슈는 §4 참조.

4.7 급여 민감권한 강화 + erp_secure — 실적재 완료(집계·인사팀 전용) (2026-07-08 신설, 같은 날 실적재)

인사(급여) 집계는 축 B의 가장 민감한 영역(P5)이라, 실 적재보다 권한 프레임워크와 격리 스키마를 먼저 갖춘 뒤(승인-준비) 같은 날 관리자 직접 실행으로 실 데이터 적재까지 완료했다.

  • 급여 민감 모듈(라이브): 급여를 별도 민감 모듈 payroll로 신설해 dept_erp_scope에 인사팀만 부여했다. 인사 페이지(portal_page.hr_2026)는 erp_module='payroll'로 이중 게이트돼, 비인사팀은 URL을 알아도 서버 단(jeil-me)에서 직접 접근이 차단된다. 관리자 콘솔(04) '권한 상세'는 급여를 ⚠ 민감으로 표시하고 일괄부여·◆ 제안값에서 제외한다(개별 지정만 가능). 정본 SQL: 이관/sql/13_payroll_module.sql.
  • 인사 데모 페이지 안전화(라이브): pages/인사_인원급여추이_2026.html의 퇴직정산 표에 있던 개인별 실급여(원단위·"실데이터·A22" 배지)를 샘플값으로 교체해 저장소·통합본에 실데이터가 남지 않도록 했고, 급여 요소는 data-sensitive="pay"로 기본 마스킹(blur)한다. 마스킹 해제는 인사팀 권한자에게만 허용하며, 통합본 iframe·미로그인 상태는 게이트와 독립적으로 fail-safe하게 마스킹을 유지한다.
  • erp_secure 스키마: 민감 분리 스키마 erp_secure를 신설하고 hr_payroll_m(월별 인원·급여총액·퇴직급여 집계만 — 개인별 행·주민번호·계좌는 설계상 영구 미포함)과 hr_access_log(조회 감사)를 만들었다. erp_secure는 REST 비노출·service_role 전용(RLS 활성화, authenticated 정책 없음 = 전면 차단)이다. 정본 SQL: 이관/sql/14_erp_secure_hr.sql.
  • 실적재 완료 + 게이트 API jeil-hr 배포(2026-07-08, 관리자 직접 실행): ETL job hr_payroll(HDF070T 월×부서 집계·HGA070T 월×전사 집계)을 확정·가동해 erp_secure.hr_payroll_m에 143행(급여 월×부서 137 + 퇴직 월×전사 6, 2026.01~06 6개월분)을 적재했다. 컬럼은 ym·dept_nm·headcount·pay_tot_amt·retire_amt뿐 — 개인별·이름·주민번호(RES_NO)·계좌는 포함하지 않는다(집계만, 퇴직은 부서 breakdown 재식별 위험으로 전사 월별만). 게이트 API jeil-hr(Edge Function) v1을 배포해 Entra 검증→dept='인사팀'/portal_admin만 통과(그 외 403)→감사 로그(hr_access_log)→hr_payroll_get() 집계 반환 흐름을 갖췄고, service_role 전용 RPC 3종(erp_secure_upsert·hr_payroll_get·hr_access_log_add)을 신설했다. 인사 페이지는 인사팀 권한자에게 이 실 집계 표(월별 인원·급여총액·퇴직)를 보여주고, 비권한자·데모는 여전히 마스킹·샘플 유지. 개인별 급여·주민번호·계좌는 설계상 앞으로도 영구 미연계.
  • 잔여 게이트: 자금(F 모듈) 실적재는 인사와 별개로 경영 승인 전 미착수, AI-Portal-HR Entra 보안그룹은 아직 미신설(현재는 dept='인사팀'/portal_admin 판정으로 대체 게이트), 유니포인트와의 재직상태(퇴직 구분 등) 정의 확인은 잔여 항목이다.
  • 커넥터 검증 결과(2026-07-08): 인사 테이블 HAA010T(인사기본)·HDF070T(급여)·HGA070T(퇴직급여)·TH101BA1은 실존이 확인됐고, 실컬럼도 확정됐다(PAY_YYMM·DEPT_NM·PAY_TOT_AMT / RETIRE_DT·RETIRE_AMT) — ETL job hr_payroll에 반영해 실적재까지 완료했다. 다만 재직상태(퇴직 구분 등) 정의는 여전히 유니포인트 협의로 확인이 필요하다.
  • 상세 진행·잔여 게이트는 02_진행상태.md P5 참조.

4.8 챗봇 권한체계 연결 + 포털 도구 권한 구멍 차단 (2026-07-20 신설)

§4.4~4.7에서 권한 체계(dept_erp_scope·payroll 모듈·jeil-hr 게이트)와 인사 집계 실적재는 이미 갖춰져 있었지만, 챗봇(jeil-chat)이 그 체계에 연결돼 있지 않아 관리자가 챗봇에 인원현황·권한을 물으면 오답이 나오는 문제가 있었다. 이를 메우기 위해 jeil-chat을 version 18 → 19로 배포했다.

  • get_my_access(신규): 호출자 본인의 역할(관리자/부서관리자/일반)·소속부서·열람가능 ERP 모듈·접근가능/불가 페이지·권한요청 방법을 반환한다. 호출자 UPN을 고정해 타인 권한을 조회할 수 없고, 전 직원이 사용할 수 있다. 권한 질의는 SYSTEM_PROMPT에서 반드시 이 도구로 실조회하도록 강제했다(추정·역산 금지).
  • get_hr_headcount(신규): 인원현황. 전 직원에게는 월별 전사 총원만, 부서별 인원 분포는 payroll 권한자(인사팀·관리자)에게만 반환한다(급여 금액은 포함하지 않음).
  • get_hr_payroll(신규): 급여 집계(인사팀·관리자 전용). erp_secure는 REST 미노출이므로 기존 jeil-hr와 동일하게 service_role RPC hr_payroll_get을 경유한다. 인사 도구(get_hr_headcount·get_hr_payroll) 접근은 허용·거부 모두 hr_access_log에 감사 기록된다(jeil-hr와 원장 재사용).
  • 소급 발견 권한 구멍 차단(중요): 기존 포털 도구 3종 get_order_summary·get_order_detail·get_inspection_pending이 그동안 ERP_TOOL_MODULE 매핑이 없어 권한 강제 없이 전 직원에게 협력사 발주번호·금액·납기를 노출하고 있었다. 이번에 pur_order 모듈 권한으로 강제 적용해 차단했다 — 2026-07-08 v14에서 "ERP 도구 8종을 pur_order로 매핑"한 작업(§4.4)은 ERP 도구 쪽만 대상이었고, 이 포털 도구 3종은 그 매핑에서 빠져 있었다(즉 이 취약점은 그날 이후 계속 실재했다).
  • 관리자 결정(2026-07-20): ①전사 총원은 전 직원 조회 가능, 부서별 인원 분포·급여액은 인사팀·관리자 전용. ②협력사 발주·검사 챗봇 조회 권한은 신규 모듈 신설 대신 기존 pur_order 모듈 권한으로 통합.
  • 경영 판단 보류(변경 안 함): dept_erp_scope가 23개 부서 전부 sales·purchase·inventory·item·pur_order를 동일 보유해 부서별 데이터 차등이 사실상 없다(payroll만 인사팀 단독 차등). 구조는 부서별 차등을 지원하므로 정책 결정만 필요한 상태이며, 이번 작업에선 손대지 않았다(02_진행상태.md §4, 00_현재상태_스냅샷.md §6).
  • 배포는 verify_jwt=false 유지, 배포본이 로컬 소스와 문자 단위 일치 검증됐다. 소스(supabase/functions/jeil-chat/index.ts)는 수정·배포 완료 상태이나 git 커밋은 관리자가 별도로 진행한다.
  • 상세 진행은 02_진행상태.md P4·P5 참조.

4.9 연동 현황 뷰 10종 전체 노출 + 관리자 콘솔 반영 (2026-07-21 신설)

연동 현황을 사내 사용자·관리자가 확인하는 화면 app/erp-status.html과 그 원천 뷰 public.v_erp_sync_overview는 이미 가동 중이었으나, 지금까지 본 관제 문서에는 등재되지 않은 채 운영되고 있었다(문서화 공백 — 이번에 소급 등재).

  • 가시성 격차 발견: 이 뷰가 1차 5종(발주·품목·매출·매입·재고)만 노출하고 있었는데, 실제 erp_ro/erp_secure에는 §4.5~4.8에서 이미 실적재를 마친 10종(추가 5종: 구매요청 pur_req·부서마스터 dept_master·사용자마스터 usr_master·부서별 ERP모듈 usr_erp_module·급여집계 hr_payroll)이 있어, 연동 자산의 절반이 이 현황 화면에서 보이지 않았다.
  • 뷰 확장: 마이그레이션 erp_sync_overview_all_sources로 v_erp_sync_overview를 10종 전체로 확장하고, sensitive(민감 표시)·sort(정렬) 컬럼을 추가했다. 급여(hr_payroll)는 erp_secure 민감 스키마이므로 실테이블을 count하지 않고 etl_meta.batch_run.rows_upserted만 사용해 금액·명세를 노출하지 않는다. 발주 원천 표기를 실제 조인 기준(M_PUR_ORD_HDR + M_PUR_ORD_DTL)으로 정정했다. security_invoker=true는 유지되며 사내(internal) 세션만 조회 가능하다(RLS 상속, 변화 없음).
  • 관리자 콘솔 반영: jeil-chat-admin을 version 9 → 10으로 배포해 이 뷰를 service_role로 조회한 결과를 응답 erp_sync로 반환하고, 04_챗봇_포털_데모UI.html 관리자 콘솔 'API' 탭에 실데이터 표를 신설했다(기존엔 '사내 운영 페이지' 카드 링크로만 접근 가능했음).
  • 연결 정보 명세: app/erp-status.html에 연결 정보 표를 추가해 원천(UNIERP MS SQL, 사외 IDC, DB JEILMNS)·접속 방식(ODBC 읽기전용, SELECT … WITH (NOLOCK), 화이트리스트 SQL, DDL/DML 없음)·적재 경로(10_ERP_DB연계/etl/etl_run.py → RPC erp_etl_upsert/민감분 erp_secure_upsert → erp_ro/erp_secure)·실행 방식(수동 실행, 자동 스케줄러 미도입, 이력은 etl_meta.batch_run)·접속정보 보관(실행환경 .env에만, 소스·문서 미기재)·데이터 격리(RLS, 협력사·anon 0행)를 명시했다. 비밀값(서버 IP·계정·비밀번호)은 어디에도 기재하지 않았다.
  • 현재 적재 실측(2026-07-21)은 00_현재상태_스냅샷.md §4, 재적재 이력 추적 공백은 02_진행상태.md §4 참조.

4.10 매입 상세(iv_dtl) 신규 연동 + 구매요청 상세 백필 — 실적재 완료 (2026-07-21 신설)

같은 날 오전(§4.9와 별개 작업)에 erp_ro.table_dict(테이블 사전) 실측으로 두 가지가 확인됐다: ①M_PUR_REQ는 HDR/DTL 분리가 없는 라인 단위 단일 테이블(49컬럼·63,423행)이라 "DTL 반영"은 미적재 상세 컬럼을 채우는 백필로 해석해야 한다는 점(원천에 단가·금액 컬럼이 없어 구매요청 금액은 여전히 연결 발주 기준이라는 기존 원칙 유지), ②M_IV_DTL(52,478행·66컬럼)이 실재하는데 그동안 M_IV_HDR 월집계(purchase_m)만 적재해 "무엇을·얼마에" 매입했는지 라인 단위로 확인할 수 없었다는 점. 이를 메우기 위해 스키마·ETL·뷰를 신설했다 — erp_ro.iv_dtl_s(PK iv_no+iv_seq_no, 품목·수량·단가·금액·부가세 + 발주 연결키 po_no/po_seq_no·입고 연결키 mvmt_no, RLS ON+internal_select_iv_dtl), pur_req_s에 조달구분(procure_type)·확정/릴리즈수량·구매그룹/조직·저장위치·수주라인·도면번호·추적번호·변경차수·MRP오더·생성/수정자 14컬럼 추가, 노출 뷰 v_erp_iv_dtl·v_erp_iv_dtl_monthly(월×품목 집계) 신설(v_erp_pur_req는 기존 컬럼·순서 보존한 채 상세 컬럼만 append), erp_etl_upsert 화이트리스트에 두 대상 추가, ETL job iv_dtl(증분 UPDT_DT 지원) 신설·pur_req SQL 확충. 이 단계에서는 적재는 관리자 실행 대기였다(00_관리체계/03_변경관리_CHANGELOG 2026-07-21 첫 줄 참조).

같은 날 오후, 관리자가 위 준비분을 직접 실행해 실적재까지 완료했다(batch_run 7건 성공·0건 실패). iv_dtl: 2026-01-05~07-20 구간 1,981행(0.9초) — 송장 213건·발주 432건, 매입금액 합계 35.6억원, 발주 연결(po_no) 100%(발주 미연결 36건은 과거 발주로 추정되는 데이터 품질 참고 항목, 임의 조치 없음). pur_req 상세 백필: --full 4,063행(2.1초) — 조달구분 채움 99.98%(미채움 1건은 ERP 원천 PROCURE_TYPE NULL), 도면번호 보유 856건.

부수적으로 발견·해소된 정합성 이슈: 신규 적재한 매입 상세를 기존 월집계(purchase_m)와 대사하니 6월 +19.2억원·7월 +1.5억원 차이가 나타났다(1~5월은 완전 일치). 원인은 데이터 오류가 아니라 적재 시점 차이(2026-07-08 문서화 이후 ERP에 새로 등록된 매입 건이 월집계에 미반영)였다 — 관리자가 purchase --full(151행)·pur_order(증분, 512행)·sales(37행)·item_master(증분, 415행)·inventory(1,847행)를 재적재해 매입 대사 차이를 전월 0으로 해소했다(매핑·로직 정상 검증). hr_payroll(민감)은 별도 승인 사항이라 이번 재적재에서 제외했다(2026-07-08 143행 유지).

상세 진행은 02_진행상태.md §1·§2 P3, 현재 적재 실측·이슈는 00_현재상태_스냅샷.md §1·§4·§6 참조.
