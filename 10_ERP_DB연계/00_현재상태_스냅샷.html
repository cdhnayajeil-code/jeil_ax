00. ERP DB 연계 — 현재상태 스냅샷

기준일: 2026-07-13(v0.10, 이전 v0.9 · 2026-07-08 / v0.8 · 2026-07-08 / v0.7 · 2026-07-08 / v0.6 · 2026-07-08 / v0.5 · 2026-07-08 / v0.4 · 2026-07-08 / v0.3 · 2026-07-08 / v0.2 · 2026-07-07) · 관리: 최동혁 · 전담 에이전트: erp-db-link-manager ERP(UNIERP) DB와 AI 포털·사내 타부서 간 연계의 현황 대장(단일 출처). 상태가 바뀌면 여기를 먼저 고치고 §7 변경 이력에 한 줄 남긴다. 비밀값(접속 IP/ID/비밀번호)은 기록 금지 — 서버 지칭은 별칭 "ERP운영DB" 만 사용.

1. 한 줄 요약

ERP운영DB(UNIERP, 사외 IDC MS SQL Server 2022)와 AI 포털의 중간 DB(D1-a안, Supabase) 1차 5종 실적재는 2026-07-07 완료됐고, 2026-07-08에는 P4(포털·챗봇 연결)가 착수·부분 완료됐다. erp_ro(품목·영업·구매·재고·발주+테이블사전) 위에 사내 전용 노출 뷰 6종(v_erp_*, security_invoker)을 신설해 REST 직접 노출 없이 안전하게 열었고, RLS 격리 실측 테스트(JWT 시뮬레이션)로 사내만 통과·협력사/anon 차단을 확인했다. 이 위에서 부서 대시보드 1개 화면(구매 매입집계) 파일럿 실연동과 챗봇(jeil-chat) ERP Tool 4종 배포(version 7 ACTIVE)까지 이뤄졌다. 다만 유니포인트 협의(P1, 읽기전용 계정·뷰 스펙)는 여전히 대기 중이고, 현재 적재분은 초안 SQL 기반이라 협의 후 재적재 가능성이 있다. 오늘 재확인한 ERP 원천 dry-run 건수(발주 2,702·품목 58,249·매출 10·매입 63·재고 659)는 중간DB 적재분(2,699/58,246/10/63/656)보다 소폭 늘어 있어, 최신 재적재를 관리자가 !로 직접 실행하기로 결정했다(명령 준비, 미실행). 같은 날 사용자-부서-사원 매핑·대사 계층을 신설·실적재까지 완료했다 — ERP Z_USR_MAST_REC(사용자 마스터, usr_id=MS 이메일 SSOT)와 B_ACCT_DEPT(부서 마스터)를 대사해 이메일 기준 사용자↔부서↔사원 관계를 조회할 수 있는 스키마·뷰·ETL job을 구축한 뒤, 관리자 직접 실행으로 실제 ERP 데이터 적재까지 완료했다: erp_ro.dept_master_s 351건, erp_ro.usr_master_s 99건. 대사 결과 정상 매핑 89건, 불일치 10건(부서불일치 6·상태이상 3·테스트 1)이며, 불일치는 정상 매핑에서 자동 제외하는 정책이 확정됐다(마이그레이션 erp_user_dept_exclude_unmatched). 이전 결정 대기였던 부서명 처리방침은 이로써 해소됐고, 상태이상 3건(활성계정인데 (퇴사)/(휴직) 표기)만 현업 확인 권고로 남았다(전용 관리 문서 05_사용자부서_매핑대사.md 참조). 같은 날 이 사용자-부서 매핑 위에 신규 ETL job usr_erp_module을 추가해 ERP 역할·메뉴 권한(Z_USR_MAST_REC_USR_ROLE_ASSO→Z_USR_ROLE_MNU_AUTHZTN_ASSO→Z_CO_MAST_MNU)을 근거로 한 부서별 ERP 모듈 '제안값(참고)'을 만들었다 — erp_ro.usr_erp_module_s 397건 실적재 완료, 뷰 v_dept_erp_suggest/public.v_erp_dept_erp_suggest로 노출해 관리자 콘솔(04) '권한 상세'의 부서별 ERP 모듈 매트릭스에 ◆ 참고표시로만 반영했다(자동 덮어쓰기 없음, 실질 통제선은 여전히 편집형 dept_erp_scope). 실측 결과 이 조직은 ERP 읽기권한이 광범위(사용자당 평균 17개 역할)해 부서 과반 임계로도 거의 전 모듈이 제안돼 ERP-역할 신호의 부서 변별력이 약함을 확인했고, 이는 '참고 제안으로만 쓴다'는 결정의 타당성을 뒷받침한다. 같은 날 P5(민감 영역) 선행 정지작업도 착수됐다 — 급여를 민감 모듈 payroll로 신설해 dept_erp_scope를 인사팀 전용으로 한정하고 인사 페이지를 이 모듈로 서버 게이트했으며(비인사팀 직접 접근 차단), 인사 데모 페이지의 개인별 실급여는 샘플값+마스킹으로 안전화했다. 민감 분리 스키마 erp_secure(hr_payroll_m 집계 테이블·hr_access_log 감사)도 승인-준비 빈 구조로 만들었지만 실 데이터는 적재하지 않았다 — 실 적재는 경영 승인·유니포인트 View 협의·(권장) AI-Portal-HR Entra 그룹 신설이 선행돼야 한다. 같은 날 저녁 챗봇(jeil-chat) ERP 프로세스(발주·구매요청·매입·입고) 응답 품질 개선도 실적용·배포됐다. 중간DB 노출 뷰 3종을 보강해 v_erp_po_pr_link·v_erp_pur_req(둘 다 앞서 같은 날 M_PUR_REQ 연동으로 신설된 뷰, 정본 10_erp_pur_req.sql)에 발주 입고수량·구매요청 입고/매입수량·외주구분·납기경과(미입고) 플래그·연결 수주번호·요청부서(공란이면 요청자 이메일→부서 매핑으로 보완) 컬럼을 append했고(기존 컬럼 순서·타입 보존, 행수 2,703 불변 검증), 발주번호 단위 총액을 올바로 집계하는 신규 뷰 v_erp_pur_top_po를 추가했다(정본 SQL 실제구축준비 자료/이관/sql/15_erp_process_link_enrich.sql). 같은 배포에서 [치명적 버그 수정] 챗봇 도구 호출 오케스트레이션을 1라운드→상한 멀티라운드(MAX_ROUNDS=4)로 교체했다 — 기존엔 1차 스트림에서 도구를 실행한 뒤 2차 스트림에 도구를 다시 안 줘서, 모델이 특정 발주에 포털 도구를 먼저 골라 허탕치면 "ERP 도구로 재조회하라"는 안내를 받고도 재호출을 못 해 안내문만 되풀이하는 사용자 루프가 있었는데, 이제 라운드마다 도구 수집→실행→누적하고 마지막 라운드만 도구 없이 최종답변하도록 해 해소했다(무한루프 3중 차단: 라운드 상한 + 직전 동일호출 반복 중단 + 마지막 라운드 강제 무도구). 상태코드 RQ(요청)→CF(확정)→PO(발주완료·입고전)→GR(입고완료)→IV(매입완료) 한글 해석을 도구 출력·프롬프트에 주입하고, 신규 도구 2종(get_erp_receipt_pending 미입고·납기경과 발주, get_erp_pur_req 상태·부서별 구매요청)을 추가해 jeil-chat version 14 ACTIVE로 배포했다. 실측 과정에서 중간DB 데이터 갭도 추가로 드러났다 — inventory_d.in_qty 전량 0·stock_qty 전량 null(출고만 유효), sales_orders_m.order_amt·collect_amt 전월 0(수주·수금 미매핑), pur_order_s.subcontra_flg 전건 'N'(외주 발주 0건) — 원천 ETL 매핑 점검이 필요해 §6에 결정대기로 정리했다. 같은 날 인사 급여 집계도 승인-준비 단계를 넘어 실적재까지 완료됐다 — erp_secure.hr_payroll_m에 ETL job hr_payroll(HDF070T 월×부서 집계·HGA070T 월×전사 집계)로 143행(급여 월×부서 137 + 퇴직 월×전사 6, 2026.01~06 6개월분)을 관리자 직접 실행으로 적재했다 — 컬럼은 ym·dept_nm·headcount·pay_tot_amt·retire_amt뿐이고 개인별·이름·주민번호·계좌는 여전히 포함하지 않는다(집계만). 이 위에 게이트 API jeil-hr(Edge Function) v1을 배포해 Entra 검증 → dept='인사팀'/portal_admin만 통과(그 외 403) → 감사 로그(hr_access_log) → 집계 반환 흐름을 갖췄고, 인사 페이지는 인사팀 권한자에게 이 실 집계를 보여준다(비권한·데모는 마스킹·샘플 유지). 자금(F 모듈) 실적재, AI-Portal-HR Entra 보안그룹 신설, 유니포인트와의 재직상태 정의 확인은 여전히 미착수·잔여 항목이다. 2026-07-13에는 재고 입출고(inventory) ETL의 데이터 커버리지 결함을 발견·해소했다 — 1차 5종 ETL 중 inventory job(M_PUR_GOODS_MVMT)만 다른 job(발주·매출·매입)과 달리 롤링 31일(daily_start) 창으로 남아 있어 최근 6·7월 데이터만 적재돼 있었는데, 이를 발주·매출·매입 job과 동일한 WHERE MVMT_DT >= ? AND MVMT_DT < ? + year_start/year_end 파라미터로 수정(커밋 811cbd8, main 배포)하고 관리자 직접 실행으로 재적재했다. 결과 erp_ro.inventory_d는 2026년 1~7월 전체 1,810행(월별 01:93·02:134·03:149·04:260·05:469·06:650·07:55)으로 늘었고 etl_meta.batch_run에 성공 이력(rows_read=1810·rows_upserted=1810)이 남았다. 참고로 발주현황 보드의 입고·매입 상태 표시는 pur_order_s(po_sts·rcpt_qty) 기반이라 이번 작업 전에도 이미 연중 전체가 반영돼 있었고, 이번 백필은 재고 입출고 일집계(inventory_d)에 한정된 보완이다.

현재 위치는 P0(정책 수립) 마무리 → P1(유니포인트 협의) 진입 대기 + P3(중간DB) 내부 준비 계속(재고 ETL 연 전체 적재 전환 포함) + P4(포털·챗봇 연결) 착수(부분 완료, 사용자-부서 매핑·부서별 ERP모듈 제안값 실적재·가동 + 챗봇 ERP 프로세스 응답 품질 개선 포함) + P5(민감 영역) 인사 급여 집계 실적재 완료(자금 미착수, Entra HR 그룹 미신설)다.
