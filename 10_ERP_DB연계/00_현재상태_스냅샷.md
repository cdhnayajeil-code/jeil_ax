## 1. 한 줄 요약

ERP운영DB(UNIERP, 사외 IDC MS SQL Server 2022)와 AI 포털의 **중간 DB(D1-a안, Supabase) 1차 5종 실적재는 2026-07-07 완료**됐고, **2026-07-08에는 P4(포털·챗봇 연결)가 착수·부분 완료**됐다. `erp_ro`(품목·영업·구매·재고·발주+테이블사전) 위에 사내 전용 노출 뷰 6종(`v_erp_*`, `security_invoker`)을 신설해 REST 직접 노출 없이 안전하게 열었고, RLS 격리 실측 테스트(JWT 시뮬레이션)로 사내만 통과·협력사/anon 차단을 확인했다. 이 위에서 부서 대시보드 1개 화면(구매 매입집계) 파일럿 실연동과 챗봇(`jeil-chat`) ERP Tool 4종 배포(version 7 ACTIVE)까지 이뤄졌다. 다만 유니포인트 협의(P1, 읽기전용 계정·뷰 스펙)는 여전히 대기 중이고, 현재 적재분은 초안 SQL 기반이라 협의 후 재적재 가능성이 있다. 오늘 재확인한 ERP 원천 dry-run 건수(발주 2,702·품목 58,249·매출 10·매입 63·재고 659)는 중간DB 적재분(2,699/58,246/10/63/656)보다 소폭 늘어 있어, 최신 재적재를 관리자가 `!`로 직접 실행하기로 결정했다(명령 준비, 미실행). 같은 날 **사용자-부서-사원 매핑·대사 계층을 신설·실적재까지 완료**했다 — ERP `Z_USR_MAST_REC`(사용자 마스터, `usr_id`=MS 이메일 SSOT)와 `B_ACCT_DEPT`(부서 마스터)를 대사해 이메일 기준 사용자↔부서↔사원 관계를 조회할 수 있는 스키마·뷰·ETL job을 구축한 뒤, 관리자 직접 실행으로 **실제 ERP 데이터 적재까지 완료**했다: `erp_ro.dept_master_s` 351건, `erp_ro.usr_master_s` 99건. 대사 결과 **정상 매핑 89건**, **불일치 10건**(부서불일치 6·상태이상 3·테스트 1)이며, **불일치는 정상 매핑에서 자동 제외**하는 정책이 확정됐다(마이그레이션 `erp_user_dept_exclude_unmatched`). 이전 결정 대기였던 부서명 처리방침은 이로써 해소됐고, 상태이상 3건(활성계정인데 (퇴사)/(휴직) 표기)만 현업 확인 권고로 남았다(전용 관리 문서 `05_사용자부서_매핑대사.md` 참조). 같은 날 이 사용자-부서 매핑 위에 **신규 ETL job `usr_erp_module`**을 추가해 ERP 역할·메뉴 권한(`Z_USR_MAST_REC_USR_ROLE_ASSO`→`Z_USR_ROLE_MNU_AUTHZTN_ASSO`→`Z_CO_MAST_MNU`)을 근거로 한 **부서별 ERP 모듈 '제안값(참고)'**을 만들었다 — `erp_ro.usr_erp_module_s` 397건 실적재 완료, 뷰 `v_dept_erp_suggest`/`public.v_erp_dept_erp_suggest`로 노출해 관리자 콘솔(04) '권한 상세'의 부서별 ERP 모듈 매트릭스에 ◆ 참고표시로만 반영했다(자동 덮어쓰기 없음, 실질 통제선은 여전히 편집형 `dept_erp_scope`). 실측 결과 이 조직은 ERP 읽기권한이 광범위(사용자당 평균 17개 역할)해 부서 과반 임계로도 거의 전 모듈이 제안돼 **ERP-역할 신호의 부서 변별력이 약함**을 확인했고, 이는 '참고 제안으로만 쓴다'는 결정의 타당성을 뒷받침한다. 같은 날 **P5(민감 영역) 선행 정지작업도 착수**됐다 — 급여를 민감 모듈 `payroll`로 신설해 `dept_erp_scope`를 인사팀 전용으로 한정하고 인사 페이지를 이 모듈로 서버 게이트했으며(비인사팀 직접 접근 차단), 인사 데모 페이지의 개인별 실급여는 샘플값+마스킹으로 안전화했다. 민감 분리 스키마 `erp_secure`(`hr_payroll_m` 집계 테이블·`hr_access_log` 감사)도 **승인-준비 빈 구조**로 만들었지만 **실 데이터는 적재하지 않았다** — 실 적재는 경영 승인·유니포인트 View 협의·(권장) `AI-Portal-HR` Entra 그룹 신설이 선행돼야 한다. 같은 날 저녁 **챗봇(jeil-chat) ERP 프로세스(발주·구매요청·매입·입고) 응답 품질 개선**도 실적용·배포됐다. 중간DB 노출 뷰 3종을 보강해 `v_erp_po_pr_link`·`v_erp_pur_req`(둘 다 앞서 같은 날 `M_PUR_REQ` 연동으로 신설된 뷰, 정본 `10_erp_pur_req.sql`)에 발주 입고수량·구매요청 입고/매입수량·외주구분·납기경과(미입고) 플래그·연결 수주번호·요청부서(공란이면 요청자 이메일→부서 매핑으로 보완) 컬럼을 append했고(기존 컬럼 순서·타입 보존, 행수 2,703 불변 검증), 발주번호 단위 총액을 올바로 집계하는 신규 뷰 `v_erp_pur_top_po`를 추가했다(정본 SQL `실제구축준비 자료/이관/sql/15_erp_process_link_enrich.sql`). 같은 배포에서 **[치명적 버그 수정]** 챗봇 도구 호출 오케스트레이션을 1라운드→상한 멀티라운드(`MAX_ROUNDS=4`)로 교체했다 — 기존엔 1차 스트림에서 도구를 실행한 뒤 2차 스트림에 도구를 다시 안 줘서, 모델이 특정 발주에 포털 도구를 먼저 골라 허탕치면 "ERP 도구로 재조회하라"는 안내를 받고도 재호출을 못 해 안내문만 되풀이하는 사용자 루프가 있었는데, 이제 라운드마다 도구 수집→실행→누적하고 마지막 라운드만 도구 없이 최종답변하도록 해 해소했다(무한루프 3중 차단: 라운드 상한 + 직전 동일호출 반복 중단 + 마지막 라운드 강제 무도구). 상태코드 RQ(요청)→CF(확정)→PO(발주완료·입고전)→GR(입고완료)→IV(매입완료) 한글 해석을 도구 출력·프롬프트에 주입하고, 신규 도구 2종(`get_erp_receipt_pending` 미입고·납기경과 발주, `get_erp_pur_req` 상태·부서별 구매요청)을 추가해 `jeil-chat` **version 14 ACTIVE**로 배포했다. 실측 과정에서 중간DB 데이터 갭도 추가로 드러났다 — `inventory_d.in_qty` 전량 0·`stock_qty` 전량 null(출고만 유효), `sales_orders_m.order_amt`·`collect_amt` 전월 0(수주·수금 미매핑), `pur_order_s.subcontra_flg` 전건 'N'(외주 발주 0건) — 원천 ETL 매핑 점검이 필요해 §6에 결정대기로 정리했다. **같은 날 인사 급여 집계도 승인-준비 단계를 넘어 실적재까지 완료됐다** — `erp_secure.hr_payroll_m`에 ETL job `hr_payroll`(`HDF070T` 월×부서 집계·`HGA070T` 월×전사 집계)로 **143행**(급여 월×부서 137 + 퇴직 월×전사 6, 2026.01~06 6개월분)을 관리자 직접 실행으로 적재했다 — 컬럼은 `ym·dept_nm·headcount·pay_tot_amt·retire_amt`뿐이고 **개인별·이름·주민번호·계좌는 여전히 포함하지 않는다**(집계만). 이 위에 게이트 API **`jeil-hr`(Edge Function) v1을 배포**해 Entra 검증 → `dept='인사팀'`/`portal_admin`만 통과(그 외 403) → 감사 로그(`hr_access_log`) → 집계 반환 흐름을 갖췄고, 인사 페이지는 인사팀 권한자에게 이 실 집계를 보여준다(비권한·데모는 마스킹·샘플 유지). 자금(F 모듈) 실적재, `AI-Portal-HR` Entra 보안그룹 신설, 유니포인트와의 재직상태 정의 확인은 여전히 미착수·잔여 항목이다. **2026-07-13에는 재고 입출고(inventory) ETL의 데이터 커버리지 결함을 발견·해소했다** — 1차 5종 ETL 중 `inventory` job(`M_PUR_GOODS_MVMT`)만 다른 job(발주·매출·매입)과 달리 롤링 31일(`daily_start`) 창으로 남아 있어 최근 6·7월 데이터만 적재돼 있었는데, 이를 발주·매출·매입 job과 동일한 `WHERE MVMT_DT >= ? AND MVMT_DT < ?` + `year_start`/`year_end` 파라미터로 수정(커밋 `811cbd8`, main 배포)하고 관리자 직접 실행으로 재적재했다. 결과 `erp_ro.inventory_d`는 2026년 1~7월 전체 **1,810행**(월별 01:93·02:134·03:149·04:260·05:469·06:650·07:55)으로 늘었고 `etl_meta.batch_run`에 성공 이력(rows_read=1810·rows_upserted=1810)이 남았다. 참고로 발주현황 보드의 입고·매입 상태 표시는 `pur_order_s`(po_sts·rcpt_qty) 기반이라 이번 작업 전에도 이미 연중 전체가 반영돼 있었고, 이번 백필은 재고 입출고 일집계(`inventory_d`)에 한정된 보완이다.

**2026-07-20에는 P4(포털·챗봇 연결)에서 그동안 비어 있던 '챗봇↔권한체계' 연결고리를 메웠다.** 발단은 관리자 계정(dh.choi@jeilm.co.kr, `portal_admin`)이 챗봇(jeil-chat)에 "2026년 인원현황"·"권한 확인"을 물었을 때 오답이 나온 문제였다. 조사 결과 권한 체계(`jeil-me`/`jeil-hr`/`dept_erp_scope`/`portal_page`)와 인원 데이터(`erp_secure.hr_payroll_m`, 2026-01~06 월별 전사 총원 113·119·101·100·94·98명)는 이미 정상 구현·적재 완료 상태였고, 원인은 **챗봇(jeil-chat)이 이 체계에 연결돼 있지 않았던 것**이었다. 이를 메우기 위해 `jeil-chat` Edge Function을 **version 18 → version 19**로 배포했다. 신규 도구 3종: ①`get_my_access` — 로그인 본인의 역할(관리자/부서관리자/일반)·소속부서·열람가능 ERP 모듈·접근가능/불가 페이지·권한요청 방법을 반환(호출자 UPN 고정으로 타인 권한 조회 불가, 전 직원 사용 가능). ②`get_hr_headcount` — 인원현황. 전 직원에게는 월별 전사 총원만, 부서별 인원 분포는 `payroll` 권한자(인사팀·관리자)에게만 반환(급여 금액 미포함). ③`get_hr_payroll` — 급여 집계(인사팀·관리자 전용). `erp_secure`는 REST 미노출이라 기존 `jeil-hr`와 동일하게 service_role RPC `hr_payroll_get` 경유. 인사 도구(`get_hr_headcount`·`get_hr_payroll`) 접근은 허용·거부 모두 `hr_access_log`에 감사 기록된다(`jeil-hr`와 동일 원장 재사용). SYSTEM_PROMPT도 보강했다 — 권한 질의는 반드시 `get_my_access`로 실조회하도록, 권한 밖 수치 추정·역산 금지, 인원은 급여대장(HDF070T) 기준 마감 전 값임을 명시.

**같은 배포에서 소급 발견된 권한 구멍도 차단했다(중요)** — 기존 포털 도구 3종 `get_order_summary`·`get_order_detail`·`get_inspection_pending`이 그동안 `ERP_TOOL_MODULE` 매핑이 없어 권한 강제 없이 **전 직원에게 협력사 발주번호·금액·납기를 노출**하고 있었다. 이번에 `pur_order` 모듈 권한으로 강제 적용해 차단했다 — 2026-07-08 v14에서 "ERP 도구 8종을 `pur_order`로 매핑"한 작업(§4 챗봇 ERP Tool)은 ERP 도구 쪽만 대상이었고, 포털 도구 3종은 그 매핑에서 빠져 있었다는 뜻이다(즉 이번 취약점은 그날 이후 계속 실재했다). 배포는 `verify_jwt=false` 유지, 배포본이 로컬 소스와 문자 단위 일치 검증됐다.

**관리자 결정(2026-07-20)**: ①인원현황 공개 등급 — 전사 총원은 전 직원 조회 가능, 부서별 인원 분포·급여액은 인사팀·관리자 전용. ②협력사 발주·검사 챗봇 조회 권한은 신규 모듈 신설 대신 기존 `pur_order` 모듈 권한으로 강제. **경영 판단 보류(이번엔 변경 안 함)**: `dept_erp_scope`가 23개 부서 전부 sales·purchase·inventory·item·pur_order를 동일 보유해 부서별 데이터 차등이 사실상 없음(payroll만 인사팀 단독 차등) — 구조는 차등을 지원하므로 정책 결정만 필요한 상태다(§6 결정대기 참조). 소스 파일(`supabase/functions/jeil-chat/index.ts`)은 수정·배포 완료 상태이나 git 커밋은 관리자가 별도로 진행한다(본 에이전트는 커밋하지 않음).
