# 02. ERP DB 연계 — 진행상태 (지속 갱신)

> **관리: 최동혁** · 전담 에이전트: `erp-db-link-manager`
> 페이즈·체크리스트·작업로그를 지속 갱신하는 대장. 작업을 수행하면 §2 체크리스트와 §3 작업로그를 함께 갱신하고, 미러(`정책관리/ERP_DB연계/`)에 동기화한다.

## 1. 현재 위치

**P0(정책 수립) 마무리 — P1(유니포인트 협의) 진입 대기 + P3(중간DB) 내부 준비 계속 + P4(포털·챗봇 연결) 착수·부분완료** (2026-07-08 기준)

```
[P0 정책수립]──▶[P1 유니포인트 협의]──▶[P2 리포팅 계층]──▶[P3 중간DB+ETL]──▶[P4 포털·챗봇 연결]──▶[P5 민감영역]
   ● 현재            ○ 대기               ○              ◐ 스키마·스크립트 준비    ◐ 착수·부분완료        ○
```

차단 요인: ①경영 티어 결정(Tier A vs B — 마스터플랜 v2.0 §11), ②마스터플랜 승인. ~~③(P3 실적재) `ERP_DB_CONN` 미입력~~ → **해소·실적재 완료(2026-07-07)**: `etl/_erp_conn.py` 신설 — `ERP_DB_CONN`이 비어 있으면 관리자 PC의 `%USERPROFILE%\.erp\`(DPAPI 암호화 저장소)에서 접속 문자열을 런타임 조립하는 폴백(`.env` 평문 저장 금지 보안규칙 준수, 서버 배포 시엔 종전대로 환경변수). 전 job `--dry-run` 검증 후 관리자 직접 실행으로 **1차 5종 실적재 완료**(발주 2026년 우선연동 포함). 남은 것: 야간배치 정기화·유니포인트 협의(SQL 초안 확정).

P1(유니포인트 협의)이 열리기 전이라도 ERP 접속이 필요 없는 내부 준비(중간 DB 스키마·적재 RPC·ETL 스크립트 작성)는 먼저 진행했다(2026-07-07). 같은 날 `erp_ro.table_dict`에 ERP 테이블 사전 2,658건(컬럼 문서화 452개, 11개 모듈)을 적재 완료했다(로컬 메타 기반, 운영 ERP DB 접속 없음). 실제 ERP 데이터 적재는 여전히 유니포인트 협의(뷰·계정) 이후에만 가능하다.

**신규 요청 접수(2026-07-07)**: 발주 데이터(`M_PUR_REQ` 발주요청·`M_PUR_ORD` 발주, §4.4 협력사 포털 매핑)를 **2026년도분만 우선연동**하는 안이 접수됐다. 현재 1차 4종(품목·영업·구매=매입·재고)에는 포함되지 않은 별도 범위이며, 본 에이전트는 문서 관제 담당으로 ERP DB 직접 조회·ETL 코드 실행을 하지 않는다(§4 이슈·메모 요청 목록 참조) — 실행은 관리자(`ERP_DB_CONN` 입력)·유니포인트 협의·ETL 개발(별도 세션/erp-db-analyst) 이후 가능하다.

**P4 착수(2026-07-08)**: 오늘 ERP→중간DB dry-run으로 원천 5종을 재확인한 결과 건수가 소폭 늘어 있었다(발주 2,702·품목 58,249·매출 10·매입 63·재고 659, 중간DB 적재분은 아직 2026-07-07 기준 2,699/58,246/10/63/656). 최신 재적재는 관리자가 `!`로 직접 실행하기로 결정했고(명령 준비, 미실행) 본 에이전트는 실행하지 않는다.

같은 날 **노출 계층(P4)을 신설**했다 — Supabase에 사내 전용 `security_invoker` 뷰 6종(`v_erp_sales_monthly`·`v_erp_purchase_monthly`·`v_erp_inventory_daily`·`v_erp_item`·`v_erp_pur_order`·`v_erp_data_asof`)을 public에 만들고(`erp_ro`/`etl_meta`는 여전히 REST 비노출), 챗봇(jeil-chat) Edge Function이 service_role로 조회할 수 있게 GRANT를 추가했다(마이그레이션 2건: `erp_pub_reporting_views`·`erp_ro_grant_service_role_read`). RLS 격리 실측 테스트(JWT 시뮬레이션)도 통과했다 — 사내: 정상(매출10·매입63·재고656·품목58,246·발주2,699·기준시각5행), 협력사(vendor): 전부 0행, anon: 차단.

이 노출 계층 위에서 실제 연결도 시작했다 — `app/lib/api.js`에 `erpApi`를 추가해 `pages/구매_거래처별매입집계_2026.html` 1개 화면을 사내 로그인 시 실데이터로 파일럿 전환했고(비로그인·협력사는 목업 폴백), 챗봇에는 ERP Tool 4종(`get_erp_sales_monthly`·`get_erp_purchase_monthly`·`get_erp_inventory_status`·`get_erp_item`)을 추가해 배포했다(version 7 ACTIVE, ERP 중간DB Tool 2단계 착수). 남은 선행·이슈는 §4 참조.

**사용자-부서-사원 매핑·대사 신설 + 실적재 완료(2026-07-08)**: ERP `Z_USR_MAST_REC`(사용자 마스터, `usr_id`=MS 이메일 SSOT)와 `B_ACCT_DEPT`(부서 마스터)를 대사하는 계층을 구축했다 — 원천 미러 2종(`usr_master_s`·`dept_master_s`) + 파싱·대사 뷰 5종(`erp_ro`) + 노출 뷰 3종(`public.v_erp_user_dept`·`v_erp_user_dept_recon`·`v_erp_dept_roster`) + ETL job 2종(`usr_master`·`dept_master`). 관리자 직접 실행(`!`)으로 **실적재까지 완료**했다 — `dept_master_s` 351건·`usr_master_s` 99건. 대사 결과 **정상 매핑 89건**, **불일치 10건**(부서불일치 6·상태이상 3·테스트 1)이며, **불일치는 정상 매핑(연동용)에서 자동 제외**하는 정책이 확정됐다(마이그레이션 `erp_user_dept_exclude_unmatched`). 이전 "부서명 처리방침 결정 대기"는 이로써 해소됐고, 상태이상 3건(활성계정인데 (퇴사)/(휴직) 표기 — 김해품질팀·영업팀·자재물류파트)만 현업 확인 권고로 남았다. 전용 관리 문서 [`05_사용자부서_매핑대사.md`](05_사용자부서_매핑대사.md) 갱신. 정본 SQL: `실제구축준비 자료/이관/sql/09_erp_user_dept_mapping.sql`.

**부서별 ERP 모듈 제안값(참고) 신설(2026-07-08)**: 위 사용자-부서 매핑 위에 신규 ETL job `usr_erp_module`을 추가해, ERP 역할·메뉴 권한(`Z_USR_MAST_REC_USR_ROLE_ASSO`→`Z_USR_ROLE_MNU_AUTHZTN_ASSO`→`Z_CO_MAST_MNU`)을 근거로 부서별 ERP 모듈 '제안값(참고)'을 만들었다. `erp_ro.usr_erp_module_s` **397건** 실적재(관리자 실행) + 뷰 `v_dept_erp_suggest`/`public.v_erp_dept_erp_suggest` 신설, 관리자 콘솔(04) '권한 상세' ERP 모듈 매트릭스에 ◆ 참고표시로 반영했다(자동 덮어쓰기 안 함 — 관리자 결정). **실측 소견**: 이 조직은 ERP 읽기권한이 광범위(사용자당 평균 17개 역할)해 부서 과반(≥50%) 임계로도 거의 모든 부서에 전 모듈이 제안됨 → ERP-역할 신호는 부서 변별력이 약하고, 실질 통제선은 여전히 편집형 `dept_erp_scope`(업무기준)임을 확인했다. 정본 SQL `실제구축준비 자료/이관/sql/12_erp_dept_module_suggest.sql`.

## 4. 이슈·메모

- **부서별 ERP 모듈 제안값(2026-07-08) — ERP-역할 신호 변별력 약함**: 사용자당 평균 17개 역할로 ERP 읽기권한이 광범위해, 부서 과반(≥50%) 임계로 한정해도 거의 모든 부서에 전 모듈(SD/MM/IM/MDM)이 제안된다. 즉 ERP 역할·메뉴 권한만으로는 부서별 실제 업무범위를 변별하기 어렵다 — **편집형 `dept_erp_scope`(업무기준)가 여전히 실질 권위**이며, 본 제안값은 관리자 콘솔에 참고표시(◆)로만 노출한다(자동 덮어쓰기 금지, 관리자 결정 확정). 후속 개선이 필요하면 역할명 세분화·메뉴 단위(MNU) 접근빈도 등 추가 신호를 검토할 것.
- **사용자-부서 매핑(2026-07-08) — 실적재 완료, 잔여이슈**:
