> **관리: 최동혁** · 전담 에이전트: `erp-db-link-manager`

**연동 현황 가시성 개선(2026-07-21 오전)**: 연동 현황 화면 `app/erp-status.html`과 원천 뷰 `public.v_erp_sync_overview`는 이미 가동 중이었으나 이번에 처음 본 관제 문서에 등재됐다(소급 등재 — 지금까지 문서화 공백이었음). 점검 결과 이 뷰가 1차 5종(발주·품목·매출·매입·재고)만 노출하고 있었는데, 실제로는 §2 P3·P4·P5에서 이미 실적재를 마친 **10종**(+구매요청·부서마스터·사용자마스터·부서별ERP모듈·급여집계)이 적재돼 있어 절반이 화면에 보이지 않는 상태였다. 마이그레이션 `erp_sync_overview_all_sources`로 뷰를 10종 전체로 확장하고(`sensitive`·`sort` 컬럼 추가, 급여는 실테이블 미count·배치건수만), `jeil-chat-admin`을 **version 9 → 10**으로 배포해 관리자 콘솔 'API' 탭에 실데이터 표를 신설했으며, `app/erp-status.html`에 연결 정보 표(비밀값 미기재)를 추가했다. 상세는 §3 작업로그, 결과 반영은 `00_현재상태_스냅샷.md` §1·§4(v0.12) 참조.

**매입 상세(`iv_dtl`) 신규 연동 + 구매요청 상세 백필 — 실적재 완료(2026-07-21 오후)**: 같은 날 오전 준비(스키마·ETL·뷰, `00_관리체계/03_변경관리_CHANGELOG` 첫 줄 참조)를 관리자가 직접 실행해 실적재까지 완료했다(`batch_run` 7건 성공·0건 실패). 신규 `iv_dtl`(`M_IV_DTL`): 2026-01-05~07-20 **1,981행**(0.9초) — 송장 213건·발주 432건, 매입금액 합계 35.6억원, 발주 연결(`po_no`) 100%(발주 미연결 36건은 데이터 품질 참고 항목, 임의 조치 없음). `pur_req` 상세 14컬럼 백필: `--full` **4,063행**(2.1초) — 조달구분 채움 99.98%(미채움 1건은 ERP 원천 `PROCURE_TYPE` NULL), 도면번호 856건. 매입 상세를 신규 적재한 김에 기존 월집계(`purchase_m`)와 대사한 결과 6월 +19.2억·7월 +1.5억 차이를 발견했으나(1~5월 일치) **원인은 데이터 오류가 아니라 적재 시점 차이**(2026-07-08 이후 ERP 신규 등록분 미반영)로 확인 — `purchase --full` 151행·`pur_order`(증분) 512행·`sales` 37행·`item_master`(증분) 415행·`inventory` 1,847행을 재적재해 **매입 대사 차이를 전월 0으로 해소**했다(매핑·로직 정상 검증). `hr_payroll`(민감)은 이번 재적재 대상에서 제외했다. 상세는 §3 작업로그, 결과 반영은 `00_현재상태_스냅샷.md` §1·§4·§6(v0.13), `01_연계기획.md` §4.10 참조.

**챗봇 ERP 조회 정합성 이슈 — RCA·조치 완료(2026-07-22)**: 사용자 제보(품목코드 `S3041-00065`의 발주·구매요청을 챗봇에 물으면 실제 ERP·중간DB에는 있는데도 "없다"고 답하거나 품목코드를 발주번호로 혼동, "supabase 데이터가 ERP와 안 맞게 연결된 것 아닌가" 우려 제기)를 조사했다. **원인분석 결론(SQL·코드 실측): 데이터·매핑 문제 아님·모델 이슈 아님 — 챗봇 도구 설계 공백이 원인**이다. 데이터·매핑은 정상 확인됐다 — `erp_ro`에 이 품목의 발주 `PO202607210001`(2026-07-21·(주)대구렌탈·341.6만원·상태 PO·연결 `PR202607020013`)과 구매요청 `PR202607020013`(2026-07-02·사업운영팀·수량1)이 품목코드·품목명·po_no·pr_no 매핑까지 정확히 적재돼 있었다. **주원인**: 품목코드로 발주/구매요청을 역조회하는 챗봇 도구가 없었다 — `v_erp_po_pr_link`에 item_code 컬럼이 있는데도 `get_erp_po_pr`·`get_erp_receipt_pending`은 po_no/pr_no/po_sts로만 필터했고, `get_erp_item`은 품목 마스터만 반환(발주·구매요청 미반환)했다. **부원인**: SYSTEM_PROMPT가 "PO/PR 번호"로만 조회를 안내하고 품목코드→발주 이력 조회 시나리오를 지정하지 않았다. **혼동 유발**: `get_erp_po_pr`가 품목코드를 po_no 자리에 넣어도 걸러내지 않고 0건을 조용히 반환해 "없음"·발주번호 오인을 낳았다. **조치(완료, `jeil-chat` version 26 배포)**: 신규 도구 `get_erp_item_orders(item)` — 품목코드/품목명으로 그 품목의 구매요청·발주·매입 이력(요청→발주→입고→매입 추적)을 반환(`v_erp_pur_req`·`v_erp_pur_order`·`v_erp_iv_dtl`를 item_code로 조회, `pur_order` 모듈 권한 강제, list 뷰). SYSTEM_PROMPT 보강(PO…/PR…/품목코드 문자-숫자 형식 구분 + 품목코드 이력 질의는 `get_erp_item_orders` 지정). `get_erp_po_pr` 입력 가드 추가 — PO/PR 형식이 아니면(품목코드 오인) 0건 대신 재안내(재시도 도구로 `get_erp_item_orders` 명시). SQL 실측으로 `S3041-00065`·`AACZ15-00002`(구매요청 12건·발주 11건) 등 정상 반환을 확인했다. **데이터 범위 한계(변경 없음, 명시만)**: `erp_ro`는 2026년만 적재(연도 필터)돼 있어 2025년 이전 발주·구매요청은 조회 불가하다 — 이전 연도 확장은 별도 결정 사항(§4 이슈·메모 참조, 임의 확장 안 함).

**중간DB(erp_ro) 정합성 SQL 검증 — 실측 완료(2026-07-22, 읽기전용)**: 위 RCA 조치와 별개로, 사용자가 제기한 우려("Supabase 중간DB가 실제 ERP와 제대로 안 맞게 연결된 것 아닌가")에 직접 답하기 위해 `erp_ro`(중간DB)의 적재 정합성을 SQL로 실측 점검했다(erp-db-link-manager는 DB에 직접 쓰지 않으며, 이번 점검은 기존 읽기전용 조회 결과를 문서화한 것). **결론: 적재 정합성 이상 없음, 구조적 오류·누락 미발견.** 적재 현황 — `pur_order_s` 3,088행(2026-01-07~07-21, 2,798품목) · `pur_req_s` 4,064행(2026-01-06~07-21, 3,300품목) · `iv_dtl_s` 1,981행(2026-01-05~07-20, 1,855품목) · `item_master_s` 58,533행(`item_code` 고유 58,533). 점검 결과 전건 이상 없음 — item_code NULL/공백 0건(3종 전부), 품목마스터 고아(발주·구매요청·매입→`item_master_s` 미존재) 각 0건, 번호 형식 위반(발주 `po_no`/`pr_no`, 구매요청 `pr_no` 정규식) 0건(100% 규약 준수), PK 중복(발주 `po_no+po_seq`, 품목마스터 `item_code`) 0건. `etl_meta.batch_run` 최근 12건 전부 success이며 모든 잡에서 `rows_read = rows_upserted`(적재 중 누락 0), 최종 적재 2026-07-21 08:09 UTC. **발견 사항 2건(둘 다 오류 아님 — 알려진 한계로 확인)**: ①발주→구매요청 고아 49건 — 발주에 `pr_no`는 있으나 `pur_req_s`에 없는 건인데, 실물 확인 결과 49건 전부 2025년 PR(`PR2025…`)을 참조하는 연초 발주(발주일 2026-01-07~01-19)였다. 매핑 오류가 아니라 `erp_ro`가 2026년만 적재하는 범위 한계의 정상적 귀결. ②(`po_no`+`item_code`) 중복 31건 — 실물 확인 결과 동일 PO 안에서 `po_seq`가 다른 별개 발주 라인이며 각각 다른 구매요청(`pr_no`)에서 유래한 것(예: `PO202601300002`/`S20111-00008`이 `po_seq` 2(`PR202601290001`)·4(`PR202601290003`)로 분리)으로, 중복 적재가 아니라 ERP 원본 구조를 정확히 보존한 결과다. 진짜 PK인 (`po_no`+`po_seq`) 중복은 0건. `S3041-00065` 재확인: 발주 1라인·구매요청 1라인 정상 존재, `get_erp_item_orders`가 정확 반환함을 라이브 재검증했다. **함께 확인된 한계·잔여 과제**: ①데이터 최신성 — 최종 적재 2026-07-21이라 당일(2026-07-22) 발생분은 미반영(ETL이 수동 실행 방식이라 실시간 아님, 사용자 안내 필요). ②범위 한계 — `erp_ro`는 2026년만 적재(2025년 이전 발주·구매요청 챗봇 조회 불가, 기존 알려진 한계 재확인). ③**미실행(잔여 과제)** — ERP 원본(`M_PUR_ORD`/`M_PUR_REQ`) 직접 대사는 이번에 수행하지 못했다. ODBC 접속 설정(`.env`) 읽기가 권한 거부되어 중단됐다 — 향후 관리자가 `10_ERP_DB연계/etl/`에서 `python etl_run.py --job pur_order --dry-run` 등으로 직접 실행해 원본 건수와 대사할 것을 §2 체크리스트·§4 이슈·메모에 잔여 항목으로 등재한다.

**협력사 거래처마스터(`vendor_master`) 재적재·정합화 + ERP 거래처마스터 원천 연동(`bp_master`) 준비(2026-07-22 오후)**: 사용자 제보(협력사 정보 화면 `app/admin-vendors.html`에서 발주거래처가 보이지 않음)를 조사했다. 원인은 `public.vendor_master`가 초기 데모 시드 17개사에 머물러 ERP 실제 발주거래처 103개사 중 87개사가 미등록이었던 것 — 추가로 기존 17건의 사업자번호는 거래처코드 기반으로 만들어진 가짜 값이었다. 관리자가 직접 재적재를 실행해 **109개사**(발주·매입·매입상세 합집합)로 정합화하고(거래처명 5건 교정, 발급 계정 4건 무영향·고아 0건), 가짜 사업자번호 17건은 제거(NULL)했다. ERP 거래처마스터(`B_BIZ_PARTNER`) 원천 연동을 위해 신규 스키마 `erp_ro.bp_master_s`(대표자주민등록번호·은행계좌번호는 최소수집으로 적재 제외)와 ETL job `bp_master`를 준비했으나 **ODBC 접속이 필요해 미실행**이다(관리자 실행 대기). 관리자 전용 RPC `public.vendor_master_sync()`(비관리자 차단·멱등성 검증)를 신설하고 관리자 화면에 "거래처 동기화" 버튼을 추가했다. 부수 효과로 협력사 모바일 포털·계정발급 Edge Function의 거래처명도 정상화됐다. 라이브 검증 완료(커밋 `c11d632`). 상세는 §3 작업로그, 결과 반영은 `00_현재상태_스냅샷.md` §1·§4·§5·§7(v0.16), `01_연계기획.md` §4.11 참조.

**재고 입출고(inventory) ETL — 롤링 31일 → 2026년 전체 적재 전환 + 재적재 완료(2026-07-13)**: 1차 5종 ETL 중 `inventory` job(`M_PUR_GOODS_MVMT`)만 다른 job(발주·매출·매입)과 달리 롤링 31일(`daily_start`) 창으로 남아 있어, 실제로는 최근 6·7월 데이터만 적재돼 있고 1~5월분은 누락돼 있던 결함을 발견·수정했다. `etl_run.py`의 `inventory` job SQL을 `WHERE MVMT_DT >= ?`(단일 파라미터, daily_start)에서 발주·매출·매입 job과 동일한 `WHERE MVMT_DT >= ? **챗봇(jeil-chat) 권한체계 연결 + 포털 도구 권한 구멍 차단 — version 18 → 19 배포(2026-07-20)**: 관리자 계정(`portal_admin`)이 챗봇에 "2026년 인원현황"·"권한 확인"을 물었을 때 오답이 나온 문제를 조사했다. 원인은 데이터·권한 측이 아니었다 — 권한 체계(`jeil-me`/`jeil-hr`/`dept_erp_scope`/`portal_page`)와 인원 데이터(`erp_secure.hr_payroll_m`, 2026-01~06 월별 전사 총원 113·119·101·100·94·98명)는 이미 정상 구현·적재 완료 상태였는데, **챗봇이 이 체계에 연결돼 있지 않았던 것**이 원인이었다. 신규 도구 3종을 배포했다 — ①`get_my_access`: 호출자 본인의 역할·소속부서·열람가능 ERP 모듈·접근가능/불가 페이지·권한요청 방법 반환(호출자 UPN 고정, 타인 조회 불가, 전 직원 사용). ②`get_hr_headcount`: 인원현황 — 전 직원은 월별 전사 총원만, 부서별 인원 분포는 `payroll` 권한자(인사팀·관리자)만. ③`get_hr_payroll`: 급여 집계(인사팀·관리자 전용), `erp_secure` REST 미노출이라 기존 `jeil-hr`와 동일 경로(service_role RPC `hr_payroll_get`). 인사 도구 접근은 허용·거부 모두 `hr_access_log`에 감사 기록(`jeil-hr`와 원장 재사용).
