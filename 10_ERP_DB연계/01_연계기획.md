# 01. ERP DB 연계 — 이후 기획 (통합 정리)

> **기준일: 2026-07-03**(2026-07-08 §4.4 P4 노출 계층, §4.5 사용자-부서 매핑 추가 갱신) · 관리: 최동혁
> ERP 데이터 활용은 **두 축**으로 진행된다: **A) 사내 타부서 데이터 개방**(마스터플랜 v2.0)과 **B) AI 포털·챗봇 연계**(중간DB 연계설계). 본 문서는 두 기획을 하나의 그림으로 통합 정리한 것이다. 상세는 각 원본 문서를 따른다.

---

## 1. 공통 대원칙 (두 축 모두 적용)

| 원칙 | 내용 |
|---|---|
| **READ-ONLY 절대성** | 연동 경로 어디에서도 운영계 쓰기 불가. 사본조차 읽기전용. |
| **부하 격리** | 소비자(부서·포털·챗봇) 트래픽은 운영계에 닿지 않는다 — 야간 사본에서만 조회. |
| **최소 권한** | 필요한 뷰·컬럼·행만. 기본 거부, 명시 허용. 개인별 급여·주민번호·계좌는 **원천 View 단계에서 제외**. |
| **스키마 추상화** | 소비자는 물리 테이블을 모른다 — 뷰(`rpt.v_*`)·집계 테이블이 계약(Contract). |
| **추적성** | 누가·언제·무엇을 조회했는지 전량 감사 로깅. |
| **점진성** | 최소 인프라로 즉시 시작(Crawl) → 수요 증가 시 승격(Walk/Run). |

## 2. 통합 목표 아키텍처

```
┌────────────────────────────────────────────────────────────┐
│ ERP운영DB (UNIERP · MSSQL 2022 · 사외 IDC · 단일노드)       │
│              ✕ 모든 소비자 직접 접근 금지                   │
└──────────────┬─────────────────────────────────────────────┘
               │ 야간 1회 갱신 (백업·복원 / 스냅샷 / ETL)
               ▼
┌────────────────────────────────────────────────────────────┐
│ ① 리포팅 계층 — 격리된 읽기전용 사본 (Tier A→B)             │
│ ② 추상화 계층 — rpt.v_* 뷰 (마스킹·RLS·스키마 계약)         │
└──────┬──────────────────────┬──────────────────────────────┘
       │ [축 A] 사내 타부서    │ [축 B] AI 포털·챗봇
       ▼                      ▼
 ③-A 직접조회(BI/Excel)   야간 ETL → PostgreSQL 중간 DB
 ③-B API(n8n/REST)          (erp_ro / erp_secure / etl_meta)
 ③-C 파일(CSV/Parquet)          │
       │                        ├─▶ 부서 대시보드(pages/) — REST API
       │                        └─▶ 챗봇(jeil-chat) — Tool Use 화이트리스트
       ▼                        ▼
┌────────────────────────────────────────────────────────────┐
│ ④ 거버넌스 — 인증(Entra)·권한(RBAC)·감사·SLA·승인·정기검토  │
└────────────────────────────────────────────────────────────┘
```

**정합 포인트**: 포털 중간 DB의 ETL도 운영계가 아닌 **리포팅 계층(①)/읽기전용 뷰(②)에서 추출**하도록 일원화한다. 원천 접근 창구를 하나로 유지해 운영 무영향 원칙을 지킨다.

## 3. 축 A — 사내 타부서 개방 (마스터플랜 v2.0 요약)

- **리포팅 계층 티어**: **Tier A**(동일 인스턴스 내 리포팅 전용 DB, 즉시·저비용)로 시작 → 소비자·조회량 증가 시 **Tier B**(별도 리포팅 서버, 물리적 완전 격리)로 승격 → 전사 BI 수요 확인 시 Tier C(BI/DW) 별도 과제. Always On은 일 배치 요건에 과잉투자로 배제.
- **전달 채널**: 소비자 유형별 선택 — BI/Excel은 읽기전용 계정 직접조회(③-A), 자체 앱은 API(③-B), 대량·이관은 야간 파일 Export(③-C). 모두 ② 뷰만 바라본다.
- **경영 의사결정 1건**: Tier A 시작 vs 곧바로 Tier B (마스터플랜 §11). 나머지는 실무 진행.

## 4. 축 B — AI 포털·챗봇 연계 (07 연계설계 요약)

### 4.1 중간 DB (PostgreSQL `jeil_portal`)

```
portal       — 사용자·크레딧·대화·감사로그(tool_audit)
erp_ro       — 영업·구매·재고·원가 (일반, 전 직원)
erp_secure   — 자금·인사 집계 (민감, Entra 그룹 인가)
knowledge    — 문서 벡터 (pgvector)
etl_meta     — 배치 실행 로그·동기화 상태
```

### 4.2 주요 매핑 (원천 → 중간 테이블)

| 화면 | UNIERP 원천(읽기만) | 중간 테이블 | 개방 단계 |
|---|---|---|---|
| 영업 수주현황 | `S_BILL_HDR/DTL`, `S_BILL_COLLECTING` | `erp_ro.sales_orders_m` | 1차 |
| 구매 매입집계 | `M_IV_HDR/DTL`, `B_BP` | `erp_ro.purchase_m` | 1차 |
| 재고 입출고 | `ITEM_DOCUMENT`, `M_PUR_GOODS_MVMT` | `erp_ro.inventory_d` | 1차 |
| 품목 마스터 | `B_ITEM` | `erp_ro.item_master` | 1차 |
| **발주현황(협력사 포털용)** | `M_PUR_REQ`(발주요청)·`M_PUR_ORD`(발주) | `erp_ro.purchase_order_*`(신설 예정) | **1차-우선, 범위축소: 2026년도분만**(2026-07-07 요청) |
| 프로젝트 원가 | `PM_PROJECT_MASTER` + 원가 | `erp_ro.project_cost_m` | 2차 |
| 자금일보 | `A_GL`, `F_LN_INFO`, `F_NOTE` | `erp_secure.cash_daily` | 2차+승인 |
| 인사 집계 | `HAA010T`, `HDF070T`(집계만) | `erp_secure.hr_payroll_m` | 2차+승인 |

> **우선순위 결정(2026-07-07)**: 발주 데이터는 정식 1차 4종(품목·영업·구매=매입·재고) 가동 전이라도, **연도를 2026년도로 좁혀 파일럿 우선연동**하기로 했다(협력사 발주포털 실데이터 전환 대비, `CLAUDE.md §4.4`·§9 매핑 재사용). 실행에는 유니포인트 협의(발주요청/발주 테이블 컬럼·연도 필터 기준 확인) 및 `ERP_DB_CONN` 입력이 선행되어야 하며, 실제 ETL 개발·실행은 `10_ERP_DB연계/etl/`(별도 개발 세션) 또는 `erp-db-analyst`(ERP_DB 작업환경) 소관이다. 상세 진행은 `02_진행상태.md` P3 참조.

### 4.3 ETL·챗봇 정책

- **배치**: 매일 03:00(유니포인트 합의), 증분 기준 + 소량 집계는 전체 재적재. 실패 시 Teams/메일 알림 + 1시간 후 재시도 1회. 모든 화면·답변에 「데이터 기준: YYYY-MM-DD 03:00」 표기.
- **챗봇**: AI가 SQL을 생성·실행하지 않는다 — **화이트리스트 조회 템플릿**(`get_sales_monthly` 등 6종 초안)만 실행. 권한은 서버 단 Entra 그룹 검사, 1회 최대 200행, `portal.tool_audit` 전량 로깅.
- **대시보드와 챗봇은 동일 중간 DB·동일 쿼리 로직 공유** — ETL·테이블 1벌 유지.

### 4.4 P4 노출 계층 (2026-07-08 착수)

포털·챗봇이 `erp_ro`/`etl_meta`를 직접 REST로 조회하지 않도록, public 스키마에 **사내 전용 `security_invoker` 뷰 6종**을 신설했다: `v_erp_sales_monthly`·`v_erp_purchase_monthly`·`v_erp_inventory_daily`·`v_erp_item`·`v_erp_pur_order`·`v_erp_data_asof`(job별 최신 성공시각). `erp_ro`·`etl_meta`는 여전히 REST 비노출이며, 기존 `erp_ro` RLS(`internal_select_*`)가 `security_invoker` 뷰에도 그대로 적용돼 사내 인증 사용자만 통과한다. 챗봇(jeil-chat) Edge Function은 service_role 전용 GRANT로 서버 측에서만 조회한다. 정본 SQL: `실제구축준비 자료/이관/sql/08_erp_pub_reporting_views.sql`.

**RLS 격리 실측 테스트(2026-07-08)**: JWT 시뮬레이션 결과 — 사내(internal): 정상 조회(매출10·매입63·재고656·품목58,246·발주2,699·기준시각5행), 협력사(vendor): 전부 0행, anon: 미부여로 차단.

이 위에서 실제 연결도 시작됐다: `app/lib/api.js`의 `erpApi`로 `pages/구매_거래처별매입집계_2026.html` 1개 화면을 파일럿 실연동했고, 챗봇에 ERP Tool 4종(`get_erp_sales_monthly`·`get_erp_purchase_monthly`·`get_erp_inventory_status`·`get_erp_item`)을 배포했다(version 7 ACTIVE). 유니포인트 협의(P1)·읽기전용 계정 전환·야간배치 자동화·Entra→Supabase 세션 브리지는 여전히 미해결이다. 상세 진행은 `02_진행상태.md` P4 참조.

### 4.5 사용자-부서-사원 매핑·대사 (2026-07-08 신설)

ERP `Z_USR_MAST_REC`(사용자 마스터)의 `usr_id`(=MS 이메일, SSOT)와 `usr_nm`(="부서명_이름")을 대사해, **이메일 기준으로 사용자↔부서↔사원 관계**를 중간DB(`erp_ro`)에서 관리하는 계층을 추가했다. 이는 축 B(포털·챗봇 연계)의 부속 계층으로, 향후 부서 기준 화면 분류·챗봇 응답의 소속 확인 등에 활용할 수 있는 **참고 데이터**다(실제 권한 인가는 여전히 Entra 보안그룹 + RLS가 담당, `CLAUDE.md §5`).

- 대상: `USE_YN='Y' AND USR_ID LIKE '%@%'` 99건. `usr_nm` 파싱(부서명_이름) → `B_ACCT_DEPT`와 대사해 29개 부서명 중 25개(95건) 일치, 불일치 4개(해외법인·외부회계·재무 세분화·admin 테스트계정)는 처리방침 결정 대기.
- 노출 뷰(사내 전용, security_invoker): `public.v_erp_user_dept`(정상 매핑)·`v_erp_user_dept_recon`(대사 불일치 리포트)·`v_erp_dept_roster`(부서별 명부).
- 스키마·뷰·ETL job(`usr_master`·`dept_master`)은 구축·적용 완료됐으나 **실적재는 미실행**(0행, 관리자 `!` 직접 실행 대기).
- 상세는 전용 관리 문서 [`05_사용자부서_매핑대사.md`](05_사용자부서_매핑대사.md) 참조. 정본 SQL: `실제구축준비 자료/이관/sql/09_erp_user_dept_mapping.sql`.

## 5. 단계별 로드맵 (통합)

| 단계 | 내용 | 선행 조건 |
|---|---|---|
| **P0** 정책 수립 | 접근정책안·마스터플랜·본 관제 체계 | (완료·승인 대기) |
| **P1** 유니포인트 협의 | 읽기전용 계정(`jeilax_ro`)·뷰·방화벽/VPN 합의 | 경영 티어 결정 |

| **P4** 포털·챗봇 연결 | `pages/` 실데이터 API 전환 + 챗봇 Tool 3종 + `tool_audit` — **2026-07-08 착수·부분완료**(§4.4: 노출 뷰 6종·RLS 격리 테스트·파일럿 화면 1건·챗봇 Tool 4종 / §4.5: 사용자-부서 매핑 계층 구축, 실적재 대기) | P3 |
| **P5** 민감 영역 | `erp_secure`(자금·인사) — 경영 승인 + Entra 그룹 | P4 + 승인 |

## 6. 하지 말아야 할 구조 (금지 목록)

| 방식 | 문제 |
|---|---|
| 포털/챗봇/BI → ERP운영DB 직접 연결 | 성능·장애 전파, AI 임의 SQL 위험 |
| MSSQL 전체 복제 | 용량·민감컬럼 과다 |
| 실시간 ERP 조회(초기) | 복잡도↑ — 일 배치로 대부분 충족 |
| AI `SELECT *` 생성 허용 | 인젝션·대량 유출 |
| OneDrive CSV만으로 ERP 대체 | 권한·갱신·감사 불가 |
| Always On 실시간 복제(현 단계) | Enterprise 라이선스·2차 서버 비용 대비 불필요 |
