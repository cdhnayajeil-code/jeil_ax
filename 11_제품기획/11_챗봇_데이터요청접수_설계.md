# 11 · 챗봇 데이터 요청접수 설계 — "조회 불가 → 요청 버튼 → 접수·처리"

> 관리자: 최동혁(dh.choi@jeilm.co.kr) · 기준일: 2026-07-22 · 상태: **Accepted — P1(권한 유형) 구현·배포 완료 / P2~P3 잔여 (ADR-011)**
> 관련: [10 응답렌더링](10_챗봇_응답렌더링_설계.md) · [05 FE·BE](05_프론트엔드_백엔드_설계.md) · [06 ERP연계](06_ERP연계_챗봇활용_설계.md) · [04 DB설계](04_데이터베이스_설계.md) · [08 보안](08_보안_데이터안정성.md) · [09 ADR](09_ADR_의사결정기록.md) · `CLAUDE.md §1.6·§5.4·§7·§13`

챗봇이 **"그 데이터는 못 보여드립니다"** 로 끝나는 지점에서, 사용자가 그 자리에서 **요청 버튼**을 눌러 접수하고, 관리자가 **요청함**에서 처리·회신하는 흐름을 설계한다.
결론부터: **새 뷰·새 화면을 만들지 않고**, 이미 있는 `notice` 뷰의 `request` 정보를 실제 접수 액션으로 승격(`actions.kind:"request"` 1종 추가)하고 **요청 원장 테이블 1개 + Edge Function 1개 + 관리자 콘솔 탭 1개**만 신설한다. 근거는 §1~§3, 설계는 §4~§8, 단계·미결정은 §9~§11.

## 1. 현재 상태 (실측, 2026-07-22)

### 1-1. "조회 불가"는 한 종류가 아니다 — 5유형, 처리 주체가 다 다르다

`supabase/functions/jeil-chat/index.ts` 실측 기준.

| # | 유형 | 발생 지점 | 현재 응답 | 카드(`__view`) | 실제 해결 주체 |
|---|---|---|---|---|---|
| A | **부서 ERP 모듈 권한 없음** | `index.ts:448~459` (게이트 1곳, 전 ERP 도구 공통) | "관리자에게 …권한을 요청해 주세요" 안내문 | ✅ `notice/deny` + `request` 칩 + `ask` 액션 | 포털 관리자(권한 부여) |
| B | **급여·인사 민감 데이터 제한** | `index.ts:987~995` | "인사팀(또는 관리자)만 열람" | ✅ `notice/deny` | 포털 관리자 + 인사팀 합의 |
| C | **중간DB 적재범위 밖(데이터 없음)** | `index.ts:764`, `1005` 등 | "2025년 이전 건은 미적재이니 원본 ERP를 확인하세요" | ❌ **없음(평문)** | ERP 연계 담당(ETL 범위 확대·유니포인트 협의) |
| D | **문서 연동 승인범위 밖** | `index.ts:1051`, `1093`, `1099` | "AI 연동 승인 범위에 없어 열람 불가" | 부분(`1051`만 `notice/info`) | AI 문서연동 담당(`ai_document_scope` 등재) |
| E | **도구 자체가 없음(미구현 영역)** | 시스템 프롬프트 `index.ts:57` | "추측하지 말고 원본 확인을 권하세요" | ❌ 없음 | 제품기획(백로그 → 신규 Tool) |

즉 사용자 눈에는 똑같이 "안 나온다"지만, **A·B는 권한 문제, C는 데이터 파이프라인 문제, D는 스코프 등록 문제, E는 기능 부재**다. 요청 기능의 핵심 가치는 **버튼을 만드는 것이 아니라 이 5유형을 자동 분류해 올바른 담당에게 꽂는 것**이다.

### 1-2. 프론트에는 이미 "요청 대상" 표시가 있다 — 버튼만 없다

`04_챗봇_포털_데모UI.html:1788~1799`

```js
} else if(v.view === 'notice'){
  …
  if(v.request){
    var rq = _cvEl('div', 'cv-req');
    rq.appendChild(_cvEl('span','cv-chip','요청 대상: ' + v.request.moduleKo + '(' + v.request.module + ')'));
    if(v.request.dept) rq.appendChild(_cvEl('span','cv-chip','소속: ' + v.request.dept));
    card.appendChild(rq);
  }
}
```

서버는 이미 `request:{module, moduleKo, dept}` 를 내려주고 있고 프론트는 그걸 **칩으로 보여주기만** 한다. **접수 경로가 없어서 정보가 화면에서 죽는다.**

### 1-3. 지금 사용자가 하는 행동 = 채널 이탈

현재 제공되는 최선의 액션은 `kind:"ask"` — **"권한 요청 초안 작성"**(`index.ts:457`, `994`)이다. 즉 챗봇이 메신저용 문구를 써주면 사용자가 **복사해서 Teams/구두로 직접** 보낸다. 결과:

- 요청이 **시스템 밖**에 존재 → 유실·중복·누락, 처리 이력 없음.
- 관리자는 **수요를 알 수 없음** → 어느 부서가 어떤 모듈을 반복 요구하는지 데이터가 없다.
- 요청자는 진행상태를 알 수 없어 **재요청**(관리자 피로).

### 1-4. 수요 계측이 아예 없다

`chat_log` 컬럼은 `upn·model·messages_count·prompt_chars·prompt_tokens·completion_tokens·est_cost_usd·tools_used·session_id·stopped` 뿐 — **거부(deny)·무데이터(no_data) 플래그가 없다.** 그래서 "조회 불가가 하루 몇 건 나는가"를 **지금은 답할 수 없다.** 요청 원장이 생기면 이 수치가 자동으로 쌓인다(§8-2).

### 1-5. 받아줄 뒷단은 이미 있다

| 자산 | 상태 | 요청 처리에서의 역할 |
|---|---|---|
| `perm_effective(upn)` RPC | ✅ 판정 SSOT(ADR-010) | 접수 시 **서버 재판정**으로 요청 정당성 검증 |
| `perm_grant` / `dept_erp_scope` | ✅ 운영 중 | 승인 시 권한 반영 대상 |
| `perm_audit` | ✅ 운영 중 | 권한 변경 감사 |
| 관리자 콘솔(`04` 탭 구조) | ✅ 운영 중 | 「요청함」 탭 추가 지점 |
| 표준 그리드 `app/lib/grid.js` | ✅ 표준(CLAUDE.md §13) | 요청함 목록 UI |
| `chat_*` 테이블 패턴(RLS 전면차단 + Edge Fn 전용) | ✅ 확립 | 요청 원장의 보안 모델 그대로 계승 |

**신설이 필요한 것은 테이블 1개·Edge Function 1개·관리자 탭 1개뿐이다.**

## 2. 문제 정의 · 목표

**문제**: 챗봇이 조회 불가를 알리는 순간 사용자 여정이 끊긴다. 해결 요청은 시스템 밖(구두·메신저)으로 나가 유실되고, 관리자는 수요를 계측하지 못하며, 권한·적재범위 정책이 데이터 없이 결정된다.

**목표(성공 정의)**

| 지표 | 현재 | 목표(도입 3개월) |
|---|---|---|
| 조회 불가 → 시스템 내 접수 전환율 | 0%(측정 불가) | 60% 이상 |
| 요청 평균 처리 리드타임 | 측정 불가 | 권한 유형 **1영업일** 내 1차 응답 |
| 동일 요청 중복 접수 | 통제 없음 | 자동 병합으로 중복 건 10% 미만 |
| 권한 정책 개선 근거 | 없음 | 부서×모듈 요청 히트맵 월 1회 리뷰 |

**비목표(이번 범위 아님)**

- 전자결재 연동(그룹웨어 상신) — 요청은 **포털 내부 티켓**이며 결재문서가 아니다.
- 자동 권한 부여 — 접수·라우팅까지만. **부여 클릭은 사람**(CLAUDE.md §1.6, §5.4).
- 요청자 임의 데이터 업로드 — 데이터 요청은 "적재 요청"이지 사용자 데이터 투입이 아니다.

## 3. 설계 원칙 (이 문서의 판단 기준)

1. **뷰 5종 고정 유지(ADR-008).** `series/ranking/record/list/notice` 외 신규 뷰 신설 금지 → 요청 버튼은 **`actions`의 새 kind 1종**으로 흡수한다.
2. **요청 컨텍스트는 서버가 만든다.** 프론트가 "모듈=payroll" 같은 값을 조립해 보내지 않는다. 프론트는 서버가 준 `request` 객체를 **그대로 반송**하고, 서버는 접수 시 **`perm_effective`로 재판정**해 정당성을 확인한다(위조 요청 차단, 서명키 불필요).
3. **접수는 실거래 액션이 아니다.** 티켓 생성까지만 → CLAUDE.md §1.6 저촉 없음. 권한 부여·ETL 확대 실행은 관리자 화면의 별도 클릭.
4. **개인정보 최소화.** 사유 입력란은 자유 텍스트지만 **급여액·주민번호·평가 등 민감값 기재 금지** 안내 + 저장 전 패턴 마스킹. 성명은 마스킹 제외(§1.7).
5. **원장은 하나.** 5유형 모두 같은 테이블에 담고 `kind`로 구분한다. 유형별 테이블 분리 금지(관리 화면 중복 방지).
6. **데이터 접근 단일 진입점(ADR-004).** 프론트는 `app/lib/api.js` 경유, DB 접근은 Edge Function(service_role)만.

## 4. 기능 설계

### 4-1. 전체 흐름

```
[사용자] 질문
   └─ jeil-chat 도구 실행 → 조회 불가 판정(A~E)
        └─ notice 카드 + actions:[{kind:"request", …}]        ← 서버가 컨텍스트 동봉
[사용자] "요청하기" 클릭 → 사유·긴급도 입력 모달 → 전송
        └─ jeil-portal-request (Edge Fn)
             ├─ 본인 확인(JWT) + perm_effective 재판정(정당성)
             ├─ 중복 open 건 탐지 → 있으면 '동조(+1)' 처리
             └─ portal_request INSERT (open)
[관리자] 콘솔 › 요청함 탭 (배지 n건)
        ├─ 상세 확인 → 상태 전이(ack/doing/done/rejected/duplicate)
        ├─ [권한 부여 화면으로] 프리필 이동  ← 실제 부여는 관리자 클릭
        └─ 처리메모 작성 → 요청자에게 회신
[요청자] 챗봇에 "내 요청 진행상황" → 조회(get_my_requests) 또는 포털 내 배지
```

### 4-2. 유형별 라우팅·처리 규칙

| kind | 트리거(§1-1) | 담당 | 처리 액션 | 비고 |
|---|---|---|---|---|
| `perm` | A | 포털 관리자 | `dept_erp_scope`(부서축) 또는 `perm_grant`(개인 예외) 편집 | 만료일 지정 권장(대행·프로젝트성) |
| `perm_sensitive` | B(급여·인사) | 포털 관리자 **+ 인사팀 확인** | 동일하되 **2인 확인** 필수 | 자동 승인 UI 제공 금지 |
| `data` | C | ERP 연계 담당(`10_ERP_DB연계`) | ETL 대상·기간 확대 검토 → 필요 시 **유니포인트 협의**(CLAUDE.md §4.6) | 협의 필요 플래그 별도 |
| `doc` | D | AI 문서연동 담당 | `ai_document_scope` 등재 검토 | 민감 컨테이너(급여·자금) 등재 금지 |
| `feature` | E | 제품기획 | 백로그 등록 → 신규 Tool/화면 | `02_SRS` FR-ID로 승격 |
| `quality` | (사용자 신고) | 데이터 소유 부서 | 값 오류·정합성 확인 | "숫자가 이상해요" 경로, §9 P3 |

### 4-3. 상태머신

```
        ┌──────────── cancel(요청자, open/ack 한정) ────────────┐
        ▼                                                       │
   [open] ──ack──► [ack] ──start──► [doing] ──complete──► [done]
        │              │                │
        └──reject──────┴────────────────┴──► [rejected]   (사유 필수)
        └──dup─────────────────────────────► [duplicate]  (원본 id 링크)
```

- `open`: 접수 직후. 요청자에게 접수번호 즉시 표시.
- `ack`: 관리자 확인(1차 응답 SLA 지점).
- `done`/`rejected`: 처리메모 필수 → 요청자 회신.
- `duplicate`: 자동 병합 시에도 사용(§4-5).

### 4-4. UI — 챗봇 카드 액션 확장(신규 뷰 없음)

`10_챗봇_응답렌더링_설계.md` P3 액션 규약에 **`kind:"request"` 1종 추가**:

```ts
// 서버(jeil-chat) — 기존 A유형 게이트(index.ts:452~459) 확장안
return {
  접근제한: true, 요청안내: true, 모듈: erpMod, 부서: scope.dept, 안내,
  __view: {
    view: "notice", title: "데이터 접근 권한 안내", kind: "deny", text: 안내,
    request: { module: erpMod, moduleKo: modKo, dept },
    actions: [
      { kind: "request", label: "권한 요청하기",          // ★신규
        req: { kind: "perm", module: erpMod, dept,
               tool: name, args_digest: "…",             // 어떤 질문에서 막혔는지(값 아님, 인자 요약)
               reason_hint: `${modKo} 데이터 열람 필요` } },
      { kind: "ask", label: "요청 문구 초안 작성", prompt: "…" }   // 기존 유지
    ]
  }
}
```

프론트(`04` 렌더러)는 `kind:"request"` 를 만나면 버튼을 그리고, 클릭 시 **사유(필수, 200자)·긴급도(보통/급함)** 모달을 띄운 뒤 `req` 객체를 **그대로** `jeil-portal-request` 로 POST한다. 접수 성공 시 버튼은 `✅ 접수됨 #REQ-0042` 로 치환(중복 클릭 차단).

**C·E 유형은 `__view` 자체가 없다**(§1-1) → 이번 작업에서 `notice/info` + `request` 액션을 신설해야 한다. 이게 프론트보다 **서버 작업량이 큰 이유**다.

### 4-5. 중복 병합

동일 `(upn, kind, module, dept)` 로 **open/ack 상태 건이 이미 있으면** 새 행을 만들지 않고 기존 건에 `+1`(동조 카운트)·최근 사유 append. 다른 사람의 동일 요청이면 `supporters` 배열에 추가 → 관리자는 **"7명이 요청 중"** 을 보고 부서축 일괄 부여를 판단할 수 있다. 이게 개인 예외(`perm_grant`) 남발을 막는 실질 장치다.

### 4-6. 남용 방지

- 사용자당 **1일 5건** 상한(초과 시 안내 반환).
- 사유 **15자 미만**이면 접수 거부(품질 하한). 프론트·서버 양쪽에서 검증.
- 이미 권한이 있는데 요청하면(재판정 결과 allow) → 접수 대신 **"이미 권한이 있습니다. 새로고침 후 재시도"** 안내.

## 5. 데이터 모델 (신설 1테이블)

```sql
-- public.portal_request — 포털 요청 원장(챗봇 조회불가 → 사용자 요청 접수)
-- RLS 전면차단 + Edge Function(service_role) 전용 — chat_* 패턴 계승
create table public.portal_request (
  id            bigserial primary key,
  req_no        text unique not null,               -- 표시용 'REQ-2026-0042'
  kind          text not null check (kind in ('perm','perm_sensitive','data','doc','feature','quality')),
  status        text not null default 'open'
                check (status in ('open','ack','doing','done','rejected','duplicate','cancelled')),
  requester_upn text not null,
  requester_dept text,
  target_module text,                               -- perm: erp_module / data: 테이블·기간 / doc: drive 경로
  target_detail jsonb not null default '{}',        -- tool·args_digest·요청 상세(개인정보 미포함)
  reason        text not null,                      -- 사용자 사유(마스킹 후 저장)
  urgency       text not null default 'normal' check (urgency in ('normal','high')),
  supporters    text[] not null default '{}',       -- 동조자 upn (중복 병합)
  dup_of        bigint references public.portal_request(id),
  assignee_upn  text,                               -- 처리 담당(라우팅 결과)
  handled_note  text,                               -- 처리·반려 사유(요청자 회신 문구)
  needs_vendor_review boolean not null default false, -- data 유형: 유니포인트 협의 필요
  created_at    timestamptz not null default now(),
  acked_at      timestamptz,
  closed_at     timestamptz
);
create index on public.portal_request (status, kind, created_at desc);
create index on public.portal_request (requester_upn, created_at desc);
alter table public.portal_request enable row level security;   -- 정책 미부여 = 전면 차단
comment on table public.portal_request is
  '포털 요청 원장 — 챗봇 조회불가(권한·데이터·문서·기능)에서 접수. RLS 전면차단, Edge Fn(service_role) 전용.';
```

**설계 근거**

- `target_detail`은 **jsonb** — 유형마다 필요한 필드가 달라 컬럼화하면 5유형×N 컬럼으로 번진다. 조회 화면은 유형별 렌더만 분기.
- `args_digest`는 **원본 인자값이 아니라 요약**(예: `ym=2024-05`) — 질문 원문·민감값 저장 금지.
- 상태 전이 이력은 별도 테이블을 만들지 않고 **`perm_audit` 패턴 재사용 여부를 P2에서 판단**(§11 미결정 3).

## 6. 백엔드 (Edge Function 1종 신설)

`supabase/functions/jeil-portal-request/index.ts`

| 액션 | 메서드 | 권한 | 동작 |
|---|---|---|---|
| `create` | POST | 로그인 사용자 | 본인 JWT 확인 → **`perm_effective` 재판정** → 중복 병합 검사 → INSERT/동조 |
| `mine` | GET | 로그인 사용자 | 본인이 낸 요청 목록·상태 |
| `cancel` | POST | 요청자 본인 | `open/ack` 한정 취소 |
| `list` | GET | **관리자** | 요청함(필터: 상태·유형·부서) |
| `transition` | POST | **관리자** | 상태 전이 + `handled_note` 기록 |

**정당성 재판정이 핵심이다.** 프론트가 `{kind:'perm', module:'payroll'}` 를 임의로 보내도, 서버는 `perm_effective(upn)` 로 **정말 그 모듈이 없는지** 확인하고 없을 때만 접수한다. 이미 있으면 접수하지 않는다 → 위조 요청·스팸이 원천 차단되고 별도 서명키가 필요 없다.

챗봇 조회용 도구 **`get_my_requests`** 를 `jeil-chat` 에 1개 추가하면 "내 요청 어떻게 됐어?"를 대화로 답할 수 있다(뷰: `list` 재사용).

- **목록**: 표준 그리드(`app/lib/grid.js`, CLAUDE.md §13). 컬럼 = 접수번호·유형·요청자(부서_이름_아이디 규약)·대상·동조수·긴급도·경과일·상태.

- **수요 통계**: 부서×모듈 요청 히트맵(월간) — 개인 예외 남발 대신 부서축 정비 판단 근거.

## 8. 보안·감사·계측

### 8-1. 위협과 대응

| 위협 | 대응 |
|---|---|
| 프론트에서 요청 페이로드 위조(타 부서·타인 명의) | 요청자는 **JWT에서만** 취득(본문 무시), 대상은 **서버 재판정**(§6) |
| 사유란에 개인정보·급여액 기입 | 입력 안내 + 저장 전 패턴 마스킹(주민번호·계좌·금액 형식). **성명은 마스킹 제외**(CLAUDE.md §1.7) |
| 스팸·항의성 대량 접수 | 1일 5건 상한 + 사유 품질 하한 + 중복 자동 병합 |
| 요청 원장 직접 조회(REST) | RLS 전면 차단, Edge Fn service_role 전용 |
| 반려 사유에 민감 정보 노출 | `handled_note`는 요청자에게 그대로 노출됨 → 관리자 입력 가이드 명시 |
| 요청 = 권한 자동 부여로 오인 | UI 문구를 "요청 접수"로 고정, 부여는 별도 화면 |

### 8-2. 계측(부수 효과가 아니라 목적)

요청 원장이 쌓이면 **지금 답할 수 없는 질문**(§1-4)에 답할 수 있다.

- 부서별 어떤 ERP 모듈이 반복 요청되는가 → **부서축 권한 기본값 재설계** 근거.
- `data` 유형 요청의 기간 분포 → **ETL 적재범위 확대 우선순위**(`10_ERP_DB연계/04` 증분동기화 기획과 직결).
- `feature` 요청 빈도 → **신규 챗봇 Tool 백로그 우선순위**(`02_SRS`).

추가로 `chat_log`에 `denied` boolean 1컬럼을 두면 **"막혔지만 요청하지 않은"** 이탈률까지 잡힌다(전환율 지표의 분모). 권장.

## 11. 미결정 사항 (관리자 결정 필요)

1. ~~**알림 채널**~~ → **결정: Teams**(2026-07-22, 관리자). 단 **Graph 앱 권한(ChannelMessage.Send) 방식은 쓰지 않는다** — Entra 앱 등록 변경이 선행돼야 하고 권한 범위가 넓다. 대신 **Teams 채널 워크플로 웹훅**(Power Automate) URL을 Supabase 시크릿 `TEAMS_WEBHOOK_URL`로 등록하는 방식을 채택했다. 코드는 배포 완료 상태이며, 시크릿이 없으면 알림만 조용히 건너뛴다(접수는 정상). 설정 절차는 §13.
2. **승인 라인** — 권한 요청에 **부서장 확인**을 넣을지(2단 승인) 아니면 포털 관리자 단독 판단으로 갈지. 현재 `perm_effective`에 `dept_admin_of` 축이 이미 있어 2단 확장은 가능.
3. **상태 이력 테이블 분리 여부** — `portal_request` 컬럼 갱신만으로 갈지, 전이 이력을 별도 로그로 남길지(감사 요구 수준에 의존).
4. **`data` 유형의 유니포인트 협의 게이트** — ETL 확대 요청이 벤더 협의를 요하면(§4-2) 요청 상태를 `doing`에 묶어둘지 별도 `blocked` 상태를 둘지.
5. **SLA 수치 확정** — 1차 응답 1영업일 / 완료 5영업일을 기본안으로 제안. 관리자 확정 필요.
6. **요청 보존기간** — `done/rejected` 건의 보존(제안: 2년) 및 개인정보 관점 파기 정책.

## 13. Teams 알림 설정 절차 (관리자 작업)

Entra 앱 등록은 **건드리지 않는다**. 채널 워크플로 웹훅 URL 하나를 서버 시크릿에 넣는 것으로 끝난다.

**① Teams에서 웹훅 URL 발급**

1. Teams → 알림을 받을 **채널** 선택 → 채널 헤더 우측 **⋯ → 워크플로**
2. 템플릿 검색창에 `웹후크`(또는 `webhook`) → **「웹후크 요청이 수신되면 채널에 게시」** 선택
3. 연결 계정 확인 → **다음** → 팀·채널 확인 → **워크플로 추가**
4. 생성 완료 화면에 나오는 **HTTP POST URL을 복사**(`https://…logic.azure.com/…` 형태)

> 이 URL은 **비밀값**이다 — 아는 사람은 누구나 해당 채널에 글을 쓸 수 있다. 문서·소스·메신저에 남기지 말고 시크릿에만 보관한다(CLAUDE.md §1.1).

**② Supabase 시크릿 등록**

Supabase 대시보드 → 프로젝트 → **Edge Functions → Secrets** → 추가:

| 키 | 값 |
|---|---|
| `TEAMS_WEBHOOK_URL` | ①에서 복사한 URL |

저장 즉시 반영된다(함수 재배포 불필요). 미등록 상태에서는 알림만 건너뛰고 접수는 정상 동작한다.

**③ 확인**

챗봇에서 권한 없는 데이터를 조회 → 카드의 **「열람 권한 요청하기」** → 사유 입력 후 접수 → 지정 채널에 `🔔 새 포털 요청 REQ-…` 카드가 도착하면 정상.

> 구 O365 커넥터(`…webhook.office.com`) URL을 쓰는 경우에도 동작한다 — 서버가 URL 형식을 보고 MessageCard/Adaptive Card 페이로드를 자동 분기한다. 다만 커넥터는 Microsoft가 단계적으로 폐지 중이라 **워크플로 방식을 권장**한다.
