04 · 데이터베이스 설계서

관리자: 최동혁(dh.choi@jeilm.co.kr) · 기준일: 2026-07-08 · 상태: Draft 관련: 03 아키텍처 · 06 ERP연계 · 08 보안 실제 DDL/RLS SQL 정본(값 복제 금지): 이관/sql/01~07

이 문서는 데이터 모델의 구조·격리·명명·매핑을 정의한다. 표의 행수는 2026-07-08 Supabase 라이브 조회(list_tables) 실측이다.

1. 스키마 3분할

스키마 용도 접근 REST 노출 상태
public 포털 자체 생성(발주·검사·계정·챗봇로그) 클라이언트 RLS + Edge Fn O(RLS 보호) 실가동
erp_ro ERP 읽기 사본(집계·스냅샷) 사내 SELECT(설계)·적재는 RPC X(비노출) 파일럿
etl_meta 배치 이력·메타 service_role 전용 X(비노출) 파일럿
erp_secure 민감(자금·인사) — X P5 예약·미생성
  • 접미사 규약: _m(월집계) · _d(일집계) · _s(스냅샷). 공통 컬럼: synced_at·src_updated·batch_id(UUID).
  • erp_ro/etl_meta를 REST에 노출하지 않고, 적재는 security definer RPC(erp_etl_upsert/erp_etl_batch)로만. → 클라이언트가 ERP 사본을 직접 못 읽음(P4 노출은 별도 뷰/RPC 설계 필요, 06).

3. ERD (핵심 관계)

                        vendor_master(bp_cd) ──1:N── vendor_account
                              │                          │
                              │ bp_cd                    │ (auth.users ← app_metadata.vendor_bp)
                              ▼                          ▼
   sp_order_header(po_no,bp_cd) ──1:1── sp_order_state(po_no, status, step)
        │po_no                              
        ├──1:N── sp_photo(po_no)            
        ├──1:N── sp_message(po_no, sender)  
        ├──1:N── sp_insp_request(po_no)     
        └──1:N── sp_inspection(po_no) ──1:N── sp_inspection_log(po_no, 차수, 판정자)

   [ERP 사본 · 조인축 없음(현재 단절)]
   erp_ro.pur_order_s(po_no?, bp_cd) …… 향후 sp_order_header.po_no 와 매핑(P4)
   erp_ro.item_master_s / sales_orders_m / purchase_m / inventory_d
   etl_meta.batch_run ──(job별 최신)──▶ v_last_success
  • 포털 격리축: bp_cd(거래처). 조인축: po_no(발주번호, PO+YYYYMMDD+일련).
  • ERP 사본(erp_ro)과 포털(public)은 현재 FK로 연결돼 있지 않다. 매핑은 06에서 설계.

4. RLS 정책 매트릭스

전 테이블 RLS ON. 두 헬퍼가 정책 전제:

  • public.is_internal() = JWT app_metadata.role == 'internal'
  • public.vendor_bp() = JWT app_metadata.vendor_bp(JSON 배열) → text[]
  • Custom Access Token Hook(public.custom_access_token_hook)이 로그인 JWT에 role·vendor_bp 주입. @jeilm.co.kr은 미지정 시 internal 자동. supabase_auth_admin만 실행, authenticated/anon revoke.
테이블군 사내(internal) 협력사(vendor) anon
sp_order_*, sp_photo, sp_message, sp_insp_request 전체(for all) 자기 bp_cd 행만(bp_cd = any(vendor_bp())) 차단
sp_inspection, sp_inspection_log 읽기+쓰기 읽기만 차단
vendor_master/account/application 관리(is_vendor_admin) 제한적 자기 차단
chat_log (정책 없음 — Edge Fn service_role만) 차단 차단
chat_work·chat_session·chat_message (정책 없음 — Edge Fn의 upn 필터가 유일한 경로. 본인 것만, ADR-009) 차단 차단
etl_meta.batch_run (정책 없음 — service_role만) 차단 차단
Storage vendor-photos 경로 첫 폴더 {bp_cd} ∈ vendor_bp()일 때 협력사 접근 비공개
  • chat_log·batch_run의 "RLS ON + 정책 없음"은 의도된 설계(클라이언트 전면 차단, Edge Fn/RPC만 접근). 보안 advisor가 INFO로 표기하나 취약점이 아님 → 08 보안 §실측.
  • is_vendor_admin() = is_internal() AND portal_admin 등록. set search_path=''로 하드닝.

5. RPC / 뷰

객체 종류 접근 용도
erp_etl_upsert(table, rows) RPC(security definer) service_role ETL 적재(청크 500)
erp_etl_batch(...) RPC(security definer) service_role 배치 start/finish 기록
update_my_vendor_contact(p_name,p_phone) RPC(security definer) authenticated 본인 담당자명·연락처 수정
custom_access_token_hook Auth Hook supabase_auth_admin JWT claim 주입
chat_append_message(...) RPC(security definer) service_role 전용(public/anon/authenticated revoke) 챗봇 메시지 원자 append(seq 채번·제목 자동·upn 검증)
etl_meta.v_last_success 뷰 (내부) job별 최신 성공시각

점검 포인트: update_my_vendor_contact가 anon/authenticated에 REST로 노출됨(보안 advisor WARN). 세션 없는 호출은 무효여야 하며, 자기 행만 수정하도록 내부 가드를 재확인할 것 → 08 §실측 결과, 09 ADR.
