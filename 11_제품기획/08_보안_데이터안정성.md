## 1. 보안 경계 4원칙 (현행 준수 사실)

| 원칙 | 구현 | 검증 |
|---|---|---|
| 비밀값 분리 | service_role·OpenAI·Entra Secret은 서버(Deno.env/Secrets)에만. 프론트는 publishable anon만. ERP 비번은 `.env` 평문 금지→DPAPI(`%USERPROFILE%\.erp\`) 또는 서버 환경변수 | `.env`·민감폴더 `.gitignore` |
| 읽기전용(ERP) | ETL SELECT+NOLOCK+파라미터 바인딩, 화이트리스트 SQL 5종. 챗봇도 AI SQL 금지, 등록 Tool만 | `etl_run.py`, `jeil-chat` |
| RLS 격리 | public 전 테이블 RLS ON, JWT claim 기반 사내/거래처 격리, Storage 경로 RLS | 라이브 실측(§3) |
| 스키마 미노출 | `erp_ro`/`etl_meta` REST 비노출, 적재는 security-definer RPC. `chat_log`·`batch_run`은 정책 없이 클라이언트 전면 차단 | 라이브 advisor(§3) |

## 2. 위협모델 (요약)

| 위협 | 표면 | 대응 |
|---|---|---|
| 협력사가 타사 발주 조회 | REST/Realtime | RLS `vendor_own`(`bp_cd=any(vendor_bp())`) |
| anon 무단 접근 | 공개 anon 키 | RLS 전면 + 민감 테이블 정책 부재(차단) |
| 특권 함수 오남용 | RPC/Edge Fn | service_role 서버 한정, 함수 내부 인증 검증 |
| ERP 운영 침해 | ETL 연결 | 읽기전용·NOLOCK·화이트리스트·별칭표기 |
| 토큰 탈취(XSS) | localStorage 토큰 | (개선 예정) BFF/HttpOnly, 입력검증·출력인코딩 |
| 업로드 악성파일 | 협력사 사진 | 형식·용량·확장자 검증(강화 필요) + 비공개 버킷·서명 URL |
| 개인정보 노출 | 화면·로그·챗봇 | 그룹 인가, 챗봇 프롬프트 가드, 최소 컬럼(단 이름은 마스킹 제외 §1.7) |

## 3. 라이브 실측 결과 (2026-07-08, Supabase MCP · 읽기전용)

### 3.1 RLS 적용 실검증(`list_tables`)

- **public 13 · erp_ro 6 · etl_meta 1 = 전 테이블 RLS ON 확인.** (표본 행수 → [04 §2](04_데이터베이스_설계.md))
- 결론: 스키마 3분할·RLS 전면 적용이 설계대로 **실제로 켜져 있음**을 확인.

### 3.2 보안 Advisor(`get_advisors security`)

| 수준 | 항목 | 대상 | 판정·조치 |
|---|---|---|---|
| WARN | SECURITY DEFINER 함수를 anon/authenticated가 REST로 실행 가능 | `public.update_my_vendor_contact` | **점검 필요**: 세션 없는 호출 무효화·자기 행만 수정하는 내부 가드 재확인. 불필요 시 EXECUTE 회수 또는 SECURITY INVOKER 전환 검토 |
| WARN | 유출 비밀번호 보호 비활성(HaveIBeenPwned) | Supabase Auth | **권고**: 협력사 계정 보안 강화 위해 활성화 검토(관리자) |
| INFO | RLS enabled·정책 없음 | `etl_meta.batch_run` | **의도됨**: service_role만 접근(비노출). 조치 불요 |
| INFO | RLS enabled·정책 없음 | `public.chat_log` | **의도됨**: Edge Fn만 기록·조회. 조치 불요 |

참고 문서: [Supabase Database Linter](https://supabase.com/docs/guides/database/database-linter).

### 3.3 성능 Advisor(`get_advisors performance`, 총 136건)

| 항목 | 건수 | 성격 |
|---|---|---|
| `multiple_permissive_policies` | 125 | 여러 permissive 정책 중복(사내 `internal_all`+`vendor_own`) — 쿼리마다 복수 정책 평가. **보안결함 아님**, 최적화 여지 |
| `auth_rls_initplan` | 8 | RLS가 행마다 `auth.<fn>()`/`current_setting()` 재평가. `(select auth.uid())` 래핑으로 최적화 권장 |
| `unused_index` | 2 | 미사용 인덱스(예: `idx_sp_message_bp_created`) |
| `unindexed_foreign_keys` | 1 | `vendor_application_matched_bp_cd_fkey` 커버링 인덱스 없음 |

> 성능 항목은 **취약점이 아니라 튜닝 대상**이다. 데이터가 커지기 전(현재 소량) 정리하면 이관·확장 시 유리. RLS initplan 최적화는 [09 ADR](09_ADR_의사결정기록.md) 개선 항목.

### 3.4 마이그레이션·확장(`list_migrations`/`list_extensions`)

- 마이그레이션 16건 확인(하드닝: `harden_function_search_path`·`revoke_execute_rls_auto_enable` 포함) → [04 §6](04_데이터베이스_설계.md).
- 활성 확장: pgcrypto·supabase_vault·pg_stat_statements·uuid-ossp·plpgsql. `vector`·`pg_cron` 미설치(향후 고도화).

## 4. 시크릿 인벤토리 (이름만 — 값 절대 미기재)

| 시크릿 | 위치 | 회전/유의 |
|---|---|---|
| `SUPABASE_SERVICE_ROLE_KEY` | Edge Fn Deno.env / 서버 | 프론트·Git 금지 |
| `SUPABASE_ANON_KEY` | `app/config.js`(공개) | publishable, RLS가 보호(설계상 노출 허용) |
| `OPENAI_API_KEY` | Edge Fn 시크릿 | 서버 전용 |
| `ENTRA_CLIENT_SECRET` | 서버/시크릿 | **만료 2026-12-09 → 11월 갱신**(미갱신 시 로그인 중단, [CLAUDE.md §6]) |
| `ENTRA_TENANT_ID`/`CLIENT_ID` | 설정 | 문서에 값 미기재(`MS_connect/`만) |
| `ERP_DB_CONN` | `.env`/DPAPI | 서버 배포 시만. 관리자 PC는 DPAPI 폴백 |

> 상세 시크릿·현황은 [`이관/00_현재시스템_상태스냅샷`](../실제구축준비%20자료/이관/00_현재시스템_상태스냅샷.md) 참조(값 복제 금지).

## 5. 감사·로깅

- **챗봇**: `chat_log`(UPN·모델·토큰·추정비용·사용도구, 대화 원문 미저장). 관리자만 Edge Fn 경유 조회.
- **계정**: `vendor_account_log`(발급·초기화 이력).
- **개선 권고**: 개인정보·금액 화면 접근 로깅 확대([CLAUDE.md §6·§11-D]).

## 6. 개인정보 처리

- 급여·주민번호·인사평가 등은 **권한 그룹 밖 화면/로그/캐시에 미노출**.
- **예외**: 직원 이름(성명)은 마스킹 대상 제외([CLAUDE.md §1.7], 2026-07-07 관리자 결정). 단 이름과 결합된 민감 값은 보호.
- 챗봇 시스템 프롬프트가 개인정보·비밀값 응답을 금지.

## 7. 데이터 안정성·백업/DR

- **기준시각**: ERP 파생 데이터는 `etl_meta.v_last_success` 표기 → 배치 실패 시 마지막 성공분 유지·인지.
- **백업**: Supabase 자동 백업(플랜 의존). 운영 전환 시 중간DB·포털DB 백업·복구 정책 수립 필요([CLAUDE.md §11-E]).
- **무결성**: ETL은 추출=적재(유실 0) 확인 방식. 배치 이력으로 재적재 추적.
