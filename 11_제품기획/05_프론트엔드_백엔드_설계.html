05 · 프론트엔드·백엔드 설계서

관리자: 최동혁(dh.choi@jeilm.co.kr) · 기준일: 2026-07-08 · 상태: Draft 관련: 03 아키텍처 · 04 DB설계 · 02 SRS

A. 프론트엔드 설계

A.1 기술스택·원칙

  • 바닐라 JS + HTML, 프레임워크·번들러 없음. 실구축 프론트(app/)와 동적 페이지 2종만 ESM 모듈 + supabase-js(CDN esm.sh). 나머지는 단일 파일 자기완결 HTML(공유 편의, [CLAUDE.md §7]).
  • 공통 팔레트: --navy #1a2f4e / --accent #2f6fb3 …, 폰트 Pretendard/Malgun Gothic.
  • 데이터 접근은 app/lib/api.js 한 곳. 화면 곳곳에 fetch를 흩지 않는다([CLAUDE.md §3.3]).

A.2 모듈 구조

app/config.js               공개설정(SUPABASE_URL, ANON_KEY=publishable, DATA_BACKEND)
   └─ app/lib/supabaseClient.js   createClient(PKCE, storageKey 'jeilax_sb_auth')
        └─ app/lib/api.js         어댑터 단일 진입점
             ├─ supabaseAdapter   실 DB(운영)
             └─ mockAdapter       localStorage 'jeilax_link_v1'(데모 폴백, 운영 미사용)
   └─ app/lib/grid.js + grid.css  표준 편집 그리드(의존성 0)
화면 HTML ── <script type="module"> import { portalApi } from "../app/lib/api.js"
  • api.js export: portalApi(발주·상태·사진·검수·메시지·프로필·Realtime), authApi(signIn/out/currentUser), adminMsgApi(사내 협력사 메시지 관리).
  • window.portalApi = adapter로 비모듈 인라인 스크립트도 호환.

A.3 화면 목록

화면 경로 데이터 상태
챗봇 포털(메인) 04_챗봇_포털_데모UI.html jeil-chat 게이트웨이(SSE) + 정적 카드 실가동(로그인 시 실 AI)
협력사 로그인 app/vendor-login.html Supabase Auth 실가동
협력사 계정관리 app/admin-vendors.html Edge Fn + grid + adminMsgApi 실가동
협력사 모바일 포털 pages/협력사_모바일_포털.html portalApi(RLS 자기 거래처) 실가동
외주 발주·검사 대시보드 pages/외주발주_검사진행현황_2026.html portalApi + Realtime 실가동
부서 페이지 8종 pages/*_2026.html, 원가 2종 정적 하드코딩 샘플 데모(ERP 미연결)
  • 정적 부서 페이지는 ERP 원천 테이블을 카드에 라벨로 표기(인사 HDF070T·HAA010T, 자금 A_GL·F_LN_INFO, 영업 S_BILL, 구매 M_IV·B_BP, 품목 B_ITEM). 실연동 전환 설계 → 06.

A.4 그리드 표준(편집 표)

  • 수정 가능한 표는 직접 <table> 대신 표준 컴포넌트 createGrid(container, options)([CLAUDE.md §13]).
  • grid.js는 UI만. 데이터 조회/저장은 콜백(onSave/onBulkAction/onRowAction/onPaste)에서 호출측이 api.js로 처리.
  • CSV 내보내기 UTF-8 BOM, 클립보드 TSV, 외부 CDN 금지. 적용: admin-vendors.html(#vmGrid).

A.5 빌드

  • _build_bundle.py → 04 데모 + 부서페이지 10종 + 설문폼을 base64 인라인한 단일 공유본 JEIL_AX_포털데모_통합본.html 재생성(오프라인 배포용). 손으로 안 고침.

B. 백엔드 설계 (Edge Functions)

B.1 공통 패턴

  • 전부 verify_jwt=false로 배포하고 함수 내부에서 직접 인증 검증(Entra 토큰은 Supabase JWT가 아니므로).
  • 클라이언트는 anon 키만. service_role 키는 Edge Function Deno.env에만. RLS 우회가 필요한 특권작업(계정생성·비번초기화·로그조회)은 전부 여기서.

B.2 함수 명세 5종

함수 입력 인증 경계 동작 쓰기 대상
vendor-provision bp_cd(들) @jeilm.co.kr AND portal_admin auth.admin.createUser(app_metadata role=vendor·vendor_bp), 임시비번 vendor_account, vendor_account_log
approve-vendor application id, 승인/거부 role=internal updateUserById로 role·vendor_bp 부여 vendor_application
vendor-reset-password user @jeilm.co.kr AND portal_admin updateUserById({password}) vendor_account_log(last_reset_at)
jeil-chat messages[] Entra Bearer → Graph /me(@jeilm.co.kr) OpenAI 프록시 + Tool 3종(§B.3), SSE chat_log
jeil-chat-admin (조회) Entra + portal_admin chat_log 집계(최근 2000) (읽기)

B.3 챗봇 오케스트레이션(jeil-chat)

사용자 질의 ─▶ 1차 OpenAI 스트림 ─(tool_calls 수집)─▶ runTool() ─▶ 결과를 tool 메시지로 첨부
           ─▶ 2차 OpenAI 스트림(최대 1라운드) ─▶ SSE로 클라이언트 렌더
  • Tool 3종(포털DB만, 읽기전용): get_order_summary(상태별 건수·총액·납기7일), get_order_detail(po_no), get_inspection_pending.
  • AI가 SQL을 생성/실행하지 않는다. 사전 등록 Tool만. ERP 직접조회 없음.
  • 감사: chat_log에 UPN·모델·토큰·추정비용·사용도구 기록(대화 원문 미저장). 기본 모델 gpt-4o-mini.
  • 표기 불일치: 04 UI·관리자 콘솔은 "Claude" 계열 모델을 표기하나 실구현은 OpenAI → 09 ADR.

B.4 데이터 접근 계약(프론트↔백엔드)

  • 프론트는 두 경로로만 데이터에 닿는다: (1) supabase-js anon + RLS(직접 테이블/Realtime/Storage), (2) Edge Function(특권·챗봇).
  • 신규 기능은 이 두 경로 외의 우회를 만들지 않는다. ERP 사본 노출도 이 원칙 위에서 뷰/RPC/Edge Fn로 → 06.

B.5 향후 백엔드 방향(참고)

  • [CLAUDE.md §3] 하이브리드 권고: 프론트/챗봇은 Node/서버리스 계열, ERP 연동 백엔드는 .NET 또는 별도 읽기 API로 분리 가능. 확정 전까지 api.js 어댑터로 스택 교체 비용을 낮춘다. 결정은 09 ADR.
