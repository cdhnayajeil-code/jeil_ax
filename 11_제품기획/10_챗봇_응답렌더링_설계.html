10 · 챗봇 응답 렌더링 설계 — 템플릿화 고도화

관리자: 최동혁(dh.choi@jeilm.co.kr) · 기준일: 2026-07-20 · 상태: Accepted — P1+P2 구현 완료(2026-07-20, ADR-008) 관련: 05 FE·BE · 06 ERP연계 · 08 보안 · 09 ADR · [CLAUDE.md §7·§13]

챗봇 응답을 읽기 좋은 구조로 렌더링하는 방식을 검토·설계한다. 결론부터: "모델에게 템플릿을 시키는 방식"은 권장하지 않고, 마크다운 렌더링(P1)을 먼저 깔고 그 위에 서버 주도 구조화 뷰(P2)를 얹는 2단 구조를 권장한다. 근거와 위험은 §3~§7.

1. 현재 상태 (실측, 2026-07-20)

1-1. 라이브 챗봇의 렌더러는 3줄이 전부다

04_챗봇_포털_데모UI.html:1360

function escHtml(s){ return s.replace(/&/g,'&amp;').replace(/</g,'&lt;').replace(/>/g,'&gt;'); }
function mdLite(s){ // 최소 마크다운: **굵게** + 줄바꿈만 (그 외는 이스케이프)
  return escHtml(s).replace(/\*\*([^*]+)\*\*/g,'<b>$1</b>').replace(/\n/g,'<br>');
}

즉 라이브 응답에서 표·목록·헤딩·코드·링크는 전부 원문 텍스트로 노출된다. 모델이 월별 실적을 표로 내면 화면에는 이렇게 찍힌다.

| 월 | 급여대상 인원 |
|---|---|
| 2026-01 | 113 |

1-2. 스타일은 이미 있는데 라이브가 도달하지 못한다

같은 파일 :72~79에 .bubble table / th / td / td.r / ul / li / h4 / .src 스타일이 이미 완비돼 있다. 그런데 이 스타일을 쓰는 건 데모 경로뿐이다.

경로 렌더 방식 결과
데모(미로그인) 하드코딩 <table> HTML을 bubble.innerHTML에 주입(:854) 표·배지·링크 정상 표시
라이브(로그인) mdLite(text) (:1395) 마크다운 원문 텍스트

로그인해서 실데이터를 보는 쪽이 데모보다 못생겨 보이는 역전 현상이 현재 상태다. 사용자가 체감한 문제의 정체는 "템플릿이 없어서"가 아니라 "이미 있는 표현력에 라이브 경로가 연결되지 않아서" 다.

1-3. 서버는 이미 구조화 데이터를 갖고 있다

jeil-chat(v19)의 도구 17종은 전부 구조화 JSON을 반환한다.

// get_hr_headcount 반환 예
{ "기준시각":"…", "월별":[{"월":"2026-01","급여대상인원":113,"부서수":25}, …] }

그런데 이 JSON은 모델에게만 전달되고, 모델이 자연어·마크다운으로 다시 서술한 결과만 화면에 간다. 구조가 한 번 소실됐다가 모델의 재구성으로 복원되는 셈이다.

이 재구성이 실제로 틀린 적이 있다는 증거가 시스템 프롬프트에 남아 있다.

"월별 표를 그릴 때는 도구가 반환한 '월별' 배열의 각 월 값을 그대로 사용하고, 값이 없는 월을 임의로 '미제공'으로 적지 마세요."

방어 문구가 프롬프트에 들어갔다는 것은 모델이 표를 잘못 그린 사고가 있었다는 뜻이다. 즉 렌더링은 미관 문제만이 아니라 정확성 문제다.

2. 무엇을 "템플릿화"라고 부를 것인가 — 세 갈래

"템플릿화"라는 한 단어에 서로 다른 세 방식이 섞여 있다. 이걸 먼저 분리해야 판단이 선다.

방식 A · 마크다운 렌더 방식 B · 모델이 템플릿 블록 방출 방식 C · 서버 주도 구조화 뷰
누가 구조를 만드나 모델(마크다운) 모델(JSON/DSL) 서버(도구 결과)
프론트가 하는 일 마크다운 → HTML 블록 파싱 → 컴포넌트 데이터 → 뷰 컴포넌트
숫자의 출처 모델이 재서술 모델이 재서술 DB 직결
실패 시 모습 평문으로 degrade(우아함) 깨진 JSON 원문 노출(추함) 실패 여지 없음
프롬프트 비용 +0 토큰 스키마 설명만큼 증가 +0 토큰
모델 교체 내성 높음 낮음(준수율 의존) 무관

2-1. 방식 A가 업계 표준인 이유

ChatGPT·Claude·Gemini·Copilot·Perplexity 모두 기본 렌더는 마크다운이다. 이유는 취향이 아니라 구조적이다.

  1. LLM은 마크다운으로 대량 학습돼 있어 별도 지시 없이도 표·목록을 자연스럽게 낸다.
  2. 렌더에 실패해도 최악이 "평문"이다. 사용자는 내용을 여전히 읽을 수 있다(graceful degradation).
  3. 새 기능·새 도구를 추가해도 렌더러를 손댈 필요가 없다.

2-2. 방식 B를 권장하지 않는 이유

"모델이 {"type":"kpi","value":98} 같은 블록을 내면 카드로 그린다"는 발상은 직관적이지만, 실무에서 다음 문제로 되돌아온다.

  • fail-ugly: 스키마를 한 글자만 어겨도 사용자 화면에 JSON 원문이 노출된다. 방식 A의 실패(평문)와 달리 복구가 안 된다.
  • 스트리밍 불가: JSON은 닫히기 전까지 파싱할 수 없다. 응답이 다 끝날 때까지 빈 화면이거나, 반쯤 그려진 블록이 깜빡인다. 지금의 실시간 타이핑 경험이 사라진다.
  • 프롬프트 비대: 도구 17종 × 블록 스키마를 프롬프트에 설명해야 한다. 현 시스템 프롬프트는 이미 길다(약 2,000자). 매 요청 입력 토큰으로 과금된다.
  • 모델 종속: 라우팅 규칙상 모델은 교체 가능하다(ai_routing_rule). 저가 모델로 라우팅되는 순간 스키마 준수율이 떨어진다.
  • 이중 유지보수: 도구를 고칠 때 프롬프트의 스키마 설명도 같이 고쳐야 한다. 둘이 어긋나는 순간 조용히 깨진다.

판단: 방식 B는 "모델에게 UI 책임을 위임"하는 구조다. UI 정확성을 확률적 생성에 맡기는 것이므로, 사내 업무 데이터를 다루는 우리 용도에는 부적합하다.

2-3. 방식 C가 우리에게 특히 잘 맞는 이유

우리는 게이트웨이(jeil-chat)를 소유하고 있고, 도구 실행 시점에 이미 정확한 구조화 데이터를 손에 쥐고 있다. 이걸 모델에게만 주고 버릴 이유가 없다.

  • 표에 찍히는 숫자가 모델을 거치지 않는다 → 환각 0, §1-3의 방어 프롬프트가 불필요해진다.
  • 프롬프트 토큰 증가 없음, 오히려 모델이 표를 그릴 필요가 없어 출력 토큰이 준다.
  • 도구가 반환하지 않은 필드는 화면에 존재할 수 없으므로, 권한 게이트가 렌더 계층까지 자동 전파된다(§6-3).

이는 새로운 발상이 아니라 이미 확립된 패턴이다(Vercel AI SDK의 tool-invocation 기반 generative UI, OpenAI Assistants의 tool outputs 스트리밍). 다만 방식 A를 대체하는 게 아니라 그 위에 얹는 것이라는 점이 중요하다.

3. 권장 설계 — 2단 구조

                     ┌─ 텍스트 채널 ─────────────┐
모델 출력(마크다운) ──┤  P1: 마크다운 렌더러       ├─→ 말풍선 본문(해석·요약)
                     └───────────────────────────┘
                     ┌─ 데이터 채널 ─────────────┐
도구 결과(JSON) ─────┤  P2: 뷰 컴포넌트 5종       ├─→ 말풍선 상단 카드(수치)
                     └───────────────────────────┘

역할 분리가 핵심이다. 모델은 해석·판단·요약을 담당하고, 수치·표는 서버가 직접 그린다. 두 채널이 독립이므로 한쪽이 실패해도 다른 쪽은 살아 있다.

3-1. P1 — 마크다운 렌더러 (기반)

mdLite를 확장한다. 지원 범위는 의도적으로 좁게 유지한다.

문법 렌더 비고
**굵게** *기울임* `코드` <b> <i> <code> 기존 유지·확장
# ~ ### <h4> 통일 말풍선 안이라 레벨 구분 불필요
- 1. <ul> <ol> 중첩 1단계까지만
GFM 표 <table> 기존 .bubble table CSS에 그대로 연결
```코드블록``` <pre><code> 가로 스크롤
[텍스트](url) <a target=_blank rel=noopener> http(s): 스킴만 허용
raw HTML 미지원(이스케이프) XSS 차단 — §6-1
  • 외부 라이브러리·CDN 금지(CLAUDE.md §13.5, 자기완결 원칙 §7) → 바닐라 자체 구현. 위 범위면 150줄 내외.
  • ~~배치: app/lib/chat-render.js(ESM)~~ → (구현 시 조정) 04 인라인 배치: 실측 결과 04는 로컬 <script src>가 전혀 없는 완전 인라인 단일 파일이고 _build_bundle.py가 전문을 그대로 내장(스크립트 src 인라인 처리 없음) — 외부 파일 분리 시 통합본·file:// 시연이 깨진다. 유일 소비처가 04이므로 인라인(주석 마커 chat-render/chat-views 구간)으로 확정, 실구축 프론트(app/) 이관 시 추출한다.

P1만으로 체감 개선의 대부분이 해결된다. §1-2에서 확인했듯 CSS는 이미 준비돼 있어 연결만 하면 되기 때문이다.

3-2. P2 — 서버 주도 구조화 뷰

SSE 프로토콜 확장. 현재 게이트웨이는 OpenAI SSE를 그대로 프록시하지만, 우리가 소유한 스트림이므로 이벤트를 추가할 수 있다. 도구 실행 직후(index.ts의 runTool 반환 지점) 한 줄을 흘려보낸다.

// 기존(변경 없음)
data: {"choices":[{"delta":{"content":"6월 재적은 98명입니다."}}]}
// 추가
data: {"jeilax":{"view":"series","title":"2026년 월별 급여대상 인원",
       "unit":"명","asOf":"…","rows":[{"k":"2026-01","v":113}, …],
       "note":"급여대장(HDF070T) 기준 · 마감 전 변동 가능"}}

하위호환: 기존 프론트는 choices가 없는 이벤트를 try/catch로 무시한다(:1396의 기존 동작). 즉 게이트웨이를 먼저 배포해도 구 프론트가 깨지지 않는다.

3-3. 뷰는 "도구별"이 아니라 "데이터 형태별" 5종으로 고정한다

여기가 이 설계의 성패를 가르는 지점이다. 도구마다 템플릿을 만들면 17개 → 앞으로 30개로 늘고 유지보수가 무너진다. 데이터 형태로 추상화한다.

뷰 형태 화면 해당 도구
series 시계열 표 + 미니 막대 매출·매입 월집계, 발주 월별, 인원현황
ranking 상위 N 순위 표 거래처Top10, 발주 상위
record 단건 상세 라벨-값 카드 + 진행단계 발주 상세, 구매요청, 검사 상세
list 목록 표(스크롤) 미입고, 검수대기, 구매요청, 문서검색
notice 안내·권한 아이콘 카드 + 액션 접근제한 응답, get_my_access

새 도구를 추가할 때 하는 일은 반환값에 view:"series" 한 줄 추가뿐이다. 프론트는 손대지 않는다.

notice 뷰의 가치: 지금 접근제한 응답은 모델이 안내 문구를 풀어 쓰는데, 이걸 카드로 그리면 "요청할 모듈명·부여 경로"가 항상 같은 자리에 나온다. 권한 요청 UX가 표준화된다(§권한 개선 작업과 직결).

3-4. P3 — 액션(선택, 후순위)

get_my_access가 접근 불가 페이지를 알고 있으므로 "권한 요청" 버튼을, 페이지 링크는 기존 .pglink-btn 스타일 카드로. 단, CLAUDE.md §1.6에 따라 버튼은 요청 초안 생성까지만이고 승인·발주 확정 등 실거래 액션은 넣지 않는다.

4. 효율성 — 정직한 평가

사용자 질문("효율적인지")에 과장 없이 답한다.

항목 평가
개발 비용 P1 약 1일, P2 약 2~3일. 뷰 5종 고정이라 이후 한계비용 거의 0
런타임 성능 마크다운 파싱은 응답당 수 ms. 단, 현재 매 delta마다 전체 재파싱(O(n²)) — 긴 답변에서 렉. §6-2에서 해결 필요
토큰 비용 월별 표 1개 ≈ 출력 300토큰. gpt-4o-mini 기준 1,000회/월이면 월 $0.2 수준 — 무시할 만함
실질 이익 비용 절감이 아니라 ① 수치 정확성(환각 제거) ② 일관성 ③ 권한 게이트의 렌더 전파

비용 절감을 근거로 삼지 말 것. 이 작업의 정당화는 정확성과 신뢰다.

5. 단계별 적용 계획

단계 범위 산출물 상태
P1 마크다운 렌더러 04 인라인 chat-render 구간(renderMd) + 스로틀·스크롤 가드 + XSS 수정, 통합본 재빌드 ✅ 완료(2026-07-20, 단위테스트 15/15)
P2-a SSE jeilax 이벤트 송출 + 도구 __view 배선(17종) jeil-chat v20 ✅ 완료(2026-07-20)
P2-b 뷰 5종 구현 04 인라인 chat-views 구간(renderView) + .cv- CSS ✅ 완료(2026-07-20)
P3 액션·후속질문 칩 — 🕐 보류(별도 결정, §1.6 실거래 금지 준수)

P1과 P2-a는 서로 독립이라 병렬 진행 가능하다. P2-a를 먼저 배포해도 구 프론트가 이벤트를 무시하므로 안전하다(§3-2).

6. 위험과 대응 — "이후에 문제없을지"

6-1. XSS (가장 중요)

마크다운 렌더러를 넣는 순간 이것이 최우선 위험이 된다.

  • raw HTML 미지원을 원칙으로. escHtml 먼저, 그 다음 마크다운 문법 치환(순서를 뒤집으면 구멍이 난다).
  • 링크는 http:/https:만 허용. javascript:·data: 차단. rel="noopener noreferrer".
  • 뷰 컴포넌트는 textContent로 값 삽입(innerHTML 금지). 도구 결과에는 ERP 품목명 등 외부 유래 문자열이 있다.
  • 기존 결함 1건: send()(:1804)가 사용자 입력에서 <만 치환하고 &를 놓친다. 같은 작업에서 escHtml로 통일할 것.

6-2. 스트리밍 중 렌더 안정성

  • 매 delta마다 bubble.innerHTML = mdLite(text) 전체 재구성 → O(n²). 긴 답변에서 체감 렉. requestAnimationFrame 스로틀(약 60ms)로 묶는다.
  • 표가 그려지는 중간에는 행이 하나씩 늘며 깜빡인다. 미완결 표 블록은 마지막 줄바꿈이 올 때까지 평문으로 유예하고 완결 시 표로 승격.
  • 스크롤: 렌더 중 사용자가 위로 스크롤했으면 자동 하단 이동을 멈춘다(현재 무조건 scrollBottom()).

6-3. 권한 (우리 프로젝트 고유 위험)

  • 뷰는 도구가 반환한 필드만 그린다. 서버가 이미 dept_erp_scope로 스코프를 걸었으므로 권한 밖 데이터는 애초에 페이로드에 없다 → 렌더 계층 권한 누수 구조적으로 차단.
  • 접근제한:true 응답은 반드시 notice 뷰로 그리고, 서버가 준 안내 문구를 그대로 표시한다(프론트가 문구를 지어내지 않는다).
  • 뷰 데이터를 localStorage·sessionStorage에 캐시하지 않는다. 급여·인원 데이터가 브라우저에 잔류하면 §1.7 위반이다.
  • 감사: chat_log는 대화 원문 미저장이 원칙. 뷰 페이로드도 저장하지 않는다(현행 tools_used 기록만 유지).

6-4. 유지보수

  • 뷰는 5종 고정. 도구별 템플릿 신설 금지를 규칙으로 명문화한다. 새 형태가 정말 필요하면 ADR 항목으로 올린다.
  • 도구 반환의 view 힌트가 누락되면 뷰 없이 마크다운만 표시(fail-safe). 화면이 깨지지 않는다.

6-5. 그리드 표준(§13)과의 경계 — 혼동 주의

CLAUDE.md §13은 편집 가능한 그리드를 app/lib/grid.js로 통일하라고 규정한다. 챗봇 응답 뷰는 읽기 전용이므로 그 대상이 아니다.

  • 챗봇 뷰에 grid.js를 쓰지 않는다(편집·붙여넣기·CSV 기능이 불필요하고 무겁다).
  • 다만 CSS 팔레트(:root navy/accent/green/red/line)는 공유해 시각적 일관성을 유지한다.
  • 챗봇 뷰 CSS는 .chatview__ 스코프로 격리한다. 전역 table/th/td 재정의 금지(§13.4와 동일 원칙).

6-6. 기타

  • 복사·재활용: 실제 <table> 태그로 그린다. div 격자로 만들면 엑셀 붙여넣기가 깨진다. 사내에서 챗봇 표를 엑셀로 옮기는 건 흔한 실무다.
  • 모바일: 표는 자체 overflow-x:auto 컨테이너에. 말풍선(.bubble)에 이미 있으나 뷰에도 개별 적용.
  • 접근성: <th scope="col">, 숫자는 우측 정렬(td.r 기존 클래스 재사용), 색상만으로 의미를 전달하지 않는다.
  • Azure 이관(07): 뷰는 순수 프론트 컴포넌트라 백엔드 교체 영향 없음. SSE 이벤트 형식만 어댑터 계약에 포함시킨다.
