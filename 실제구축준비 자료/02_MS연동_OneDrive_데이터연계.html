3OneDrive/SharePoint 데이터 연계 — 설계 원리

"OneDrive 데이터 적용"은 두 가지 성격이 다른 요구를 포함하므로 분리 설계합니다.

요구예시 질문적용 방식
A. 내 문서 작업"내 OneDrive에서 지난주 회의록 찾아 요약해줘", "이 견적서 초안 검토해줘"위임 권한 실시간 검색 — 로그인한 사용자의 권한으로 Graph 검색 → 결과를 챗봇 컨텍스트에 주입
B. 회사 공통 지식"연차 규정 알려줘", "출장비 한도는?", "품의서 양식 어디 있어?"AI 지식 라이브러리(전용 SharePoint) 사전 인덱싱 — 전 직원 동일 답변, 출처 링크 제공
문서 원천 참고 — 사내 문서중앙화(Cloudium)와의 관계 당사 업무문서는 OneDrive와 사이버다임 Cloudium(문서중앙화)에 혼재합니다. 본 장의 방식 A/B는 OneDrive·SharePoint 영역을 다루며, Cloudium 문서의 연계 경로(벤더 API·큐레이션 반출·AI Assistant 비교)는 06 문서에서 별도 검토합니다. 규정·양식류는 원천이 어디든 AI 지식 라이브러리로 모아 단일 인덱싱하는 원칙(06 문서 2장)을 따릅니다.

4데이터 거버넌스 — "어떤 문서를 AI에 보여줄 것인가"

구분정책
지식 라이브러리 등재 기준전 직원 열람 가능 문서만 (규정·양식·공지·매뉴얼). 개인정보 포함 문서, 인사평가, 계약서, 원가·견적 원본은 등재 금지
등재 절차부서 → 등재 요청 → 관리자(최동혁) 검토 → 라이브러리 업로드. 사이트 쓰기 권한은 관리자만, 일반 직원은 읽기 전용
문서 책임자문서별 메타데이터(담당 부서·개정일) 기록 — 개정 시 교체 책임을 부서에 부여, 분기 1회 최신성 점검
개인 문서(방식 A) 보호검색·요약은 사용자 본인 세션에서만 수행, 문서 본문은 대화 로그에 원문 저장하지 않음(요약 결과만 저장) — 04 문서 로그 정책과 연동
외부 전송 고지문서 내용이 Claude API(Anthropic)로 전송됨을 이용 안내에 명시 — API는 기본적으로 입력을 모델 학습에 사용하지 않음(03 기획서 보안 표 준용)
금지 사항 — 설계로 강제 ① 앱 단독 권한으로 전 직원 OneDrive를 읽는 권한(Files.Read.All application)은 신청하지 않는다 — 사고 시 전사 문서 유출 경로가 됨. ② 지식 라이브러리 외 사이트는 Sites.Selected 허용 목록에 추가하지 않는다. ③ Graph 권한 추가는 반드시 본 문서 갱신 + 관리자 동의 기록과 함께.
