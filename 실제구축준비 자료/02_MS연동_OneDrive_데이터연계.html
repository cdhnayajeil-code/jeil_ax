3OneDrive/SharePoint 데이터 연계 — 설계 원리

"OneDrive 데이터 적용"은 두 가지 성격이 다른 요구를 포함하므로 분리 설계합니다.

요구예시 질문적용 방식

7적용가능성 검토 — 현행 실측 기준 (2026-07-09 추가)

본 문서 §3~5는 계획이었고, 그 사이 챗봇 jeil-chat이 실제 가동됐다. 이 절은 지금 라이브인 구조를 기준으로 OneDrive 문서 연계의 적용가능성을 재검증한 결과다. (문의 검토 — 코드·설정 변경은 미수행)

결론 — 적용 가능(높음) "OneDrive 권한을 개별로 그대로 적용"하려는 목표는 방식 A(위임·라이브 Graph 검색)가 정석이다. Microsoft Graph가 사용자 본인 권한으로 검색을 자동 필터(보안 트리밍)하므로 포털이 별도 권한 테이블을 만들 필요 없이 OneDrive ACL이 곧 통제선이 된다. 현행 jeil-chat의 도구 호출 구조에 "문서 검색" 도구 하나를 얹는 형태로 자연스럽게 확장된다.

7-1. 현행 실측 상태 (이 문서 작성 이후 가동분)

7-3. 방식 A vs B — 권한 관점 재확인

방식권한 처리난이도·비용적합
A. 라이브 위임Graph 자동 보안 트리밍 — 권한 유출 위험 낮음, 항상 최신낮음(Graph 호출 위주)1차 권장
B. RAG 인덱스색인에 파일별 ACL 저장 + 권한 변경 동기화(보안 트리밍)를 직접 설계 — 어긋나면 유출높음(벡터DB·임베딩·동기화)2차(품질 필요 시, 권한 트리밍 선결)
RAG(방식 B)의 핵심 리스크 사전 인덱싱은 검색 품질이 좋지만, 요청 사용자 권한으로 결과를 필터하는 보안 트리밍이 난제다. ACL 색인·동기화가 어긋나면 권한 없는 사람에게 문서가 노출된다 — 급여·인사 등 민감 데이터에서 특히 위험(§1·§6). 그래서 §3에서 정한 대로 B는 전 직원 공개 문서만 담는 전용 라이브러리로 한정하고 Sites.Selected 최소권한으로 운영하는 것이 옳다.

8AI 작업 범위 거버넌스 — ai.jeil 화이트리스트 + 폴더 지정 모델 (2026-07-09 기획·구현)

방식 A는 로그인한 사용자 본인 토큰으로 검색하므로, AI가 뒤지는 범위 = 그 사람이 볼 수 있는 모든 문서다(§7-5에서 dh.choi 기준 31,980건). 권한 유출은 없으나 범위를 좁히는 장치가 없다 — "OneDrive 폴더가 너무 많아 관리가 안 된다"는 우려의 실체가 이것이다. 본 절은 "AI가 애초에 손댈 수 있는 컨테이너"를 화이트리스트로 한정하는 거버넌스를 설계한다. (기획 — 코드·설정 변경은 미수행)

8-7. 폴더 지정 모델 — 3단계 스코프 + 관리자/사용자 지정 (2026-07-09 구현)

이번 파일럿에서 한 사이트에 문서 라이브러리가 2개(SiteAssets vs Shared Documents)라 자동 추측이 엉뚱한 쪽을 잡는 사고가 있었다(§8-6). 해법은 "자동 추측"을 버리고 관리자/사용자가 대상 폴더를 명시적으로 지정하는 것. 이를 위해 ai_document_scope를 site/library/folder 3단계 스코프로 확장했다(구현·v18 배포 완료).

단계의미저장(ai_document_scope)
folder(권장)라이브러리 내 특정 폴더만(예: ERP_DML). 가장 정밀 — 같은 라이브러리 다른 폴더는 제외scope_type='folder', drive_id+path_prefix(폴더 웹URL)
library문서 라이브러리(드라이브) 전체scope_type='library', drive_id+path_prefix(라이브러리 URL)
site사이트 전체(모든 라이브러리) — 거칠어 지양scope_type='site', drive_id별 행 + 사이트 URL
jeil-chat v18 강제 방식(이중 게이트 + 폴더 경로) 검색은 KQL path:"<지정경로>"로 한정하고, 결과를 승인 driveId 일치 AND 경로가 지정 접두로 시작하는지 하드 필터한다(경로 스코프 누수 대비). 판독도 문서 webUrl이 지정 폴더 하위인지 2차 검증. 즉 폴더 레벨까지 정밀 강제되고, 그 위에 사용자 위임 권한 트리밍(Graph)이 더해진다.

지정 방법 (사용자 친화 순)

방법절차주체
A. 링크 붙여넣기(권장)관리자 콘솔에 SharePoint 폴더 링크를 붙여넣으면 Graph /shares/{link}/driveItem로 해석 → site_id·drive_id·item_id·web_url 자동 추출 → folder 스코프로 등록. 비개발자도 링크 복사만. 라이브러리 오지정 원천 차단관리자/부서담당
B. ai.jeil 공유 + 폴더 확정사이트/폴더에 ai.jeil 계정을 멤버로 추가(자기서비스) → 배치가 후보 사이트를 열거해 "지정 대기"로 올림 → 관리자가 그 사이트의 어느 폴더인지 1클릭 확정부서담당+관리자

핵심 원칙 — "후보는 자동, 최종 폴더 지정은 명시적" ai.jeil 멤버십은 어느 사이트가 후보인지만 자동으로 좁힌다(자기서비스 편의). 그러나 구체적으로 어느 라이브러리/폴더를 AI에 열지는 사람이 확정한다 — 이번 SiteAssets 오지정 같은 자동 추측 사고의 재발을 원천 차단. 미확정 후보는 "지정 대기"로 남고 자동 등록하지 않는다.
