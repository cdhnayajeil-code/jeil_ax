EXECUTIVE SUMMARY

  • UNIERP 운영 DB에 챗봇·대시보드가 직접 연결되지 않습니다. 사외 IDC MSSQL → 야간 배치 → Azure PostgreSQL 중간 DB → 조회 API·Tool Use 순으로만 데이터가 흐릅니다.
  • 챗봇과 pages/ 부서 대시보드는 같은 중간 DB를 쓰되, 같은 조회 API·쿼리 로직을 공유합니다 — ETL은 1벌만 유지.
  • AI는 임의 SQL을 생성하지 않습니다. 사전 검증된 화이트리스트 템플릿 + Entra RBAC + 행 수 상한(200행)으로 안전하게 조회합니다.
  • 민감 영역(자금·인사)은 erp_secure 스키마 + Entra 그룹으로 분리하며, 개인 단위 민감정보는 중간 DB에 넣지 않습니다.

4동기화 배치 (ETL) 정책

항목정책
주기기본 매일 03:00 — ERP 백업·마감과 겹치지 않게 유니포인트 합의
방식증분 (수정일시 기준) / 소량 집계는 전체 재적재
변환코드→명칭, 월별 집계 — 챗봇이 코드 해석 없이 바로 답할 형태
실패Teams/메일 알림 + 1시간 후 재시도 1회
UI모든 화면·답변에 「데이터 기준: YYYY-MM-DD 03:00」 표기
네트워크파일럿: IP 허용+TLS → 운영: IPSec VPN (03 문서 3장)
03:00 스케줄 → ERP View SELECT (jeilax_ro, 증분) → 코드·명칭 변환 + 집계 → PostgreSQL UPSERT (erp_ro / erp_secure) → etl_meta 갱신 (건수, 시각, batch_id) → (실패 시) 알림 + 1회 재시도

방식문제
챗봇 → UNIERP 직접 연결ERP 성능·장애 전파, AI 임의 SQL 위험
MSSQL 전체 복제용량·민감컬럼 과다, 비용↑
