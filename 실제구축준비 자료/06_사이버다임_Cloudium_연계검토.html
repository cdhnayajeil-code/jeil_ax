사내 문서중앙화 데이터의 AX 포털 직접 연결 가능성 — 연계 경로 5안 평가 · 벤더 질의서 · 권고 로드맵

검토 결론 (요약)

  • Cloudium 자체의 공개 API 문서는 확인되지 않으나, 같은 회사 상위 제품 Destiny ECM이 Open API·외부 시스템 연동을 공식 지원하므로(그룹웨어·ERP·검색엔진 연동 실적 다수) 구축형 Cloudium에서도 연동 경로가 열려 있을 가능성이 높습니다. "직접 연결이 되느냐"의 최종 답은 사이버다임 공식 질의로만 확정 가능합니다 — 본 문서 4장의 질의서를 발송하는 것이 첫 행동입니다.
  • 벤더 회신과 무관하게 즉시 실행 가능한 안전 경로가 있습니다: Cloudium의 "외부 반출 승인" 절차를 통해 규정·양식 등 공개 가능 문서를 AI 지식 라이브러리(02 문서)로 큐레이션 반출하는 방식 — 문서중앙화 보안 정책을 그대로 준수하면서 규정 Q&A를 먼저 가동할 수 있습니다.
  • 사이버다임이 자체 AI Assistant(문서 기반 챗봇, 구축형 지원)를 출시했으므로, "Cloudium 문서 Q&A는 벤더 기능, AX 포털은 ERP·OneDrive·범용 AI"로 역할을 나누는 대안도 비용·도입 속도에 따라 유효합니다. 단, 챗봇 창구가 둘로 나뉘는 단점이 있어 비교 견적 후 결정을 권고합니다.
  • 가상 드라이브 우회 읽기, DB 직접 조회는 벤더 승인 없이는 금지 — 문서중앙화 도입 취지(유출 통제)와 유지보수 계약을 훼손합니다.

1조사로 확인된 사실 (2026-06-12 기준)

항목확인 내용
제품 구조사이버다임은 Destiny ECM(대기업·공공, 연동 중심)과 cloudium(중소·중견, 패키지형) 2개 제품 운영. Cloudium은 Windows 탐색기(가상 드라이브)·웹 UI·모바일 앱 제공
당사 도입 형태구축형/구축과금형 — 자체(사내·IDC) 서버 설치형. 클라우드 과금형 대비 서버 측 구성·연동 옵션 폭이 넓고, AI Assistant 지원 대상
보안 통제 기능로컬 PC 저장 차단, 직급·부서별 접근 권한, 외부 반출 승인 절차, 저장매체 제어, 전 활동 이력 기록 — 연계 설계 시 이 통제들을 우회가 아니라 활용해야 함
외부 연동 단서Destiny ECM: "Open API를 통한 외부사이트 연동", Iframe·SQL·API·RSS 포틀릿, 그룹웨어·메신저·검색엔진 연동 공식 지원. Cloudium: MS Teams 연동 기능 보유(외부 시스템 연동 역량의 방증), 내부/외부 URL 공유 제공
AI Assistant (벤더 자체 AI)2026년 업데이트로 탑재 — 중앙화 문서 기반 질의응답·요약·의미 검색, 업무 카테고리별 Assistant 구성(개인/공용), 단일 파일 AI 채팅. 접근 권한에 따라 답변 범위 자동 조정. 과금형(클라우드)에서는 미지원 — 당사(구축형)는 지원 대상
미확인 (질의 필요)Cloudium의 API 제공 여부·범위·인증 방식, AI Assistant의 LLM 구조(온프레미스 sLLM인지 외부 API인지)·추가 비용, 시스템 계정 정책, 반출 자동화 가능성

출처: 사이버다임 cloudium 제품 페이지, AI Assistant 공식 블로그(Product), Destiny ECM 제품 페이지·소개자료 — 말미 참고 링크

2문서 원천 지도 — Cloudium·OneDrive 혼재 환경의 교통정리

당사는 업무문서가 Cloudium과 OneDrive에 나뉘어 있습니다. AI 연계 설계 전에 "무엇이 어디 있고, AI에는 어떤 경로로 보여줄지"를 먼저 고정합니다.

원천담긴 문서(통상)AI 연계 경로
Cloudium (문서중앙화)부서 업무문서 원본 — 보고서·계약·도면·품의 등 통제 대상 핵심 문서본 문서의 검토 대상 — ① 벤더 API(확인 중) ② 큐레이션 반출 ③ AI Assistant(벤더 기능)
OneDrive/SharePointMS365 협업 문서, 개인 작업 문서, Teams 공유 파일02 문서 확정 설계 — 위임 권한 실시간 검색(방식 A)
AI 지식 라이브러리 (신설, SharePoint)전 직원 공개 가능한 규정·양식·지침의 AI용 사본02 문서 방식 B(사전 인덱싱) — 원본 출처가 Cloudium이든 OneDrive든 여기로 모아 단일 인덱스
설계 원칙 — "AI용 단일 창구는 지식 라이브러리" 규정·양식처럼 전 직원 대상 문서는 원천이 어디든 AI 지식 라이브러리 한 곳에 사본을 모아 인덱싱합니다(출처 메타데이터에 Cloudium 경로 기록). 이렇게 하면 Cloudium API 확보 여부와 무관하게 규정 Q&A가 먼저 가동되고, 추후 API가 열리면 "라이브러리 사본 동기화"만 자동화로 바꾸면 됩니다 — 설계 변경 없이 수작업이 자동화로 진화하는 구조입니다.

3연계 경로 5안 평가

경로방식실행 가능성평가
① 벤더 공식 API 연동 사이버다임 제공 API/연동 모듈로 포털 게이트웨이가 문서 검색·메타데이터·본문 조회 (Destiny ECM의 Open API에 준하는 기능을 Cloudium 구축형에 적용 가능한지 질의) 질의 필요 1순위 목표 — 권한 체계를 살린 정식 연동. 검색·본문 추출이 되면 ERP처럼 Tool Use 도구화. 비용·개발 범위는 회신 후 판단
② 큐레이션 반출 → AI 라이브러리 Cloudium 외부 반출 승인 절차로 공개 가능 문서만 SharePoint 라이브러리로 사본 등재 → 02 문서 방식 B 인덱싱 즉시 가능 단기 실행안 — 반출 통제 정책 100% 준수, 추가 비용 0. 단점: 수동 등재·개정 시 교체 부담 (분기 점검으로 관리)
③ 벤더 AI Assistant 활용 Cloudium 안에 탑재된 AI 챗봇을 별도 도입 — 중앙화 문서 Q&A는 이 기능이, AX 포털은 ERP·OneDrive·범용 AI 담당 견적 필요 보완재 — 권한 연동·문서 커버리지는 최상이나 챗봇 창구 이원화(직원 혼선), 추가 라이선스 비용, Claude 포털과 대화이력·크레딧 통합 불가. 비용이 합리적이면 "Cloudium 전용 Q&A"로 병행 가치
④ 가상 드라이브 인덱싱 전용 Windows 머신에 Cloudium 에이전트+시스템 계정 → 탐색기 드라이브를 인덱서가 직접 읽음 기술적 가능 벤더 서면 승인 없이 금지 — 반출 통제 우회 소지, 에이전트 보안정책 충돌, 유지보수 계약 위반 리스크. 승인 시에도 ②의 자동화 수단으로만 한정
⑤ 서버 DB·스토리지 직접 조회 Cloudium 서버의 DB/파일 저장소를 직접 읽기 불가 간주 금지 — 저장 암호화·비공개 스키마, 패치 시 파손, 계약 위반. 검토 대상에서 제외

6보안·정책 점검 (문서중앙화 취지와의 정합)

쟁점정리
반출 통제와 AI 연계의 충돌충돌하지 않게 설계 — 모든 사본 이동은 기존 반출 승인 절차를 통과시키고, 승인 이력이 Cloudium에 남게 함. "AI를 위해 통제를 푼다"가 아니라 "통제된 문만 통과한다"
권한 매핑 (직접 연계 시 최대 난제)Cloudium 권한(직급·부서)과 포털 권한(Entra 그룹)은 별개 체계 — 경로 ① 채택 시 API가 사용자별 권한을 반영하는지(질의 2번)가 핵심. 반영 안 되면 공개 등급 문서만 연계(라이브러리와 동일 효과)로 범위 축소
민감 문서계약서·인사·원가 원본은 어떤 경로로도 AI 인덱스에 넣지 않음 — 02 문서 등재 금지 기준, 03 문서 민감도 분류와 동일 원칙 적용
외부 API 전송 고지반출·연계된 문서가 Claude API로 전송될 수 있음을 이용 수칙에 포함 (04 문서 7장) — AI Assistant 채택 시에도 LLM 데이터 처리 방식 확인(질의 3번)
이력·감사포털 측 문서 조회도 tool_audit(04 문서)에 전건 기록 — Cloudium 이력과 포털 이력이 양쪽에 남는 이중 감사 구조
