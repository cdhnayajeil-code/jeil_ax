1. 데이터 접근 권한 모델 (누가 무엇을 볼 수 있나)

원칙(CLAUDE.md §5): 권한 판정은 백엔드(게이트웨이)에서. 프론트 메뉴 숨김은 UX일 뿐이며, 실제 차단은 게이트웨이·DB 레벨에서 한다. 권한의 기준은 Entra 보안그룹이다.

1.1 권한 등급 설계 (Entra 보안그룹 매핑)

등급Entra 그룹챗봇 기능데이터 접근 범위
전체 관리자AI-Portal-Admins전 기능 + 관리자 콘솔(사용량·권한·모델 설정)전 모듈 (읽기전용)
부서 사용자AI-Portal-Finance / -HR / -Sales 등챗봇 + 부서 데이터 질의소속 부서 매핑 모듈만 (예: 자금팀 → 자금일보·잔액)
일반 사용자(그룹 미소속 사내 계정)챗봇 일반 질의·문서 초안전사 공개 데이터만 (규정·공지)
협력사Entra 테넌트 분리 — Supabase Auth 별도챗봇 미제공 (발주 포털만)자기 발주 건만 (RLS 행 수준 보안, 기가동)
⚠ 관리자 선행 작업: Entra 보안그룹 생성·인원 배정은 관리자(최동혁)가 Entra 관리센터에서 직접 수행(§11.C 체크리스트). 그룹 생성 후 게이트웨이가 Graph /me/memberOf로 그룹을 판독하는 방식 — 코드 변경만으로는 적용 불가.

1.2 권한 판정 흐름 (게이트웨이 구현 방식)

[사용자 질의] → jeil-chat 게이트웨이 ① Entra 토큰 검증 (Graph /me) ── 실패 → 401 ② 그룹 조회 (Graph /me/memberOf, 5분 캐시) → 등급 결정 ③ 질의가 데이터 도구를 요구하면: 등급별 도구 화이트리스트 검사 ── 권한 밖 → "권한 없음" 안내 ④ 도구 실행 시 데이터 레벨 2차 필터 (부서 컬럼·행 수준) → 응답 생성 ⑤ chat_log 기록: 사용자·등급·사용 도구·데이터 범위
  • 이중 차단: 도구 목록(어떤 도구를 쓸 수 있나) + 데이터 필터(도구 안에서 어떤 행·컬럼을 볼 수 있나). 급여·주민번호 등 PII 컬럼은 어떤 등급에도 챗봇 경유 노출 금지(§7 개인정보 원칙).
  • 감사: 데이터 도구 호출은 일반 질의보다 상세히 기록(조회 모듈·조건) — 민감 화면 우선 로깅(CLAUDE.md §6).

2. 데이터 적용 방식 (챗봇이 데이터를 읽는 방법)

방식은 Tool Use(함수 호출) 를 기본으로 한다: 모델이 직접 DB에 붙는 게 아니라, 게이트웨이가 등록한 "조회 함수"를 모델이 호출하고 게이트웨이가 실행·필터 후 결과만 모델에 전달한다. ERP 원본 직접 조회는 금지(CLAUDE.md §4 — 읽기전용 + 중간 DB 경유).

2.2 공통 규칙

  • 모든 도구는 게이트웨이 코드에 사전 등록된 것만 실행(모델이 임의 SQL 생성 금지). 도구 입력은 파라미터 바인딩만.
  • 도구 결과에는 기준일·출처를 붙여 응답에 표기("2026-07-06 중간DB 기준").
  • 전송 전 필터: 주민번호·계좌·전화 패턴 마스킹, 금칙어 차단(외부 API 전송 전 적용 — 04 문서 §2 계승).
  • 실거래 액션(발주 확정·결재) 도구는 만들지 않는다 — 조회·집계·초안까지만(CLAUDE.md §1.6).

3. 사용량 관리 (측정 → 한도 → 비용)

3.1 현행 측정 (가동 중)

  • chat_log: 사용자(UPN)·모델·맥락 메시지 수·질문 문자수·시각. 관리자 콘솔 "사용량" 탭에서 오늘/7일/누적·사용자별·최근 20건 실조회.

3.2 고도화 순서

순번항목내용
U-1토큰·비용 기록OpenAI 스트림의 usage(stream_options.include_usage)를 게이트웨이가 캡처해 chat_log에 입력/출력 토큰·추정비용 컬럼 추가 → 사용자·월별 실비용 집계
U-2월 사용자 한도등급별 월 호출/토큰 한도(라이트·스탠다드·프로 — 데모UI 예시안 계승). 게이트웨이가 호출 시 잔여량 검사, 초과 시 안내+신청 절차
U-3한도·이상 알림월 지출 임계(예: 80%) 도달, 비정상 급증(1인 폭주) 시 관리자 메일 알림. OpenAI 콘솔 Usage Limit도 이중 설정
U-4보존 정책chat_log 1년 보존 후 자동 파기(04 문서 로그 정책 계승) — pg_cron 삭제 배치

4. 관리자 콘솔 고도화 (실데이터 ↔ 기획 예시 매핑)

데모UI 관리자 콘솔의 각 탭에서 상단 "● 실데이터" 섹션은 실측, 그 아래는 실구축 기획 예시 화면이다. 예시 항목을 실기능으로 전환하는 순서:

탭지금 실데이터인 것실기능 전환 대상(우선순위순)
API 설정게이트웨이 상태·운영 모델·인가 정책·입력 상한① 연결 테스트 버튼(게이트웨이 핑) ② 모델 전환(OPENAI_MODEL 변경 절차 안내) ③ 키 교체 절차서(분기 1회)
사용량오늘/7일/누적 호출, 사용자별, 최근 20건① U-1 토큰·비용 표시 ② U-2 등급·한도 관리(실 저장) ③ 크레딧 신청·승인 워크플로
모델 설정(운영 모델 표기만)① 모델 활성/라우팅 규칙 실저장(게이트웨이 설정 테이블) ② 시스템 프롬프트 관리자 편집

구현 원칙: 관리자 콘솔의 "저장"이 실제로 반영되려면 설정 저장 테이블(chat_config)과 게이트웨이의 설정 재로딩이 필요 — 콘솔 고도화는 설정 테이블 신설 → 게이트웨이 참조 → 콘솔 편집 UI 순으로 진행한다.
