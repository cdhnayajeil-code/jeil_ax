1. 결론 먼저 (3줄 요약)

⚠️ 실측: 그래프 토큰이 비어 있었음

2026-06-30 관리페이지에서 확인 — 로그인돼 있어도 provider_token(MS Graph 토큰)이 실제로 없음. 그래서 메일이 안 나감.

ℹ️ Entra 권한은 "미검증"

코드는 Mail.Send를 요청하도록 설정돼 있으나, 토큰이 아예 없어 실제 동의·발송은 아직 검증 못 함(과거 "동의 완료"는 단정이었음).

✅ 해결 적용

메일발송 클릭 시 토큰이 없으면 자동 재로그인 → 복귀 후 자동 발송. Vercel 환경변수는 이 앱과 무관(프론트가 안 읽음).

메일 발송은 브라우저 → Microsoft Graph (/me/sendMail)로 직접 호출됩니다. 서버나 .env의 저장된 토큰이 아니라, 로그인한 사용자의 브라우저 세션 토큰을 씁니다. 그래서 "어느 PC냐"는 상관없습니다.

3. 협력사 메일발송이 실패하는 진짜 이유

코드(sendCredMail())는 이렇게 동작합니다:

// 로그인 세션에서 MS Graph 토큰을 꺼냄
const token = session?.provider_token;
if (!token) return { ok:false,
  reason: "메일 발송 토큰 없음 — SSO 재로그인 + Mail.Send 권한 필요" };

// 이 토큰으로 Microsoft에 직접 메일 발송 요청
fetch("https://graph.microsoft.com/v1.0/me/sendMail", { headers:{ Authorization:"Bearer "+token } ...})

provider_token이 비면 곧바로 그 오류가 납니다. 비는 경우는 둘:

원인 A

이메일/비번으로 로그인함

로그인 창에 SSO 버튼과 이메일/비번 폼이 둘 다 있음. 비번으로 들어오면 MS Graph 토큰 자체가 없음 → provider_token 없음.

원인 B (더 흔함)

SSO 했어도 토큰이 곧 사라짐

provider_token은 로그인 콜백 직후 그 세션에만 존재. 새로고침·1시간 경과·세션 갱신 시 사라짐. 현재 scope에 offline_access가 없어 재발급도 불가.

⮕ 그래서 "다른 PC의 .env에 토큰이 있어야 하나?"는 아니오입니다. 이 기능은 저장된 토큰을 안 쓰고, 그 자리에서 SSO 로그인하면 토큰이 생깁니다.

🔬 2026-06-30 실측 검증 결과

관리페이지(ai.jeilm.co.kr/app/admin-vendors.html)에서 직접 확인: dh.choi로 로그인된 상태였지만 provider_token이 비어 있었고, provider_refresh_token도 없음(scope에 offline_access 미포함). 즉 원인 B(세션 자동갱신 시 토큰 소실 + 재발급 불가)가 실제 원인으로 확정. 챗봇포털에서 이어받은 세션이라 발송 시점엔 이미 토큰이 사라진 상태였습니다.

5. "Vercel 환경변수 미등록" — 문제 되나?

✅ 이 앱에서는 문제 없음

Vercel에 올라가는 건 정적 HTML이고, Supabase 접속값은 app/config.js에 직접 적혀 있습니다. 빌드 단계·vercel.json·process.env 치환이 전혀 없어 프론트는 Vercel 환경변수를 읽지 않습니다. 따라서 Vercel env가 비어 있어도 로그인·메일발송 동작은 영향받지 않습니다. (anon 키는 설계상 공개해도 되는 키 — 실제 보호는 RLS가 담당)

⚠️ 대신 진짜 확인할 곳

계정 생성·비번 초기화(vendor-reset-password 함수)가 안 되면, Vercel이 아니라 Supabase Edge Function의 Secrets에 SERVICE_ROLE 등이 등록돼 있는지 확인해야 합니다.

6. 해결 방법

✅ 적용 완료 — 자동 재인증 발송 (2026-06-30, admin-vendors.html)

이제 관리자는 표의 '📧 메일발송' 버튼만 누르면 됩니다. 내부 동작:

1
발송 직전 provider_token 확인
2
토큰이 없으면 무프롬프트 Microsoft 재로그인(이미 MS 로그인 상태면 화면 깜빡임만) — 발송 의도를 sessionStorage에 잠시 보관
3
복귀 직후 신선한 토큰으로 비번 재발급 + 메일 자동 발송(resumePendingMail)

토큰이 살아있으면 재로그인 없이 즉시 발송. 실패 시 임시비번이 화면에 표시돼 직접 전달할 수 있습니다.

🟡 그래도 안 되면 (수동 폴백)

1
로그인 창에서 'Microsoft 사내 계정으로 로그인(SSO)' 버튼으로 로그인 (이메일/비번 폼 말고)
2
그래도 실패하면 화면에 뜬 임시 비밀번호를 협력사에 직접 전달(메일·메신저)

재인증 후에도 토큰을 못 받으면 Entra의 Mail.Send 권한/관리자 동의가 실제로 돼 있는지 확인이 필요합니다(아래 근본 수정).

🟢 근본 수정 (운영 전환 시 권장)

방법 1

offline_access 추가

SSO scope에 offline_access를 넣어 refresh 토큰을 받고, 발송 직전 Graph 토큰을 재발급. 간단하지만 토큰을 프론트가 보관.

방법 2 (권장)

백엔드 발송으로 전환

메일 발송을 Supabase Edge Function으로 옮기고 앱 권한 + client secret으로 서버에서 발송. 어느 PC·세션이든 안정적.

7. 보안 주의 (꼭)

  • SUPABASE_SERVICE_ROLE_KEY · ENTRA_CLIENT_SECRET는 절대 프론트(config.js)나 Vercel에 두지 않기. (현재 config.js엔 공개키만 있어 정상)
  • .env는 git에 올리지 않기 — 실제 비밀값 포함 (CLAUDE.md §1)
  • Entra 앱 설정 변경(권한·redirect URI)은 관리자가 직접, 코드만 바꾸면 실패 (§5.2)
  • Entra Client Secret 만료 2026-12-09 → 11월 중 갱신 필요

8. 확인 체크리스트

  • 메일 발송 전 'Microsoft SSO' 버튼으로 로그인했는가 (이메일/비번 아님)
  • 로그인 직후 새로고침 없이 바로 발송했는가
  • Supabase Edge Function Secrets에 SERVICE_ROLE 등 등록돼 있는가
  • (근본수정) offline_access 추가 또는 백엔드 발송 전환을 검토했는가
  • .env·secret이 git/프론트에 노출돼 있지 않은가
